## なぜ多くの越境ECサイトは、凍結ページや封存ページをインデックスさせるべきではないと分かっているのに、補件撤回の解凍ページはそのままGoogleに入れてしまうのか？

最近、この手のケースをかなり見ます。

多くのチームは、凍結ページ・封存ページ・作廃ページの扱いにはすでに慎重です。これらはたいてい業務フロー用のページであって、公開SEOランディングページではないと分かっているからです。ところが処理が **unfreeze / release / unlocked** の段階に進むと、システムがまた別のURL群を自動生成しがちです。たとえば以下のようなものです。

- `/claim/document-withdraw-unfreeze`
- `/claim/document-withdraw-release`
- `/claim/document-withdraw-unlocked`
- `/finance/document-withdraw-unfreeze-detail`
- `/wallet/document-withdraw-release-record`
- `/claim/document-withdraw-unfreeze?case=xxx`

こうしたURLがインデックスされるとき、社内ではだいたい次のように考えられています。

- 解凍時間、理由、担当者があるので薄いページには見えない
- 実際に「解凍後どうすればいいか」を調べる人はいる
- 解凍ページは凍結ページより“結果ページ”っぽく見える
- システムが自動生成するので、後回しになりやすい
- 続行、再送信、履歴確認、サポート連絡などの導線があり、ヘルプページのように見える

でも、サイトがしばらく運用されるとすぐ分かります。**大半の補件撤回解凍ページは、結局のところ特定案件の状態復帰ページ、権限復旧ページ、フロー通知ページです。ある記録がいつ解凍されたか、操作を再開できるか、制限解除後に何が変わったかをその案件の当事者に伝えるためのものであり、公開検索の安定したニーズに答えるページではありません。**

検索ユーザーが本当に知りたいことは、たいてい次のような内容です。

- なぜ withdrawal が frozen から unfrozen になったのか
- 解凍後に再提出が必要なのか
- unfreeze / restore / release / unlock の違いは何か
- 「解凍済み」と表示されるのに操作できないのはなぜか
- まずルールページを見るべきか、ヘルプを見るべきか、サポートに聞くべきか

つまり、**SEO向きなのは非公開の解凍URLではなく、「補件撤回の解凍とは何か」「解凍後の次の手順」「制限解除後も操作できない理由」などを説明する公開ページであることが多い**ということです。

## 補件撤回の解凍ページは何のためのページなのか？

### 1. いちばん大事な役割は、その記録が今どの状態に戻ったかを現在のユーザーに知らせること

一般的な解凍ページは、次のような確認のために使われます。

- withdrawal が解凍・解除・復旧されたこと
- 解凍日時、解凍理由、発動したルール、次のアクション
- 変更が自動復旧なのか、手動審査なのか、リスク制御解除なのか
- サポート、運営、財務、ユーザーが同じ案件をどう理解すべきか

つまり、このページは検索経由の初見ユーザーよりも、すでにそのフローに入っている人のためのページです。

### 2. 案件ID、アカウント権限、リスクログ、処理ステップに強くひも付いている

こうしたページにはよく次の情報が出てきます。

- claim ID、注文番号、案件番号
- freeze 時刻、unfreeze 時刻、更新時刻
- 現在の状態、解除理由、復旧範囲
- 再提出、異議申立て、エクスポート、再開が可能かどうか
- リスクメモ、審査担当、ルールバージョン、処理履歴
- 閲覧・送信・支払い・アップロードを続行できる権限の有無

情報が具体的であるほど、公開検索向けのSEOページには向きません。

### 3. 業務上の価値はあるが、それだけでSEO向きのページになるわけではない

ここを混同するチームは多いです。

- フロー上で重要だから、検索でも重要だと思ってしまう
- 結果と次のアクションがあるから、流入ページにできると思ってしまう
- 少し説明文があるから、公開ヘルプコンテンツだと考えてしまう

でもSEOで大事なのは一つです。**そのページが、公開されていて、長期的に、繰り返し検索される問いに安定して答えられるかどうか。**

多くの解凍ページは、そこを満たしていません。

## 補件撤回の解凍ページはどう扱うべきか？ まずはこの5パターンに分けて考えたほうがいい

### 1. 標準的な解凍ページ、制限解除ページ、操作復旧ページなら、通常は重点SEOページにする必要はない

これがいちばん多いケースです。

こうしたページはたいてい、次の特徴を持っています。

- 特定案件、特定記録、特定アカウントに結び付いている
- 1つの復旧状態だけを説明している
- アカウントの前提がないと理解しづらい
- 公開説明ページではなく、フロー通知ページである
- 検索ユーザーにとっての長期的価値が低い

ひと言で言えば、**標準的な補件撤回解凍ページは、状態復帰ページであって公開コンテンツページではありません。**

### 2. 本当の検索ニーズが「なぜ解凍されたのか」「解凍後に何をするのか」なら、非公開解凍ページにそのキーワードを背負わせない

ここでページ選定を間違えることがよくあります。

「解凍後は再提出できるのか」「withdrawal unfreeze とは何か」を調べている人は、`/claim/document-withdraw-unfreeze?id=xxx` のような非公開URLを見たいわけではありません。

その人が知りたいのは、たとえば次のようなことです。

- なぜ記録が frozen から unfrozen に変わったのか
- 解凍で操作権限は自動的に戻るのか
- 解凍済みなのに送信できないのはなぜか
- unfreeze / restore / release / unlock は実務上どう違うのか
- まずルール、ログ、サポートのどれを確認すべきか

こうした需要は、公開ヘルプページ、FAQ、トラブルシューティングページ、ルール説明ページで受けるほうが自然です。

### 3. もしそのページ自体が公開説明ページやルールページなら、別枠で評価していい

unfreeze、release、unlock と書いてあるページを全部まとめてブロックすればいいわけではありません。

たとえば次のようなページがあるなら話は別です。

- withdrawal がなぜ解凍されるのかを説明する公開ページ
- frozen / unfrozen / sealed / restored の違いをまとめたヘルプページ
- 制限解除後の進め方を案内するガイドページ
- 解凍済みなのに送信できない人向けの診断ページ

この種のページは、非公開の解凍ページとは役割が違います。

アカウント前提なしで理解でき、ルールや実務上の案内があり、URLが安定していて、検索意図に明確に合っているなら、公開インデックス対象ページとして個別に評価できます。

### 4. システムが unfreeze / release / unlock / detail / print / export のような派生URLを作るなら、まとめて制御する

問題は1本のURLだけでは終わりません。

実際には次のような派生も出てきます。

- `/claim/document-withdraw-unfreeze`
- `/claim/document-withdraw-release`
- `/claim/document-withdraw-unlocked`
- `/wallet/document-withdraw-release-record`
- `/claim/document-withdraw-unfreeze/print`
- `/claim/document-withdraw-unfreeze/export?case=xxx`

こうなると検索エンジンには、状態や時間やパラメータだけが少し違うほぼ同じURL群に見えます。これが index bloat の温床です。

だから解凍ページを扱うときは、主URLだけでなく、release、unlock、detail、print、export まで一括で監査したほうがいいです。

### 5. もともと解凍ページを順位獲得させるつもりがないなら、noindex、ログイン制御、sitemap、canonical、権限、内部リンクをまとめて揃える

多くのSEO問題は、ページがあること自体よりも、シグナルが食い違っていることから起きています。

よくある例は次の通りです。

- ページは noindex なのに sitemap では unfreeze URL を送っている
- 本体ページはログイン必須なのに print や export は公開のまま
- canonical が unfreeze / release / unlock 間でバラバラ
- パラメータ付きURLが大量のクロール対象を生む
- 公開ヘルプページは薄いのに、私的フローページの方が情報量が多い

解凍ページをSEO対象にしないと決めたなら、中途半端に直さないことです。

## 補件撤回解凍ページでよく見るSEOの失敗パターン4つ

### 1. 復旧結果や操作ボタンがあるだけで、インデックスされる価値があると思ってしまう

業務情報があることと、公開検索価値があることは別です。

### 2. unfreeze、release、unlock をまとめてクロールさせてしまう

同じフローを支えるページなのに、似たURLにシグナルが分散します。

### 3. 本来は公開説明ページで取るべき検索需要を、非公開の解凍ページで取りにいこうとする

長期的な流入価値があるのは、たいてい公開説明ページのほうです。

### 4. 「解凍ページはSEOしない」と言いながら、sitemap、テンプレート、メッセージセンター、exportリンクで露出させ続ける

これでは低価値URLが検索結果に残り続けます。

こうした需要があるなら、非公開ページを押し上げるのではなく、ちゃんとした公開ページを作るべきです。

### ステップ3: 公開説明ページと非公開フローページを完全に分ける

次の3種類をはっきり区別します。

- 検索ユーザー向けの公開ページ
- 現在の利用者向けの非公開解凍ページ
- サポート、運営、財務向けの内部記録ページ

この3つを同じインデックス方針で扱わないほうがいいです。

### ステップ4: クロールとインデックスのシグナルを揃える

順位対象にしないなら、次を一括で揃えます。

- noindex ルール
- ログインや権限チェック
- sitemap 戦略
- canonical 設定
- パラメータ制御
- 内部リンク露出

### ステップ5: 成果を見るときは「解凍ページが落ちたか」だけで判断しない

むしろ次を見るべきです。

- 低価値な unfreeze URL が検索結果から減っているか
- クロール資源が product、help、blog に戻っているか
- 公開説明ページの表示が安定してきたか
- ユーザーが本来見せたいページに着地しているか

これが本当に見るべきSEOの結果です。
