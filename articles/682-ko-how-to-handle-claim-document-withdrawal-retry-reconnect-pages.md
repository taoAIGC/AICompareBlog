## 왜 많은 크로스보더 이커머스 팀은 화이트스크린, 크래시, 멈춤, 폴링 페이지는 정리하면서 재연결 페이지는 놓치기 쉬울까?

가장 큰 이유는 **겉보기엔 그렇게까지 망가진 페이지처럼 보이지 않기 때문**입니다.

재연결 페이지는 보통 열리긴 열리고, “다시 연결 중”, “세션 복구 중” 같은 문구도 보입니다. 버튼이 살아 있는 경우도 있어서 팀 입장에서는 그냥 잠깐 거쳐 가는 복구 단계라고 생각하기 쉽습니다. 그래서 검색엔진이 봐도 큰 문제 없겠지 하고 넘어가게 됩니다.

하지만 실제로는 그 판단이 가장 위험한 경우가 많습니다.

대부분의 retry reconnect 페이지는 공개 콘텐츠 페이지가 아닙니다. 네트워크 끊김, 세션 만료, 앱 복귀, 상태 동기화 실패 같은 상황에서 **진행 중이던 특정 작업을 이어 붙이기 위한 임시 복구 레이어**에 가깝습니다. 예를 들면 이런 URL입니다.

- `/claim/document-withdraw/retry-reconnect`
- `/claim/document-withdraw/retry-network-recovery`
- `/claim/document-withdraw/retry-session-restore`
- `/merchant/claim/document-withdraw/retry-reconnect?case=xxx`
- `/claim/document-withdraw/retry-resume-flow`

이 페이지가 해결하는 건 공개 검색 질문이라기보다,

- 왜 갑자기 재연결 단계로 들어갔는지
- 네트워크를 복구하는 건지, 세션을 살리는 건지, 케이스 상태를 다시 맞추는 건지
- 사용자가 기다려야 하는지, 새로고침해야 하는지, 다시 들어와야 하는지

같은 **현재 진행 중인 플로우의 문제**입니다.

---

## 이런 페이지는 실제로 무슨 역할을 할까?

### 1. 핵심 역할은 끊긴 플로우를 잠시라도 살려두는 것

보통 이런 작업을 합니다.

- 네트워크, 토큰, 로그인 상태, websocket, bridge 복구 확인
- 중요한 요청 자동 재시도
- 복구 후 상세 페이지, 결과 페이지, 이전 단계 중 어디로 돌릴지 결정
- 지원팀과 모니터링을 위한 복구 지점 제공

즉 본질적으로는 **플로우 복구 페이지**이지, 콘텐츠 페이지가 아닙니다.

### 2. 케이스 정보, 권한, 디바이스 상태 같은 문맥에 강하게 의존한다

이런 URL은 원래 문맥을 벗어나면 의미가 급격히 약해집니다. caseId, 권한, 디바이스, 앱 버전, 캐시, 세션 상태, 복구 로직에 많이 의존하기 때문입니다.

문맥 의존성이 강할수록 공개 SEO 페이지로는 부적합합니다.

### 3. 비즈니스상 유용하다고 해서 SEO 가치까지 높은 건 아니다

UX 측면에서는 도움이 될 수 있습니다. 하지만 검색엔진 입장에서 보면 대부분 이런 말만 하고 있습니다.

**“현재 작업이 다시 연결을 시도하고 있습니다.”**

이건 장기적으로 검색 유입을 받을 만한 공개 답변 페이지와는 거리가 있습니다.

## 어떻게 처리해야 할까? 먼저 이 5가지 경우를 나눠서 보자

### 1. 표준적인 reconnect / network recovery / session restore 페이지라면 핵심 SEO 페이지로 보지 않는 편이 낫다

이런 특징이 많습니다.

- 현재 케이스를 벗어나면 단독으로 이해하기 어렵다
- 네트워크, 세션, 복구 결과에 따라 내용이 계속 바뀐다
- 검색 유입보다 현재 사용자를 플로우 안에 붙잡아 두는 데 가치가 있다

정리하면 **공개 SEO 랜딩페이지보다는 내부 프로세스 페이지로 관리하는 게 자연스럽습니다.**

### 2. 실제 검색 의도가 “왜 계속 재연결되나?”, “재연결 실패하면 어떻게 하나?”라면 그 설명은 공개 도움말 페이지가 맡아야 한다

사용자가 알고 싶은 건 내부 URL 자체가 아니라,

- 왜 계속 재연결이 뜨는지
- 네트워크 문제인지 세션 만료인지
- 웹에서는 돌아오는데 앱에서는 안 되는 이유가 뭔지
- 새로고침, 재로그인, 문의 중 무엇을 먼저 해야 하는지

같은 일반화된 답변입니다. 이런 건 FAQ, 도움말, 장애 가이드가 더 적합합니다.

### 3. 같은 URL이라도 시간, 기기, 연결 상태에 따라 화면이 바뀐다면 인덱싱은 더 신중해야 한다

재연결 페이지는 본질적으로 불안정합니다. 처음엔 “다시 연결 중”, 몇 초 뒤엔 복구 완료, 다른 기기에선 세션 만료 같은 일이 흔합니다.

페이지 자체가 안정적인 답변이 아니라면 검색결과에 안정적으로 남길 이유도 약합니다.

### 4. reconnect / network-recovery / session-restore / resume-flow 같은 변형 URL이 여러 개라면 묶어서 감사해야 한다

이 문제는 보통 한 페이지로 끝나지 않습니다. 비슷한 URL이 여러 개 생기면,

- 구조가 거의 중복되고
- 크롤링 예산을 소모하고
- canonical 신호가 흔들리고
- 앱, 이메일, 지원 도구를 통해 계속 노출됩니다

### 5. 애초에 랭킹 대상이 아니라면 noindex, 로그인 제어, canonical, sitemap, 노출 경로를 한 번에 정리해야 한다

자주 보이는 실수는 이렇습니다.

- noindex를 원하지만 sitemap에는 계속 들어간다
- 로그인 필요 페이지인데 일부 링크는 익명 접근이 된다
- reconnect, status, result 사이 canonical이 충돌한다
- 캐시나 websocket 복구 때문에 크롤러와 사용자가 보는 상태가 다르다
- 앱, 이메일, 내부 도구가 계속 URL을 밖으로 노출한다

한 군데만 고쳐서는 정리가 잘 안 됩니다.

## 자주 보는 SEO 실수 4가지

### 1. “자동 복구되니까 색인돼도 괜찮다”라고 생각한다

자동 복구와 공개 검색 가치 여부는 다른 문제입니다.

### 2. 화이트스크린이나 에러 페이지만 치우고 이런 ‘멀쩡해 보이는 중간 URL’을 방치한다

오히려 이런 페이지가 더 오래 남습니다.

### 3. 내부 reconnect URL이 설명 역할까지 하도록 억지로 맡긴다

검색에서 더 잘 보이는 건 보통 공개 FAQ나 설명 페이지입니다.

### 4. 색인 상태만 보고 URL이 어디서 계속 노출되는지는 보지 않는다

앱, 이메일, 지원, 모니터링에서 계속 노출되면 문제는 반복됩니다.

## 실제로 점검한다면 이런 순서를 추천한다

### 1단계: 복구 계열 URL을 전부 수집한다

### 2단계: 공개 검색 니즈와 내부 플로우 니즈를 분리한다

### 3단계: 공개 도움말 페이지와 내부 프로세스 URL을 완전히 나눈다

### 4단계: noindex, canonical, 로그인 제어, sitemap, cache, service worker, websocket, polling, deep link를 함께 본다

### 5단계: “색인에서 빠졌는가”만 보지 않는다

저가치 URL이 검색결과에서 줄었는지, 정말 보여주고 싶은 공개 페이지의 노출이 늘었는지도 봐야 합니다.
