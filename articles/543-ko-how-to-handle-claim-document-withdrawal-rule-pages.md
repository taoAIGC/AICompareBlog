## 설정 페이지, 옵션 페이지, 선호도 페이지까지 정리했는데 왜 rule 관련 URL은 계속 검색결과에 남을까?

가장 큰 이유는 rule 이라는 단어가 헷갈리기 쉽기 때문이다.

백엔드 설정 화면만큼 내부 페이지처럼 보이지도 않고, 결과 페이지만큼 일회용처럼 보이지도 않는다. 그래서 많은 팀이 혹시 이 페이지가 검색 유입을 받을 수 있지 않을까 하고 망설인다.

대표적인 URL은 이런 식이다.

- /claim/document-withdraw-rule
- /claim/document-withdraw-rules
- /claim/document-withdraw/match-rule
- /claim/document-withdraw/channel-rule
- /claim/document-withdraw/approval-rule
- /claim/document-withdraw/rule-detail

하지만 실제로 이런 페이지 대부분은 공개 검색 수요를 해결하지 않는다. 내부 로직, 즉 어떤 조건에서 철회가 가능하고 어떤 역할이 우선이며 어떤 채널이 다른 채널을 덮어쓰는지 정하는 페이지다.

사용자가 진짜로 검색하는 건 보통 이런 것이다.

- 왜 규칙이 적용되지 않았는가
- 우선순위는 어떻게 동작하는가
- 왜 계정마다 다른 규칙이 걸리는가
- 규칙을 바꿨는데 왜 프론트는 안 바뀌는가
- rule, setting, preference 차이는 무엇인가

---

## 규칙 페이지는 실제로 무엇을 하는가

### 1. 워크플로가 어떻게 움직일지 정의한다

보통 다음을 다룬다.

- 철회 허용 여부 조건
- 역할, 채널, 사이트별 매칭 로직
- 우선순위와 덮어쓰기 규칙
- 알림, 승인, 예외 처리

즉, 먼저 내부 운영 페이지이지 공개 SEO 랜딩 페이지가 아니다.

### 2. 맥락 의존도가 매우 높다

주로 이런 정보가 들어간다.

- tenant, site, channel, role
- 매칭 조건과 우선순위
- 상태, 버전, 적용 범위
- 승인 체계와 예외 처리

구체적일수록 공개 검색에는 더 맞지 않는다.

### 3. 업무상 중요하다고 해서 색인 가치가 있는 건 아니다

업무에 중요하다는 것과 검색엔진에 남길 가치가 있다는 것은 전혀 다른 문제다.

## 어떻게 처리할까? 나는 이 5가지로 나눠서 본다

### 1. 일반적인 rule page, match rule page, channel rule page 라면 핵심 SEO 페이지로 보지 않는 게 맞다

가장 흔한 경우다.

### 2. 실제 검색 수요가 “왜 규칙이 안 먹히는가”라면, private rule page를 억지로 노출시키지 말아야 한다

이럴 때 필요한 건 공개 도움말, FAQ, 점검 가이드다.

### 3. 공개 규칙 설명 문서라면 따로 평가할 수 있다

다음 조건을 만족하면 색인을 검토할 수 있다.

- 로그인 없이 이해 가능하다
- 개인 설정이 아니라 공개 규칙을 설명한다
- URL 이 안정적이다
- 예시와 점검 단계가 있다
- 실제 검색 의도와 맞는다

### 4. rule, rules, match-rule, channel-rule, rule-detail, rule-log 같은 변형 URL 이 많다면 한꺼번에 관리해야 한다

메인 URL 만 막고 변형을 열어두면 문제는 계속 남는다.

### 5. 애초에 검색 유입을 받지 않게 할 생각이라면 noindex, 로그인 제한, sitemap, canonical, 파라미터 처리까지 같이 맞춰야 한다

자주 보는 실수는 이렇다.

- noindex 인데 sitemap 에 남아 있음
- 메인 페이지는 닫혀 있는데 변형은 열려 있음
- canonical 이 뒤죽박죽임
- 파라미터 URL 이 중복을 양산함

---

## 자주 보는 SEO 실수

### 1. 규칙이 많으면 SEO 가치도 높다고 생각하는 것

실제로는 내부 복잡도가 높다는 뜻에 가깝다.

### 2. 메인 rule 페이지만 정리하는 것

detail, log, 기타 변형이 남아 있으면 문제는 해결되지 않는다.

### 3. 공개 검색 의도를 private 페이지로 받으려는 것

대부분은 공개 설명 페이지가 더 맞다.

### 4. 규칙은 바꿨는데 색인 전략은 안 바꾸는 것

로직, 권한, 렌더링이 바뀌면 sitemap 과 canonical 도 같이 바뀌어야 한다.

---

## 점검 순서

### 1단계: 규칙 관련 URL 을 전부 모은다

### 2단계: 실제 검색 의도를 확인한다

### 3단계: 공개 문서와 내부 페이지를 분리한다

### 4단계: 크롤링과 색인 신호를 한 번에 점검한다

### 5단계: 제대로 된 결과를 본다

단순히 인덱스에서 사라졌는지만 볼 게 아니라, 저가치 URL 이 줄었는지, 정말 보여주고 싶은 공개 페이지가 더 잘 보이는지 봐야 한다.

---

## 마무리

핵심은 조건이 몇 개냐가 아니다. 그 페이지가 내부 프로세스를 위한 것인지, 공개 검색 수요에 답하는 것인지가 핵심이다.

내부 로직, 역할 차이, 채널 매칭, 승인 흐름을 위한 페이지라면 보통 private page 로 다루는 게 맞다. 규칙 오류나 우선순위 관련 검색 유입이 필요하다면, 그 목적의 공개 문서를 따로 만드는 편이 훨씬 낫다.
