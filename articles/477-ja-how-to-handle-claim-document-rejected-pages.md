## なぜ多くの越境ECサイトでは、本来検索流入を受けるべき公開ヘルプページではなく、補件却下ページのようなURLが先にインデックスされてしまうのか？

このパターンは本当によく見ます。

商品ページ、カテゴリーページ、ブログ、FAQ、ポリシーはそこそこ整っているのに、補償申請やアフターサービス、書類提出フローを追加した瞬間に、次のようなURLが増えていきます。

- `/claim/document-rejected`
- `/claim/material-rejected`
- `/case/supplement-denied`
- `/after-sales/document-review-failed`
- `/appeal/material-rejected`
- `/merchant/claim/reject-document`

そして最初はこう考えがちです。

- ユーザーは実際に「なぜ却下されたのか」を知りたがっている。
- 画面には日時、却下理由、次のアクションがあるので、空ページには見えない。
- 「書類が却下された理由」「却下後どうすればいいか」を検索する人もいる。
- すでにページがあるなら、ロングテールを拾えるかもしれない。

一見すると筋が通っています。

でも運用が進むと問題が見えてきます。**多くの補件却下ページは、結局のところ特定ケースに紐づいたワークフロー結果ページです。役割は「今回提出した書類は通らなかった」と現在のユーザーに伝えること。公開検索向けの、安定して再利用できる説明ページではありません。**

検索ユーザーが本当に知りたいのは、たとえば次のようなことです。

- なぜ補償申請の書類は却下されるのか
- 却下後に再提出できるのか
- ぼやけた画像、情報不一致、ページ不足でどのように落ちやすいのか
- 修正して出し直すまでの猶予はどれくらいか
- 却下は申請全体の失敗を意味するのか

つまり、SEO向きなのは個別ケースの却下URLではなく、公開のヘルプ記事、FAQ、再提出ガイド、却下理由の説明ページです。

## 補件却下ページは、そもそも何を解決するページなのか？

### 1. いちばん大事な役割は「今回の提出物は通らなかった。修正・差し替え・補足が必要だ」と伝えること

このページは通常、次のために存在します。

- 提出書類が審査を通過しなかったことを知らせる
- 画像の不鮮明、ページ欠落、情報不一致、期限切れなどの理由を示す
- 再提出、補足説明、サポート連絡、フロー終了などの次の行動を案内する
- ユーザー、サポート、審査側が同じ状態を共有できるようにする

つまり、Googleから初めて来た人ではなく、すでにフローの中にいる人のためのページです。

### 2. 多くの場合、ケースID、提出履歴、却下理由、再提出期限に強く紐づいている

典型的には次のような情報があります。

- 注文番号または申請番号
- 直近の提出日時
- 却下日時と却下理由
- どのファイルや項目が通らなかったか
- 再提出が可能かどうか
- 修正版提出の期限
- サポート、チケット、メッセージセンターへの導線

ケース固有の情報が濃いほど、長期的なSEOランディングページには向きません。

### 3. 業務上重要でも、SEO向きとは限らない

ここを混同するチームは多いです。

- 今のユーザーにとって重要だから、検索でも重要なはず
- 却下理由が書いてあるから、検索流入を取れるはず
- ボタンや説明があるから、情報検索にも対応できるはず

でもSEOが見るのはそこではありません。**そのページが、多くの検索ユーザーにとって再利用可能で安定した疑問に答えているかどうかです。**

大半の却下ページはそれを満たしません。特定の1人が、特定のケースを、特定の瞬間に理解するためのページにすぎません。

## 補件却下ページはどう扱うべきか？ まずはこの5パターンに分けて考える

### 1. ただの標準的な rejected page や review fail page なら、通常は主要SEOページとして扱わない

これはもっともよくあるケースです。

こうしたページには共通点があります。

- 特定ケースに依存する
- 役割が「この審査ステップは落ちた」と伝えることにある
- 注文やアカウントの文脈なしでは外部ユーザーに理解しにくい
- 安定した公開説明ページではなく、ワークフロー上のチェックポイントである
- 長期的な検索価値が弱い

要するに、**標準的な補件却下ページは workflow page であって、public content page ではない**ということです。

### 2. 実際の検索意図が「なぜ却下されたのか」「却下後どうすればいいのか」なら、個別ケースの rejected URL にその役割を持たせない

「書類が却下されたらどうするの？」と検索している人が見たいのは、`/claim/document-rejected?id=xxx` のような個別ページではありません。

その人が知りたいのはたいてい次のような内容です。

- よくある却下理由
- 再提出までの猶予
- 再アップロード前に何を直せばいいか
- 却下が申請全体の終了を意味するのか
- 再提出より先にサポートへ連絡すべき場面

こうした意図は、公開ヘルプページ、FAQ、ポリシー、状態説明ページで受けるべきです。

### 3. もしそのページ自体が、却下理由や再提出ルールを説明する公開ガイドで、内容も安定しているなら、個別にインデックス可否を評価できる

`rejected` が含まれるURLを全部ブロックすればいいわけではありません。

たとえば次のようなページです。

- よくある却下理由を説明する公開ガイド
- 却下後の再提出方法を説明する公式ページ
- 書類が通らなかったときの対処を説明するヘルプページ
- 画像がぼやけている、情報が食い違う、ページが足りないときの解説ページ

こうしたページが、

- 特定アカウントやケースに依存せず理解できる
- 目的と範囲が明確
- URLが安定している
- 実際の検索意図に対応している
- 初めて来たユーザーにも単体で価値がある

のであれば、それは workflow page ではなく public service page に近く、別で評価できます。

### 4. システムが under-review、approved、rejected、result、resubmit をまとめて生成するなら、1つのクラスターとして管理する

問題は1本のURLだけとは限りません。

実際には次のような派生URLも一緒に出てきます。

- `/claim/document-under-review`
- `/claim/document-approved`
- `/claim/document-rejected`
- `/claim/document-result`
- `/claim/document-resubmit`
- `/claim/document-rejected?case=xxx`

すると、

- 本文がほぼ同じで状態だけ違う
- under-review、approved、rejected、resubmit がまとめてインデックスされる
- メールやメッセージ通知からリンクが流れ続ける
- 検索エンジンがどのURLを残すべきか判断しづらい

という問題が起きます。だから、**rejected page だけを単体で見ないこと**が大事です。

### 5. ランキングさせたくないなら、noindex、ログイン保護、sitemap、canonical、パラメータ、キャッシュをまとめて揃える

ここで起きるSEO問題の多くは、実はシグナルの不一致です。

たとえば：

- ページには `noindex` があるのに sitemap では rejected URL を送っている
- ページが self-canonical なのに result や resubmit と競合している
- 本来ログイン必須なのに匿名リクエストでもHTML全文が返る
- case、ticket、source、language などのパラメータ付きURLが別ページとしてクロールされる
- cache や prerender の設定で本来非公開の内容が露出してしまう

SEO入口にしないと決めたなら、1つだけ直して終わりにしないことです。

## この手のページでよく見るSEOの失敗 4つ

### 1. 却下理由が書いてあるだけで、検索価値が高いと考えてしまう

### 2. rejected、result、resubmit、help を混在させて、サイト内に似たような workflow URL を増やしてしまう

### 3. パラメータ付き rejected URL を sitemap や内部リンクに流してしまう

### 4. 公開の情報検索を、個別ケースのページで取ろうとしてしまう

かなり無駄の大きい判断です。

## 今すぐ rejected pages を監査するなら、この順番で見る

### ステップ1: 拒否関連のURLタイプを全部洗い出す

最低でも：

- under-review pages
- approved pages
- rejected pages
- result pages
- resubmit pages
- パラメータ付き rejected URLs

### ステップ2: どの検索意図を公開コンテンツで受けるべきか決める

たとえば：

- なぜ補償申請の書類は却下されるのか
- 却下後に再提出できるのか
- どんなファイルが審査で落ちやすいのか
- 却下後の猶予はどれくらいか
- 却下は申請全体の失敗を意味するのか

### ステップ3: 公開説明ページと private workflow pages を完全に分ける

### ステップ4: 技術シグナルとフロールールをまとめて確認する

### ステップ5: インデックス削除だけで成功を測らない

見るべきなのは、

- 公開ヘルプページが「書類が却下された理由」系の検索を取り始めたか
- rejected、result、resubmit が少しずつインデックスから抜けているか
- crawl budget が商品、カテゴリ、ブログ、ヘルプに戻っているか
- 検索流入ユーザーがより早く答えにたどり着けているか

## まとめ

**大事なのは、そのページに “rejected” という語が入っているかではありません。特定のケースを処理するページなのか、それとも公開の検索課題に答えるページなのか、そこです。**

主な役割が内部ワークフローを進めることなら、workflow page として扱うべきです。もし「なぜ却下されたのか」「却下後どうするか」「ぼやけたファイルをどう直すか」といった検索を取りたいなら、private page を検索に出すのではなく、公開の解説記事、FAQ、状態ガイドをきちんと作ったほうが強いです。
