## なぜ白画面やクラッシュ、フリーズ、ポーリング系のURLは整理したのに、再接続ページだけは取りこぼされやすいのか？

理由はシンプルで、**見た目がそこまで壊れていないから**です。

再接続ページは、真っ白でもなければ完全なエラーページでもありません。開くことはできるし、「再接続中」「セッションを復元しています」といった文言も出るので、チーム側が「これは一時的な回復画面だし、検索エンジンに見つかっても大きな問題ではないだろう」と判断しがちです。

でも、そこが一番危ないところです。

実際のところ、こうした retry reconnect ページの多くは公開コンテンツではありません。ネットワーク断、セッション切れ、アプリ復帰、状態同期失敗といったタイミングで、**いま進んでいる1件の処理を途中で落とさないための回復用URL**です。たとえば次のようなURLです。

- `/claim/document-withdraw/retry-reconnect`
- `/claim/document-withdraw/retry-network-recovery`
- `/claim/document-withdraw/retry-session-restore`
- `/merchant/claim/document-withdraw/retry-reconnect?case=xxx`
- `/claim/document-withdraw/retry-resume-flow`

このページが解決しているのは、公開検索の疑問というより、

- なぜ今再接続に入ったのか
- 復元しているのはネットワークなのか、ログイン状態なのか、ケース状態なのか
- ユーザーは待つべきか、更新すべきか、入り直すべきか

といった**進行中フローの問題**です。

---

## 再接続ページは何のために存在しているのか

### 1. 役割の中心は、途切れたフローをその場でつなぎ直すこと

典型的な再接続ページは、次のようなことを行います。

- ネットワーク、トークン、ログイン状態、websocket、bridge の回復確認
- 重要なリクエストの自動再試行
- 復旧後に詳細ページ、結果ページ、前ステップのどこへ戻すか判断
- サポートや監視向けの回復ポイント提供

つまり本質的には**ワークフロー回復ページ**であって、コンテンツページではありません。

### 2. ケース情報、権限、端末状態などの文脈に強く依存する

この種のURLは、元の文脈を外れると一気に意味が薄くなります。caseId、権限、端末、アプリ版、キャッシュ、セッション状態、回復ロジックに強く依存するからです。

文脈依存が強いほど、公開SEOページには向きません。

### 3. 業務上便利でも、SEO上価値があるとは限らない

ここを混同するチームは多いです。

このページはUX的には役立つかもしれません。でも検索エンジンにとっては、結局のところ

**「いまこのタスクが再接続を試みています」**

としか答えていないケースが大半です。長期的に検索流入を受ける公開回答としては弱いです。

## どう処理するべきか？ まずはこの5パターンに分けて考える

### 1. 標準的な reconnect / network recovery / session restore でしかないなら、基本的には主力SEOページにしない

よくある特徴は次の通りです。

- 現在のケースを離れると単独では理解しづらい
- ネットワークやセッション状況で表示が変わる
- 価値の中心は検索流入ではなく、進行中ユーザーの救済にある

要するに、**公開SEO入口というより、私有フローURLとして扱うほうが自然**です。

### 2. 本当に検索されているのが「なぜ再接続が続くのか」「再接続失敗時はどうするのか」なら、その説明は公開ヘルプページで受けるべき

ユーザーが知りたいのは内部URLそのものではなく、

- なぜ何度も再接続になるのか
- ネットワーク問題なのか、セッション切れなのか
- Webでは戻るのにアプリでは戻らない理由は何か
- まず更新すべきか、再ログインすべきか、問い合わせるべきか

といった一般化された答えです。これは FAQ やヘルプ、障害切り分け記事に向いています。

### 3. 同じURLでも時間・端末・接続状況で表示が変わるなら、インデックスはさらに慎重に

再接続ページは不安定です。最初は「再接続中」、数秒後には復旧、別端末では失敗、ということが普通に起きます。

ページ自体が安定した答えでないなら、検索結果で安定した答えとして残す理由は薄いです。

### 4. reconnect / network-recovery / session-restore / resume-flow のような変種が複数あるなら、まとめて監査する

この手の問題は1ページだけで終わらないことが多いです。複数の似たURLが並ぶと、

- 構造がほぼ重複する
- クロールバジェットを消費する
- canonical がぶれる
- アプリ、メール、サポート導線から露出し続ける

といった問題が起きます。

### 5. そもそも順位対象にしないなら、noindex・ログイン制御・canonical・sitemap・露出元をまとめて整える

よくある失敗は、

- noindex にしたいのに sitemap では送っている
- ログイン必須のはずなのに一部リンクは匿名で開ける
- reconnect と status と result の canonical が競合する
- キャッシュや websocket 回復でクローラーとユーザーの見える状態が違う
- アプリやメールや社内ツールがURLを出し続ける

といったものです。1か所だけ直しても収まりません。

## よく見るSEOの失敗パターン4つ

### 1. 「自動で復旧するから、インデックスされても平気」と考える

自動復旧することと、公開検索向きであることは別です。

### 2. 白画面やエラーページだけ掃除して、こういう“普通に見える中間URL”を放置する

むしろこちらのほうが残りやすいです。

### 3. 内部の reconnect URL に説明役まで背負わせる

検索で評価されやすいのは、公開FAQや解説ページです。

### 4. インデックス状況だけ見て、どこからURLが出続けているかを見ない

アプリ、メール、サポート、監視から出続ける限り、問題は再発します。

## 実際に監査するならこの順番がおすすめ

### 1. 回復系URLを全部集める

### 2. 公開検索ニーズと内部フローニーズを切り分ける

### 3. 公開ヘルプページと私有フローURLを完全に分ける

### 4. noindex、canonical、ログイン制御、sitemap、cache、service worker、websocket、polling、deep link をまとめて見る

### 5. 「インデックスから落ちたか」以外も確認する

低価値URLが検索結果から減っているか、本来見せたい公開ページの表示が増えているかも重要です。

## まとめ

補件撤回リトライの再接続ページは、多くの場合“コンテンツページ”ではありません。処理が一瞬切れたあとに、そのタスクを続行するための**一時的な回復シェル**です。

UX上は大事でも、SEO上の価値が高いとは限りません。**プロダクト上必要な理由**と**検索エンジンに残す理由**を分けて考えるだけで、サイト構造はかなり整理しやすくなります。
