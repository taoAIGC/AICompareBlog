## 많은 웹사이트에서 멤버십 시스템, SaaS 백엔드, 사용자 계정 시스템을 개발하는 이유는 무엇인가요? 결국 출시될 가능성이 가장 높은 것은 제품 페이지가 아니라 로그인 후 일련의 사용자 센터 페이지, 계정 페이지, 대시보드 페이지일까요?

이것은 실제로 많은 사람들이 생각하는 것보다 더 일반적입니다.

특히 회원 사이트, 도구 사이트, 강좌 사이트, 커뮤니티 사이트 또는 SaaS를 구축할 때 시스템이 온라인 상태가 되면 다음 주소가 사이트에 천천히 나타나기 쉽습니다.

- `/계정`
- `/대시보드`
- `/회원센터`
- `/사용자/프로필`
-`/계정/주문`
- `/dashboard?tab=billing`
- `/profile?from=헤더`
- 로그인 후 포인트 페이지, 쿠폰 페이지, 알림 페이지, 주문 페이지, 보안 설정 페이지, 이메일 바인딩 페이지 등의 하위 페이지도 있습니다.

많은 팀이 이 페이지를 볼 때 일반적으로 첫 번째 반응은 다음과 같습니다.

- 공식 페이지이므로 참여에 지장이 없어야 합니다.
- 사용자 센터는 로그인 페이지보다 콘텐츠가 많고 완성도가 높은 것 같습니다. 어쩌면 약간의 트래픽이 발생할 수 있습니다.
- 일부 브랜드워드 사용자는 이미 계정입구를 검색하고 있을 것이므로 개인센터 페이지를 내놔도 상관없습니다.
- 시스템에서는 기본적으로 생성합니다. 지금은 보관하고 나중에 정리하세요.

완전히 불합리한 것처럼 들리지는 않습니다.

하지만 실제로 오랫동안 해보면 이런 종류의 페이지가 종종 다음과 같이 나타나는 것을 발견하게 될 것입니다.

- 페이지는 로그인 상태에 따라 크게 달라지며, 사용자마다 다른 콘텐츠가 표시됩니다.
- 로그인하지 않은 상태에서는 빈 쉘이고, 로그인한 후에는 개인정보입니다.
- `tab`, `from`, `source`, `scene`과 같은 매개변수가 너무 많으면 대략적인 URL을 쉽게 표기할 수 있습니다.
- 사용자가 검색결과에서 클릭하면 먼저 '먼저 로그인하세요' 또는 맥락 없이 백그라운드 홈페이지가 표시되는 경우가 많습니다.
- 실제로 검색 요구 사항을 처리해야 하는 상품 페이지, 가격 페이지, 기능 페이지 및 도움말 페이지가 강화되지 않았습니다.

그래서 최근에 **사용자 센터 페이지 처리 방법**을 공부하고 계시다면 먼저 매우 간단한 판단을 내리겠습니다.

**대부분의 사용자 센터 페이지는 본질적으로 "공개 검색 랜딩 페이지"가 아닌 "계정 사용 프로세스 페이지"입니다. 유지, 전환, 재구매, 사용자 운영 측면에서는 확실히 중요하지만, 대부분의 경우 핵심 SEO 페이지로 홍보하기에는 적합하지 않습니다. 실제로 해야 할 일은 단순히 방치하거나 전면적으로 차단하는 것이 아니라 먼저 이 URL이 로그인한 사용자에게 서비스를 제공하는지 아니면 공개 검색 질문에 답변하는지 구별하는 것입니다. **

## 사용자센터 페이지는 어떤 문제를 해결하나요? 단순히 '로그인하면 보이는 홈페이지'가 아닙니다.

사용자 센터 페이지에 대한 많은 사람들의 이해는 여전히 다음과 같습니다.

- 이용자가 로그인 후 들어가는 곳
- 개인정보 수집, 주문, 구독 관리를 위한 페이지
- 웹사이트의 매우 일반적인 백엔드 페이지

이러한 이해는 틀린 것은 아니지만 여전히 너무 얕습니다.

실제 사용자 센터 페이지는 일반적으로 최소한 세 가지를 해결합니다.

### 1. 핵심 임무는 로그인한 사용자가 계속 작업을 수행할 수 있도록 하는 것이며 공개 검색 질문에 답변하는 것이 아닙니다.

사용자 센터 페이지의 핵심 작업은 일반적으로 다음과 같습니다.

- 계정 상태 확인
- 주문, 구독, 프로필 및 보안 설정 관리
- 끝나지 않은 작업 계속하기
- 더 자세한 기능 하위 페이지를 입력하세요.

즉, "아직 검색하고 비교하는 사람들"을 섬기는 것이 아니라 "이미 시스템에 등록하고 입력한 사람들"을 섬기는 것에 더 가깝습니다.

### 2. 개인정보, 권한 상태, 행사장 입장 등과 관련이 있는 경우가 많습니다.

예를 들면:

- 내 주문
- 내 강좌
- 내 청구서
- 내 쿠폰
- 내 프로필
- 보안 설정
-팀 권한
- API 키 관리

이는 사용자 센터 페이지가 일반적으로 공개 색인에 적합한 안정적이고 통합된 페이지가 아니라 ID, 권한, 패키지 및 상태에 따라 변경되는 페이지 유형임을 결정합니다.

### 3. 비즈니스 가치는 있지만 SEO 가치가 있다는 의미는 아닙니다.

사용자 센터는 물론 중요합니다.

그렇지 않으면 멤버십 시스템이 운영될 수 없으며, 갱신 및 운영이 혼란스러울 것입니다.

그러나 중요한 사업이라고 해서 자연 검색을 수행하는 것이 적합하다는 것을 의미하지는 않는다는 점에 유의해야 합니다. **

사용자는 검색할 때 일반적으로 다음 사항에 더 관심을 갖습니다.

- 이 제품은 어떤 문제를 해결할 수 있나요?
- 패키지 요금은 어떻게 청구하나요?
- 시도해 보셨나요?
- 기능과 경쟁제품의 차이점은 무엇인가?
- 사용 및 시작 방법
- A/S, 환불, 청구서 처리 방법

이러한 검색 요구 사항을 처리하는 데 실제로 더 적합한 것은 일반적으로 사용자 센터 페이지가 아니라 다음과 같습니다.

- 제품 페이지
- 가격 페이지
- 기능 소개 페이지
- 도움말 센터
- FAQ 페이지
- 튜토리얼 페이지
- 시나리오 계획 페이지

이러한 문제가 사용자 센터 페이지에서 최종적으로 해결된다면 입구를 교묘하게 디자인한 것이 아니라 콘텐츠 레이어와 계정 레이어가 혼합되어 있을 가능성이 높습니다.

## 사용자센터 페이지는 어떻게 처리하나요? 먼저 이 5가지 상황을 별도로 처리하는 것이 좋습니다.

### 1. 표준 개인 센터, 회원 백엔드, 계정 개요 페이지일 경우 일반적으로 핵심 SEO 페이지로 취급할 필요가 없습니다.

가장 일반적인 카테고리입니다.

예를 들면:

- 회원센터 홈페이지
- 개인 계정 개요 페이지
- 강좌 백엔드 홈페이지
- SaaS 콘솔 홈페이지
- 판매자 백엔드 홈페이지

이러한 유형의 페이지는 물론 기존 사용자에게 중요하지만 문제도 분명합니다.

**대부분의 공개 검색 요구 사항을 충족하기 위해 존재하지 않습니다. **

사용자가 검색 엔진에서 클릭한 후 일반적인 반응은 다음과 같습니다.

- 아직 제품에 대한 명확한 이해가 되지 않습니다. 어떻게 백엔드에 먼저 들어갈 수 있나요?
- 공개 설명 없이 내 모든 정보 항목이 여기에 있는 이유는 무엇입니까?
- 이 페이지에서 정확히 무엇을 알 수 있나요?
- 로그인하지 않으시면 보실 수 없습니다. 로그인했다면 콘솔일 뿐입니다.

따라서 표준 사용자 센터 페이지는 일반적으로 초점 SEO 페이지보다는 프로세스 페이지로 더 적합합니다.

한마디로: **사용자 센터 페이지는 매우 중요할 수 있지만 이 중요성은 대부분 자연 검색이라는 의미가 아닌 제품 운영 측면에서 중요합니다. **

### 2. 실제 검색 값이 기능 설명, 가격 정책, 애프터 서비스 규칙 및 튜토리얼인 경우 사용자 센터 페이지에서 이에 대한 트래픽을 처리하지 마십시오.

이런 오해도 매우 흔합니다.

일부 팀은 무의식적으로 다음과 같은 느낌을 받을 것입니다.

- 사용자 센터에는 많은 정보가 있으므로 포함시켜도 나쁠 것이 없습니다.
- 브랜드 단어가 들어온 뒤 바로 백엔드로 가는 것도 일종의 네비게이션이다.
- 환불, 주문, 구독 등의 기능이 모두 담겨 있으며, 사용자가 직접 찾아볼 수 있습니다.

그러나 SEO 관점에서 보면 이는 불안정한 경우가 많습니다.

사용자는 종종 다음 이외의 항목을 검색하기 때문입니다.

- 내 계정 홈페이지는 어디에 있나요?
- 대시보드는 어떤 모습인가요?

대신:

- 구독 취소 방법
- 인보이스 다운로드 방법
- 어떤 결제수단이 지원되나요?
- 강좌를 반복해서 볼 수 있나요?
- 팀버전과 개인버전의 차이점은 무엇인가요?
- API 키를 생성하는 위치

이러한 질문은 다음이 더 적절하게 질문합니다.

- 가격 페이지
- 도움말 센터 기사
- 지원 문서
- 환불 정책 페이지
- 구독 안내 페이지
- 튜토리얼 페이지 이용하기

인수합니다.

**로그인 후 페이지에서 공개적으로 공개해야 할 내용을 숨기지 말고, 사용자 센터 페이지에서 검색 트래픽을 얻는 데 도움이 되기를 기대하세요. **

### 3. 사용자 센터 시스템에서 대시보드, 프로필, 빌링, 주문, 보안 등의 하위 경로 및 매개변수 URL을 생성할 경우 먼저 "기능 URL"과 "인덱스 URL"을 구분해야 합니다.

많은 웹사이트의 사용자 센터 페이지의 실제 문제는 `/dashboard`가 하나만 있다는 것이 아니라 다양한 변형도 생성된다는 것입니다.

- `/dashboard?tab=billing`
- `/계정?장면=갱신`
- `/profile?from=헤더`
- `/member/orders?page=2`
- `/계정/보안?소스=이메일`
- `/dashboard?plan=team`

이 주소는 물론 로그인 후 제품 처리에 유용합니다.

그러나 SEO의 경우 다음을 의미하는 경우가 많습니다.

- 거의 중복된 URL이 다수 있음
- 매개변수는 페이지 신호를 분해합니다.
- 가져온 리소스는 백그라운드 페이지와 상태 페이지에서 소모됩니다.
- 일부 링크는 사용자를 만료된 상태 또는 오류 컨텍스트로 안내합니다.

따라서 사용자 센터 페이지를 다룰 때 특히 실용적인 조치는 다음과 같습니다.

- 먼저 계정 처리에 필요한 URL이 무엇인지 구별하세요.
- 검색 엔진이 실제로 봐야 할 공개 페이지를 구별합니다.
- 매개변수 유형의 배경 주소는 기본적으로 모든 곳에 출력되지 않습니다.
- 버튼, 내비게이션, 이메일, 도움말 문서의 입구를 통일하는 것이 가장 좋습니다.

**사용자 센터 페이지에서 통제할 수 없을 가능성이 가장 높은 것은 본문이 아닌 배경 하위 경로 및 매개변수인 경우가 많습니다. **

### 4. "개인 공개 홈페이지"와 "비공개 이용자 센터"를 모두 운영하는 경우, 두 페이지를 완전히 분리하시기 바랍니다.

이는 특히 간과하기 쉽습니다.

많은 제품에는 실제로 동시에 두 개의 페이지 세트가 있습니다.

- 한 세트는 공개적으로 접근 가능한 개인 홈페이지, 작가 홈페이지, 강사 홈페이지, 상점 홈페이지
- 로그인해야만 볼 수 있는 개인센터, 설정페이지, 주문페이지, 정보페이지가 한 세트로 구성되어 있습니다.

두 가지 유형의 페이지 모두 '사용자'와 관련이 있는 것처럼 보이지만 SEO 가치는 전혀 다릅니다.

공개 홈페이지이고 다음 조건을 충족하는 경우:

- 명확한 캐릭터, 브랜드, 매장 정보를 가지고 있어야 합니다.
- 콘텐츠 출력, 작품, 평가, 사례 등이 포함됩니다.
- 페이지는 로그인 상태에 의존하지 않습니다.
- 검색 질문에 독립적으로 답변할 수 있습니다.

그러면 인덱스 값을 가질 수 있습니다.

하지만 개인 사용자 센터, 내 계정, 내 주문, 내 구독 등 단순한 페이지라면 일반적으로 추가 프로세스 페이지입니다.

**가장 두려운 점은 템플릿이 거의 똑같아 보여서 결과적으로 공개해야 할 작성자의 홈페이지와 공개해서는 안되는 개인 센터를 기술적으로 혼합하여 하나의 세트로 만든다는 것입니다. **

일단 혼합되면 나중에 나타날 가능성이 가장 높은 것은 다음과 같습니다.

- 공개 페이지와 비공개 페이지 간의 신호 싸움
- 정식 규칙은 균일하지 않습니다.
- 템플릿, 탐색, 사이트맵이 함께 엉망입니다.
- 검색 엔진은 크롤링해야 할 항목과 크롤링하지 말아야 할 항목을 구분할 수 없습니다.

### 5. 사용자 센터 페이지를 순위에 참여시키지 않으려면 NOINDEX, 로그인 차단, 사이트맵, 내부 링크, Canonical을 함께 통합합니다.

많은 사용자 센터 페이지 SEO 문제는 "이런 페이지가 있는지"가 아니라 신호가 서로 싸우기 때문에 발생합니다.

일반적인 상황은 다음과 같습니다.

- 팀에서는 사용자 센터 페이지가 중요하지 않다고 말합니다.
- 사이트맵도 계정관련 URL을 넘겨줬습니다.
- 머리글, 바닥글 및 도움말 문서는 이러한 주소를 모든 곳의 일반 페이지로 연결합니다.
- 로그인하지 않은 상태에서는 한 페이지로 돌아가게 되며, 로그인 후에는 다른 콘텐츠를 보실 수 있습니다.
- 정식 규칙은 아직 통일되지 않았습니다.

결국에는 다음과 같이 됩니다.

- 검색 트래픽을 수신하도록 허용할 의도가 없습니다.
- 검색 엔진에는 "이 페이지가 중요할 수 있습니다"라는 힌트가 계속 표시됩니다.
- 홍보해야 할 공개 페이지의 집중도가 충분하지 않습니다.

사용자 센터 페이지에 주요 SEO 진입을 하지 않기로 결정한 경우 일반적으로 보다 안정적인 아이디어는 다음과 같습니다.

- 명확하고 통일된 계정 입력 규칙만 유지하세요.
- 사이트맵은 실제로 순위에 참여하고 싶은 공개 페이지를 제출하는 데 중점을 둡니다.
- 비공개 서브페이지와 매개변수 페이지는 최대한 노출을 최소화해야 합니다.
- 내부 링크를 위해 백엔드 페이지를 콘텐츠 페이지로 포장하지 마세요.
- NOINDEX, 정식, 로그인 차단 및 템플릿 출력이 일관됩니다.

기술적 신호를 두고 싸우지 마세요. 사용자센터 페이지를 포함할지 말지를 두고 논쟁하는 것보다 이것이 더 중요하다.

## 지금 내가 보는 가장 흔한 사용자 센터 페이지 SEO 실수 4개는 사이트를 망치는 경우가 사이트를 망칠 가능성이 더 높습니다.

### 1. 브랜드 단어가 탐색 요구를 가져올 것이라고 믿기 때문에 사용자 센터 페이지는 자연스럽게 순위에 적합합니다.

브랜드 단어는 실제로 "로그인"의 필요성을 가져올 수 있습니다.

그러나 탐색 요구 사항이 사용자 센터 페이지가 최고의 공개 랜딩 페이지임을 의미하지는 않습니다.

대부분의 경우 공식 웹사이트 홈페이지, 도움말 센터, 통합 계정 입력 페이지가 비공개 대시보드보다 이러한 유형의 검색에 더 적합합니다.

### 2. 로그인 페이지에서 환불, 청구서, 갱신 및 계정 보안에 대한 공개 지침을 숨깁니다.

이는 '공개설명 내용'과 '개인 운영 입구'의 전형적인 혼합이다.

공개적으로 설명해야 할 사항은 공개 도움말 페이지에 정직하게 게시해야 합니다.

규칙을 배우기 위해 사용자가 먼저 로그인하도록 하지 마세요.

### 3. 주문페이지, 포인트페이지, 쿠폰페이지, 보안설정페이지, 알림페이지가 모두 크롤링 가능한 URL로 노출됩니다.

백엔드 구조만 완료된 것 같지만 실제로 사이트에는 이미 몇 가지가 더 있습니다.

- 상태 얇은 페이지
- 매개변수 변형 페이지
- 비공개 페이지
- 독립적인 검색값이 거의 없는 페이지 처리

이러한 페이지가 너무 많으면 사이트 내의 신호가 쉽게 희미해집니다.

### 4. 사용자 센터 페이지 SEO를 하고 싶지 않다고 하는데 템플릿, 탐색, 사이트맵, 표준은 모두 제 역할을 합니다.

당신을 가장 두렵게 하는 것은 당신이 어떤 선택을 하느냐가 아닙니다.
중요하지 않다고 말하면서도 엄밀히 말하면 "중요하다"고 계속 암시하는 것입니다.

이런 종류의 신호 싸움은 단순히 계정 페이지를 유지하는 것보다 더 큰 피해를 주는 경우가 많습니다.

## 사용자센터 페이지도 포함해야 할까요? '통합 답변'을 먼저 묻지 말고 공개 검색 요청을 수락하는지 먼저 물어보세요.

많은 사람들이 결국 같은 질문으로 돌아올 것입니다.

**사용자센터 페이지도 포함해야 하나요? **

나의 판단 방법은 항상 매우 간단했습니다. 남들한테 어떻게 하는지 물어보기보다는 먼저 이 페이지는 뭐하는 페이지인지 물어봅니다.

주로 다음 작업을 수행하는 경우:

- 개인 계정 개요 표시
- 주문, 청구, 구독 및 보안 설정 관리
- 로그인 후 기능 입장 제공
- 개인 알림, 포인트, 쿠폰 또는 학습 진행 상황을 기록합니다.
- 사용자를 보다 자세한 백그라운드 작업 페이지로 보냅니다.

그렇다면 여전히 기능 페이지, 프로세스 페이지 또는 개인 페이지일 가능성이 높습니다.

이러한 유형의 페이지는 비즈니스에 매우 중요할 수 있지만 대부분의 경우 주요 자연 검색 항목이 될 필요는 없습니다.

공개크리에이터 홈페이지, 공개강사 홈페이지, 공개스토어 홈페이지 등 일부 경우에만 페이지 자체가 검색질문에 독립적으로 답변할 수 있으며, 단순한 전통적인 사용자센터 페이지가 아닐 수도 있습니다.

하지만 표준 개인 센터, 회원 백엔드, 계정 개요 페이지라면 SEO KPI를 외우도록 강요하지 마세요.
