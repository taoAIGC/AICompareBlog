## Why do many websites like to “jump all over the site” as soon as they change their links?

This is so common.

When many people undergo website revision, column adjustment, or domain name migration, their first reaction is not to sort out the relationship between the old page and the new page, but to go directly to the jump rules:

- All old links will jump to the homepage
- All old columns jump to the first page of the new column
- If you can't find the corresponding page, jump to a place where you can open it first.
- Even product pages, article pages, and event pages are all crammed into the same target page

On the surface, it seems quite positive:

- Users will not see errors reported
- Crawlers can also continue to access
- The number of background 404s seems to have decreased.

But the problems of many sites start from here.

Because **301 redirection is not a tool to "hide error reports", but to "clearly tell search engines: this page has been permanently moved." **

If you have never thought clearly about where to move this page, the more you do 301, the more mixed the signal will be.

So if you are also studying how to do 301 redirection recently, let me give you a direct conclusion:

**301 is not always better, and not all old URLs should be jumped. The truly effective approach is to first clarify the relationship between pages, and then decide who should jump, where to jump, and which ones should not be jumped at all. **

---

## What exactly does a 301 redirect tell search engines? It’s not a catch-all patch, more like a “moving notice”

Many people understand 301 as an SEO remedial action.

In fact, to be more precise, 301 is telling search engines:

> The content of this old address has been permanently moved to another new address.

Pay attention to the two most critical words here: **Permanent**.

In other words, 301 is more suitable for these scenarios:

- The page has officially changed its URL
- Adjustment of path structure after website revision
- Merge multiple duplicate contents into one new one
- Switch the old domain name to the new domain name
- The old product page will be taken over by the new version page

Its value is not to "skip over and be done with it", but to help search engines and users understand:

- Old pages do not disappear out of thin air
- There is an inheritance relationship between old and new content
- Signals accumulated previously can be continued to the new page
- Users click on the old link and will not completely go astray

So whether 301 is good or not does not depend on whether you have done it, but on:

**Is this jump reasonable, accurate, and consistent? **

## How to do 301 redirect? I would suggest handling these 5 situations separately first.

### 1. If the page has just changed its address permanently, just do a one-to-one jump. Don’t be lazy and jump to the home page.

This is the most standard category.

for example:

- Article URL changed
- The link structure of the product page has been changed
- The column page has been changed to a new path
- Move the website from the old directory to the new directory

The most stable approach in this situation is:

**Old URL 301 to the most corresponding new URL. **

Here are the two most fearful ways of lazy writing:

- All old pages jump to the home page
- All old pages jump to the same column page

The problem with this is obvious:

- The user originally wanted to see specific content, but was led astray
- Search engines cannot see the true correspondence between the old page and the new page
- A lot of irrelevant URLs pointing to the same place, the signal will be weird

If there is a one-to-one correspondence between the old page and the new page, then jump honestly one-to-one. This is the cleanest way.

---

### 2. Multiple old pages are merged into a stronger new page, which can be jumped, but only if the search intention is consistent.

Some websites will encounter this situation:

- Several old articles with similar content are merged into one complete version
- Collate multiple old product models into a new product page
- Several small topics with scattered traffic are integrated into one main page

There is no problem in doing 301 at this time, but there is a prerequisite:

**The user intent before and after the merger must be captured. **

For example, if three pieces of content all talk about the same issue, but the old version is loosely written and shallow, it is usually reasonable to merge them into a new, more complete piece of content.

But if the old page talks about issue A and the new page talks about issue B, and you forcefully jump to page B just because it has a higher weight, this kind of jump will most likely only make users more confused.

When I judge whether this kind of page can be jumped, I usually look at three points:

- Is the topic close enough?
- Will the user feel "that's what I'm looking for" when they click on it?
- Does the new page really meet the old needs?

If these three points are met and then merged and jumped, the effect is usually more stable.

---

### 3. When revising the website, migrating the directory, or changing the domain name, you can do rule jumps in batches, but don’t just save trouble.

This type is the most likely to cause big problems in technical SEO.

for example:

- `/blog/xxx` changed to `/articles/xxx`
- The old site was moved from `abc.com` to the new domain name
- Overall reconstruction of the category page path
- PC station and mobile station merged

In this case, batch rule jump is a normal operation.

But the problem often lies in:

- The rules are written too boldly
- Special pages are not processed separately
- The parameter page, paging page, and filter page all jump together.
- It should be one-to-one mapping, but it is too fast to be done in one pot

So I would rather do this for this type of migration:

1. Export the old URL list first
2. Make new and old mappings for main pages
3. Batch processing that can match rules
4. Separate verification of important traffic pages and core included pages

In one sentence:

**Batch jump can be done, but only if you know what you are jumping in batches. **

---

### 4. When removing products, adjusting columns, or deleting pages, first check to see if there is a "real replacement page"

This type is different from a permanent address change.

Because some pages are not moved, but are really gone.

Common scenarios include:

- A product is no longer available for sale
- An event ends
- A certain topic is no longer maintained
- A tab is cleared

At this time, many people will react conditionedly:

- Let’s talk about 301 first
- At least don't give it a 404

But I would suggest asking first:

**Does it have a suitable alternative page? **

If there is, for example, the old product page corresponds to the upgraded model and the user needs are basically the same, then it is reasonable to skip it.

If not, if you force it to jump to the category page, home page, or an "almost" content page, the experience is usually not good.

So not all deleted pages should be 301.

If there is an alternative page, jump to it accurately;
Without the replacement page, the 404 is just a 404.

Don't mess up the clear signals just to make the report look good.

---

### 5. In scenarios such as temporary activities, short-term out-of-stock, and A/B testing, 302 should be used in many cases, and 301 should not be rushed.

This is the most confusing point for many people.

301 stands for permanent migration, 302 is more suitable for temporary adjustments.

For example, these situations:

- The event page is temporarily offline and will be restored in a few days.
- The product is temporarily out of stock, but the page will come back
- A/B testing short-term traffic cuts
- The holiday theme is temporarily switched to the event version

If you do 301 as soon as you come up, you are telling the search engine:

> Don't look at this old page anymore, it has been permanently moved.

But if you put the old page back after two weeks, the signals before and after will fight.

So don’t think of 301 as a universal jump.

**Use 301 for permanent changes, and 302 for temporary changes; when there is no replacement page, the original page will be retained if it should be retained, and 404 will be returned if 404 is required. **

## The 4 most common 301 errors I see now are more likely to mess up SEO than not skipping them.

### 1. All old links will jump to the home page.

This is the most typical "looks very diligent, but actually saves the most brains".

What users want to see are old articles, old products, and old categories, but the results are always sent back to the homepage, and the experience is basically not very good.

For search engines, this is not a clear migration, but a bunch of irrelevant URLs being forced onto the same page.

---

### 2. The jump chain is too long, A jumps to B, B then jumps to C

This situation is particularly common after repeated website revisions.

-Old links will jump to the old column first
- Jump to the old column and then the new column
- Jump to the details page at the end of the new column

People are tired looking at them, and so are reptiles.

A more stable approach is:

**Try to let the old URL lead directly to the final target page in one step. **

The shorter the jump chain, the cleaner the signal, and the more stable the crawling and access experience.

---

### 3. Only jumps have been made, but the intra-site links, canonical, and sitemap are still old.

Many sites feel that they are done after completing 301.

In fact, this is only half done.

If you have told the search engine that "the old page has been moved" but the site is still there:

- Navigation link old URL
- Old URL linked within the text
- sitemap is still the old URL
- canonical also points to the old address

Then you are fighting yourself.

After the 301 is completed, the on-site references should also be updated accordingly, otherwise search engines will keep accessing the old address repeatedly.

---

### 4. Use JS jump, meta refresh, and canonical to replace the real 301

This is also a very common misunderstanding in technology.

Let’s talk about the conclusion first:

- `canonical` is not a jump
- JS jump is not equal to standard 301
- meta refresh is more like a patch, not the first choice

If you know for sure that the page has been permanently migrated, the most stable way is to return a standard 301 from the server.

Don't make do with some "looks-like" approach, because the signals received by search engines are not exactly the same.

## If you want to check the 301 settings of the website now, I recommend going through them in this order.

### Step 1: List all old URLs

Don't change based on your feelings.

First, try to sort out the old addresses from Search Console, on-site crawlers, logs, old sitemaps, and historical page data.

### Step 2: Classify each old URL

At least it is divided into these categories:

- Change address permanently
- Merge to new page
- Temporary adjustments
- Deleted without replacement page
- Bad link or invalid access

### Step 3: Only do 301 for pages with a clear new destination.

The point is not to jump too much, but to jump accurately.

### Step 4: Check if there are any jump chains and false jumps

Key spot checks:

- Important traffic pages
- Highly included pages
- Old pages with many external links
- Pages with significant fluctuations in visits after the revision

### Step 5: Synchronously update in-site references

Change internal links, navigation, breadcrumbs, canonical, and sitemap together, and don’t let old URLs continue to circulate in the site.

## Last sentence

The real difficulty of 301 redirection is never to write a jump rule.

But you have to figure it out first:

- Is this page permanently moved?
- Does it have a real corresponding new page?
-Whether it feels reasonable for users to click on it
- Have other signals within the website been updated together?

Think about this clearly, 301 is doing the migration;
Jump around if you don't think clearly, 301 just changes the question from "visible" to "hidden".

So if you are still struggling with how to do 301 redirection, here is my advice:

**Distinguish the relationship first, and then make the jump. Being precise is usually more effective than jumping too much. **
