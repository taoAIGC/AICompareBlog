## 왜 많은 사이트는 중복 URL이나 파라미터 페이지를 보면 일단 전부 홈으로 canonical을 보내려고 할까?

이런 장면은 정말 자주 보인다.

사이트가 커지면 어느 순간 “겉보기엔 비슷한 URL”이 계속 생긴다.

- 파라미터가 붙은 링크
- 필터 페이지와 정렬 페이지
- 인쇄용 페이지
- 모바일 버전이나 대체 버전
- 같은 콘텐츠가 여러 경로에서 열리는 경우
- 슬래시 유무가 다른 주소
- http / https 잔존 주소

그러면 보통 반응은 이렇게 나온다.

- 이 페이지는 애매하니 홈으로 canonical
- 이 필터 페이지는 약하니 상위 카테고리로 canonical
- 2페이지 이후는 필요 없으니 1페이지로 canonical
- 이 URL은 귀찮으니 다른 곳으로 몰아버리자

겉으로는 정리처럼 보이지만, 실제로는 신호를 더 어지럽히는 경우가 많다.

왜냐하면 canonical은 “마음에 안 드는 페이지를 치우는 스위치”가 아니기 때문이다.

만약 지금 **canonical 설정 방법** 을 고민하고 있다면, 가장 먼저 잡아야 할 핵심은 이것이다.

**canonical의 역할은 여러 페이지가 매우 비슷하고 검색 의도도 거의 같을 때, 그중 어떤 URL을 대표 페이지로 봐야 하는지 검색엔진에 알려주는 것이다. 내용과 의도가 이미 꽤 달라졌다면 canonical은 오히려 억지스럽고 약한 신호가 된다.**

즉 canonical은 “어느 페이지가 대표인가”를 정하는 도구이지, “이 페이지는 보기 싫으니 치워 달라”는 도구가 아니다.

## canonical은 무엇을 해결하는가? “다른 페이지는 무시해라”라는 뜻이 아니다

많은 사람이 canonical을 이렇게 이해한다.

- 중요하지 않은 페이지 처리용
- 중복 콘텐츠를 한 번에 해결하는 방법
- 하나의 URL만 색인시키는 명령

하지만 실제로는 조금 다르다.

### 1. 어떤 URL이 대표 버전인지 정리해 준다

페이지 자체는 거의 같은데 URL만 달라지는 경우가 있다.

예를 들면:

- 같은 상품 페이지에 서로 다른 파라미터가 붙는 경우
- 캠페인 링크가 달린 글
- 인쇄용 버전
- 같은 내용이 여러 경로로 열리는 경우

이때 canonical은

**이 URL이 메인 버전이다**

라고 말해 주는 역할을 한다.

### 2. 비슷한 페이지의 신호를 대표 페이지로 모으는 데 도움을 준다

핵심은 “도움”이라는 점이다.

canonical은 강한 힌트지만 301처럼 강제 이동은 아니다.

보통 이런 메시지에 가깝다.

- 이 페이지들은 상당히 비슷하다
- 대표는 이 URL로 봐 줬으면 한다
- 다른 페이지가 존재하더라도 중심 신호는 이 URL로 모으고 싶다

그래서 canonical은 301, robots 차단, noindex의 대체품이 아니다.

### 3. 같은 콘텐츠의 URL 변형이 너무 많을 때 생기는 잡음을 줄인다

많은 사이트 문제는 콘텐츠가 너무 많은 게 아니라, 같은 콘텐츠의 URL 변형이 너무 많은 데 있다.

예를 들어 같은 페이지가

- 기본 URL
- 파라미터 URL
- 카테고리 경로 URL
- 태그 경로 URL
- 인쇄 URL

처럼 퍼져 있으면 검색엔진은 어떤 주소를 대표로 볼지 추가 판단을 해야 한다.

바로 여기서 canonical이 유용해진다.

단, 전제는 하나다. **페이지끼리 정말 충분히 비슷해야 한다.**

## canonical은 어떻게 설정할까? 먼저 이 5가지 상황을 나누는 편이 낫다

### 1. 내용은 같고 URL만 다르다면 canonical을 메인 페이지로 보내도 된다

가장 깔끔한 케이스다.

예:

- `/seo-guide?utm_source=x`
- `/product/123?from=twitter`
- `/article/abc?replytocom=18`
- `/page?print=1`

이런 페이지는 보통 다음 특징이 있다.

- 본문 내용이 거의 같다
- 사용자가 보려는 것도 같다
- 검색 의도도 바뀌지 않는다
- tracking, 인쇄, 시스템 문제 때문에 URL만 달라진다

이럴 때 canonical은 잘 맞는다.

보다 안정적인 방식은 보통 이렇다.

- 메인 페이지에는 self canonical
- 파라미터 버전이나 인쇄 버전은 메인 페이지를 canonical 대상으로 지정
- sitemap에는 메인 버전만 넣기
- 내부 링크도 메인 버전을 우선 사용하기

이게 진짜 신호 정리다.

---

### 2. 결과 집합이나 검색 의도가 달라졌다면 상위 카테고리나 홈으로 억지 canonical을 하지 말자

여기서 사고가 정말 많이 난다.

흔히 다음 페이지들을 기계적으로 상위로 몰아버린다.

- 필터 페이지
- 정렬 페이지
- 지역 페이지
- 사이트 내 검색 결과
- 하위 토픽 페이지

하지만 이런 페이지는 겉보기는 비슷해도 실제 노출되는 결과 묶음이 이미 달라졌을 수 있다.

예를 들면:

- 남성 신발
- 남성 신발 + 검정색
- 남성 신발 + 42사이즈
- 남성 신발 + 낮은 가격순

이 페이지들이 전부 색인될 필요는 없다. 하지만 그렇다고 해서 자동으로 모두 상위 카테고리나 홈으로 canonical 해야 하는 것도 아니다.

결과와 의도가 달라졌다면 “같은 페이지다”라고 말하는 설득력이 약해진다.

확인할 질문은 이 정도다.

- 내용이 정말 매우 비슷한가
- 사용자 의도가 거의 같은가
- 진짜 대표 페이지를 하나만 두고 싶은가

이게 불분명하면 canonical을 땜빵으로 쓰지 않는 편이 낫다.

---

### 3. 도메인 간 재발행이나 미러에서는 메인 버전이 분명할 때만 canonical을 써야 한다

cross-domain canonical도 오용이 많다.

자주 보는 상황은 이렇다.

- 공식 사이트가 원문을 올리고 제휴사가 같은 내용을 올림
- 공식 사이트와 리셀러 사이트에 같은 설명문이 있음
- 매체가 원문을 재발행함
- 스테이징이나 미러 사이트에 본사이트와 같은 내용이 있음

이때 기준은 “어느 도메인을 올리고 싶은가”가 아니라,

- 어느 쪽이 원본 또는 메인 운영 버전인가
- 내용이 정말 많이 비슷한가
- 신호를 합치는 관계가 맞는가

여야 한다.

내용이 크게 번역되었거나, 현지화되었거나, 사용자층이 다르거나, 많이 고쳐 썼다면 cross-domain canonical이 맞지 않을 수 있다.

특히 다음은 피하는 편이 좋다.

- 번역 페이지를 원문으로 canonical
- 국가별 페이지를 글로벌 페이지로 canonical
- 재작성 페이지를 원문으로 canonical
- 서로 다른 상품 페이지를 브랜드 홈으로 canonical

**cross-domain canonical은 출처 표시가 아니라, 대표 페이지 관계를 표현하는 신호다.**

---

### 4. 페이지네이션, 다국어 페이지, 국가 페이지, 실제 변형 페이지는 역할이 다르면 하나로 뭉개지 말자

이것도 아주 흔한 실패 포인트다.

대충 이렇게 처리하는 경우가 많다.

- 2페이지 이후를 전부 1페이지로 canonical
- 일본어 페이지를 영어 페이지로 canonical
- 프랑스 페이지를 글로벌 페이지로 canonical
- 모든 변형 옵션 페이지를 한 페이지로 canonical

하지만 이런 페이지 상당수는 단순 중복이 아니다.

예를 들면:

- **페이지네이션** 은 같은 주제 안에서도 다른 아이템 묶음을 담는다
- **언어 페이지** 는 다른 사용자에게 맞는다
- **국가 페이지** 는 가격, 조건, 제공 내용이 다를 수 있다
- **옵션/변형 페이지** 는 재고, 사양, 사용 상황이 다를 수 있다

이럴 때는 보통

- 각 페이지에 self canonical을 두고
- 언어/국가는 hreflang으로 관계를 설명하고
- 페이지네이션과 변형 페이지 역할을 구조적으로 정리하는 편이

더 자연스럽다.

canonical은 독립적인 역할이 있는 페이지를 지워버리는 도구가 아니다.

---

### 5. 그 페이지를 핵심 페이지로 키울 생각이 없다면 canonical만이 아니라 sitemap, 내부 링크, 301, noindex까지 함께 봐야 한다

많은 사이트가 실패하는 이유는 canonical이 없어서가 아니다.

신호끼리 서로 싸우기 때문이다.

전형적인 혼란은 이렇다.

- 페이지는 A로 canonical
- sitemap은 B를 제출
- 내부 링크는 C를 주로 밀어 줌
- 서버는 D를 E로 301
- 템플릿은 또 noindex를 붙임

이러면 검색엔진은 물론 운영팀도 무엇을 메인으로 삼고 싶은지 헷갈리게 된다.

그래서 canonical을 넣기 전에 먼저 확인해야 한다.

- 이 페이지는 남길 것인가, 합칠 것인가, 리다이렉트할 것인가
- 대상 페이지와 정말 충분히 비슷한가
- 사이트 내부에서 여전히 강한 링크를 주고 있지 않은가
- sitemap에는 어떤 URL을 넣어야 하는가
- 대상 페이지 자체는 안정적인가

**canonical은 신호 설계의 한 부분일 뿐, 혼자서 구조 문제를 해결해 주지 않는다.**

## 자주 보는 canonical 실수 4가지

### 1. 일단 전부 홈으로 보내기

가장 흔한 실수다.

하지만 홈은 상품 페이지, 카테고리 페이지, 글 페이지, 필터 페이지와 같은 의도를 가지지 않는다.

검색엔진 입장에서는 관계를 거칠게 처리하는 것으로 보이기 쉽다.

홈은 쓰레기통이 아니다.

### 2. 충분히 비슷하지 않은 페이지끼리 canonical을 쓰기

무서운 건 안 쓰는 것보다 엉뚱하게 쓰는 것이다.

예:

- 서로 다른 카테고리를 canonical로 묶기
- 다른 지역 페이지를 하나의 일반 페이지로 보내기
- 내용이 많이 다른 필터 페이지를 루트 카테고리로 보내기
- FAQ 페이지를 상품 페이지로 보내기

내용과 의도가 분명히 다르면 canonical의 설득력은 빠르게 떨어진다.

### 3. canonical이 noindex, robots, sitemap, 내부 링크와 충돌하기

기술 SEO 점검에서 정말 자주 보인다.

예:

- canonical은 있는데 robots로 크롤 차단
- canonical 대상 페이지가 noindex
- sitemap이 비우선 URL을 제출
- 네비게이션은 계속 비정규 URL을 링크

이건 정리가 아니라 잡음이다.

### 4. canonical 태그 구현 자체가 잘못되기

기본적인 문제지만 많이 나온다.

흔한 오류는:

- 한 페이지에 canonical이 여러 개 있음
- 템플릿에서 상대경로가 깨짐
- canonical 대상이 404이거나 또 다른 리다이렉트임
- A가 B를 가리키고 B가 C를 가리키는 체인

canonical은 관계를 명확하게 만드는 도구인데, 구현이 불명확하면 신호도 약해진다.

## 지금 canonical 설정을 점검한다면 이런 순서가 이해하기 쉽다

### 1단계: canonical 관계 지도를 뽑아 보기

최소한 다음을 확인한다.

- 어떤 페이지가 self canonical인지
- 어떤 페이지가 다른 URL을 가리키는지
- 많은 페이지가 이상한 한 URL로 몰리고 있지는 않은지
- 템플릿이 페이지네이션이나 지역 페이지를 일괄 처리하고 있지는 않은지

### 2단계: URL 모양이 아니라 내용과 의도를 비교하기

봐야 할 건 이거다.

- 본문 내용이 정말 많이 비슷한가
- 결과 집합이 거의 같은가
- 사용자가 해결하려는 문제가 같은가

canonical은 “주소가 비슷한가”가 아니라 “페이지가 비슷한가”로 판단해야 한다.

### 3단계: 대표 페이지 쪽도 안정적인지 확인하기

이상적인 상태는 다음과 같다.

- 대표 페이지 자체가 self canonical
- 다시 다른 URL로 리다이렉트되지 않음
- noindex가 아님
- sitemap과 내부 링크도 그 URL을 메인으로 지지함

### 4단계: 다른 신호와 방향을 맞추기

확인할 것은 다음과 같다.

- 301과 canonical 방향이 같은지
- sitemap이 대표 URL을 제출하는지
- 내부 링크가 대표 URL을 중심으로 향하는지
- 파라미터 처리 규칙과 템플릿 동작이 맞는지

### 5단계: 실제 결과를 보기

중요한 건 태그가 있느냐만이 아니다.

정말 봐야 할 것은:

- 검색엔진이 최종적으로 어떤 URL을 대표로 선택했는지
- 중요한 페이지의 색인이 더 안정됐는지
- 중복 URL 잡음이 줄었는지
- 노출과 트래픽이 원하는 URL에 더 잘 모였는지

## 마지막으로

canonical에서 진짜 어려운 부분은 태그 문법이 아니다.

어려운 건,

- 어떤 페이지끼리가 정말 같은 그룹인지
- 어떤 페이지는 겉보기만 비슷하고 의도는 다른지
- 장기적으로 어떤 URL에 신호를 모으고 싶은지
- 사이트 전체 신호가 그 판단을 뒷받침하는지

를 먼저 결정하는 일이다.

이게 정리되면 canonical은 꽤 안정적인 정규화 도구가 된다.

반대로 정리가 안 되어 있으면,

- 귀찮은 페이지는 일단 canonical
- 약한 페이지도 일단 canonical
- 중복처럼 보이면 전부 홈으로

같은 식이 되고, 결과는 더 명확해지는 게 아니라 더 복잡해진다.

그래서 아직도 **canonical은 어떻게 설정할까?** 를 고민하고 있다면, 내 조언은 간단하다.
