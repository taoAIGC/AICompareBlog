## なぜ越境ECサイトでは、本来順位を取るべき公開ガイドではなく、差し戻しページや revision-required のURLが先に表に出てしまうのか？

よくあるパターンです。

クレーム審査や補足資料の確認に入ると、システムが次のようなURLを自動生成しがちです。

- `/claim/returned`
- `/claim/return-for-revision`
- `/appeal/need-more-info`
- `/after-sales/send-back`
- `/case/back-to-user`
- `/merchant/claim/returned-record`

一見すると、理由や状態や再提出導線があるので内容がありそうに見えます。でも実際には、特定案件、特定アカウント、特定の差し戻し理由に強くひも付いています。検索ユーザーが来ても「追加資料を提出してください」「ログイン後に詳細を確認してください」で終わることが多いです。

**つまり、多くの差し戻しページは公開SEOページではなく、業務フロー用のページです。**

---

## 差し戻しページは何のためのページか

### 1. すでにフローに入っている人のためのページ

主な役割は次の通りです。

- なぜ差し戻されたかを伝える
- 何を追加提出すべきかを示す
- 再提出するのか、サポートに連絡するのかを案内する
- どの審査工程で戻されたのかを示す

### 2. 個別案件と期限情報に強く依存する

多くのページには以下が含まれます。

- 注文番号
- クレーム番号
- 差し戻し理由
- 不足資料
- 再提出期限
- 現在の審査ステップ

だからこそ、長期的に検索流入を受ける公開コンテンツにはなりにくいです。

### 3. 業務価値とSEO価値は別物

ユーザーが本当に検索するのは、たとえば次のような内容です。

- なぜクレームが差し戻されるのか
- 差し戻し後に何を出せばいいのか
- 差し戻しと却下の違い
- 再提出できる期間はどれくらいか

こうした疑問は、公開ガイドやFAQで受けるべきです。

## 差し戻しページの扱いで先に分けたい5つのケース

### 1. 標準的な差し戻しページや再提出ページなら、通常はSEOの主役にしない

こうしたページはたいてい、ログイン必須、案件依存、内容が変動、外部ユーザーへの汎用性が低い、という特徴があります。

### 2. 検索価値があるのは「なぜ差し戻されたか」「次に何を出すか」であって、私有ページそのものではない

検索ユーザーは内部画面ではなく、理由と対処法を知りたいのです。

### 3. returned / need-more-info / send-back / revision-required / returned-record をまとめて管理する

問題は1枚のページではなく、関連URL群が一緒に増えることです。history や record まで出ると重複が増えます。

### 4. 公開説明ページと私有フローページを完全に分ける

公開ページは「差し戻し理由」「必要資料」「再提出方法」を説明し、私有ページは個別案件だけを扱うべきです。

### 5. ランクさせないなら、noindex、ログイン制限、sitemap、canonical、内部リンクをそろえる

テンプレートだけ noindex にしても、sitemap や内部リンクが押していると意味がありません。

### 2. 差し戻しページ、却下ページ、補足資料ページ、再提出ページを混ぜる

### 3. history、record、confirm のような派生ページまで公開してしまう

### 4. 本来作るべき公開ガイドを作らない

### 2. 公開ページで受ける検索意図を整理する
### 3. 公開ページと私有ページを分離する
### 4. 技術シグナルをまとめてそろえる
### 5. 差し戻しページが消えたかではなく、公開ガイドが伸びたかを見る

## 最後にひと言

**差し戻しページをインデックスさせるべきかどうかは、「次の操作があるか」ではなく、そのURLが公開検索ニーズに答えているかどうかで決まります。**

個別案件を扱うだけならフローページとして管理し、検索流入を取りたいなら「差し戻し理由」「必要資料」「再提出方法」を公開コンテンツとして作るほうがずっと健全です。
