## 왜 많은 크로스보더 이커머스 사이트는 실제로 검색 유입을 받아야 할 도움말 페이지보다 결안 페이지, 아카이브 페이지, closed URL이 먼저 노출될까?

이건 꽤 자주 생기는 일이다.

분실 보상, 파손 보상, 사후 분쟁, 이의 제기, 중재 같은 흐름을 사이트에 붙이기 시작하면 시스템은 보통 이런 URL을 만든다.

- `/claim/closed`
- `/claim/closure`
- `/after-sales/closed`
- `/case/closed`
- `/appeal/closure`
- `/arbitration/closed`
- `/claim/closure?caseId=20260327032`

여기에 결안 페이지, 아카이브 페이지, 사건 종료 페이지, 인쇄 페이지, export 페이지, 결안 확인 페이지 같은 변형도 계속 늘어난다.

많은 팀은 처음에 이렇게 생각한다.

- 정식 페이지니까 인덱싱돼도 괜찮다
- 사용자가 “클레임 closed 뜻”, “결안 후 어떻게 해야 하나”를 검색하니 유입을 받을 수도 있다
- 최종 결과, 날짜, 금액, 상태가 있으니 내용도 꽤 있어 보인다
- 시스템이 기본으로 만든 것이니 나중에 정리하면 된다

하지만 시간이 지나면 문제가 드러난다.

- 특정 주문, 특정 케이스, 특정 사용자에 강하게 묶여 있다
- 구글에서 들어온 사용자가 “로그인 필요”, “사건 보관 처리됨”, “접근 권한 없음”부터 본다
- closed, archived, completed, reopen, print, export 변형이 거의 비슷한 URL을 대량으로 만든다
- 실제 검색 의도를 받아야 할 설명 페이지, FAQ, 규정 페이지는 약한 상태로 남는다
- 검색엔진은 공개 답변 페이지가 아니라 내부 프로세스 페이지를 크롤링한다

그래서 **결안 페이지를 어떻게 처리할지** 고민할 때 핵심은 이거다.

**대부분의 결안 페이지는 공개 SEO 랜딩 페이지가 아니라 프로세스 종료 페이지이자 아카이브 페이지다. 운영상 중요할 수는 있지만, 그 자체가 검색 유입용 페이지라는 뜻은 아니다.**

## 결안 페이지는 실제로 무슨 역할을 할까?

그냥 “이 사건은 끝났습니다”라고 알려주는 페이지가 아니다.

### 1. 이미 프로세스 안에 들어온 사용자를 위한 페이지다

주요 목적은 보통 이렇다.

- 사건이 공식적으로 종료됐는지 알려준다
- 지급 완료 종료인지, 거절 종료인지, 자동 종료인지, 자료 부족 종료인지 보여준다
- 다시 열 수 있는지, 추가 증빙이 가능한지, 고객센터로 가야 하는지 알려준다
- 사용자, CS, 물류사, 플랫폼이 같은 최종 상태를 보게 만든다

### 2. 비공개 데이터와 강하게 연결돼 있다

예를 들면:

- 주문번호
- 운송장 번호
- 클레임 ID
- 종료 사유
- 종료 시각
- 지급 상태
- 재오픈 가능 여부

이런 구조라서 공개 검색용으로 안정적인 페이지가 되기 어렵다.

### 3. 운영 가치는 있지만, SEO 가치와 같지는 않다

사용자가 검색하는 건 보통 이런 것이다.

- 왜 내 클레임이 종료됐는가
- 종료 후에도 이의 제기가 가능한가
- 종료 후 보상금은 언제 들어오는가
- 자동 종료는 무슨 뜻인가
- closed와 archived는 같은가

이런 검색 의도는 보통 다음 같은 공개 페이지가 더 잘 받는다.

- 결안 설명 페이지
- 종료 사유 설명 페이지
- 지급 FAQ
- 재오픈 규정 페이지
- 사후 처리 상태 설명 페이지

## 결안 페이지는 이렇게 처리하는 편이 낫다: 먼저 이 5가지를 나눠 보자

### 1. 일반적인 결안 페이지, 아카이브 페이지, closed 페이지라면 SEO 핵심 페이지로 밀 필요가 없는 경우가 많다

이런 페이지는 보통:

- 로그인 필요
- 주문번호나 케이스 ID 필요
- 하나의 구체적인 사건에만 연결됨
- 상태가 바뀌면 내용도 바뀜
- 외부 사용자에게 안정적인 읽을거리가 아님

### 2. 실제 검색 가치가 “왜 종료됐는가”, “다음엔 뭘 해야 하는가”, “재오픈 가능한가”라면 그 의도를 비공개 페이지에 맡기지 말아야 한다

더 적합한 건 공개 페이지다.

- 종료 사유 안내
- 종료 후 다음 단계 안내
- 재오픈 규정
- 지급 시점 FAQ
- closed 와 archived 차이 설명

### 3. `closed`, `closure`, `archived`, `completed`, `print`, `export` 같은 변형 URL을 같이 관리해야 한다

문제는 한 페이지가 아니라 거의 같은 페이지 묶음이 계속 늘어나는 것이다.

### 4. 공개 설명 페이지와 비공개 결안 페이지가 둘 다 있다면 역할을 완전히 분리해야 한다

“case closed가 무슨 뜻인가”를 설명하는 공개 페이지와 실제 사건의 결안 페이지는 완전히 다르다.

**하나는 질문에 답하는 페이지이고, 다른 하나는 프로세스를 진행하는 페이지다. 섞이면 안 된다.**

### 5. 랭킹시킬 생각이 없다면 noindex, 로그인 제어, sitemap, canonical, 내부 링크를 함께 정리해야 한다

자주 보이는 실수는 이렇다.

- noindex를 넣었는데 sitemap에는 계속 제출함
- 본문은 로그인 필요인데 print나 export는 공개됨
- canonical이 애매함
- 도움말 센터나 내부 검색이 계속 이런 페이지를 노출함

## 결안 페이지에서 자주 보는 SEO 실수 4가지

### 1. “최종 결과 + 날짜 + 금액”이 있으니 검색에도 좋을 거라고 생각한다

그렇지 않다.

검색 사용자가 원하는 건 이유, 설명, 다음 행동이다.

### 2. 결안 페이지, 상세 페이지, 아카이브 페이지, 인쇄 페이지, export 페이지, 재오픈 페이지가 같이 노출된다

그러면 얇고 비슷한 페이지가 한꺼번에 늘어난다.

### 3. 중요한 정보를 로그인 뒤에만 숨겨 둔다

예를 들면:

- 왜 자동 종료됐는지
- 지급까지 얼마나 걸리는지
- 언제 재오픈이 가능한지
- 추가 이의 제기가 가능한지

이런 정보는 공개 페이지로 분리하는 편이 SEO에 훨씬 유리하다.

### 4. “이 페이지로 SEO 안 한다”고 말하면서 기술 신호는 전혀 다르게 보낸다

sitemap, 템플릿, 내부 링크가 제각각이면 검색엔진은 혼란스러울 수밖에 없다.

## 지금 바로 점검한다면 이 순서가 좋다

### 1단계: 결안 관련 URL 유형을 전부 뽑아 본다

결안, 아카이브, 상세, print, export, reopen을 함께 본다.

### 2단계: 각 페이지가 구체적인 사건용인지, 공개 질문용인지 판단한다

주문이나 사건, 사용자 없이 의미가 없으면 프로세스 페이지일 가능성이 크다.

### 3단계: 실제 검색 수요를 공개 페이지로 분리한다

우선순위는 보통 이렇다.

- 결안 설명 페이지
- 자동 종료 사유 페이지
- 지급 FAQ
- 재오픈 규정 페이지
- 종료 후 이의 제기 가이드

### 4단계: 기술 신호를 통일한다

noindex, 로그인 제어, sitemap, canonical, 내부 링크를 확인한다.

### 5단계: “결안 페이지가 인덱싱됐는가”만으로 성패를 보지 않는다

더 중요한 건:

- 공개 설명 페이지가 트래픽을 받는지
- 사용자가 맞는 페이지로 착지하는지
- CS 반복 문의가 줄었는지
- 결안 관련 파생 페이지가 대량으로 나타나지 않는지

## 그럼 결안 페이지는 인덱싱해야 할까?

**내 답은 단순하다.**

- 비공개 프로세스 페이지라면 SEO 핵심 페이지로 다루지 말 것
- 결안, 아카이브, export 페이지라면 적극적으로 인덱싱을 밀 필요가 없는 경우가 많다
- 유기적 검색 유입이 필요하다면 종료 사유, 지급 시점, 재오픈 규정, 상태 설명을 공개 페이지로 따로 만들 것
- 공개 지식 베이스처럼 예외적인 경우에만 결안 관련 페이지 자체가 랜딩 페이지 역할을 할 수 있다

그래서 **결안 페이지를 어떻게 처리할까** 고민 중이라면 내 조언은 이거다.

**먼저 프로세스 종료 페이지로 보고, 그다음에 인덱싱 여부를 결정하라. 존재한다는 이유만으로 결안 페이지, 아카이브 페이지, closure 파라미터 URL을 검색결과로 밀어 넣지 않는 편이 낫다.**
