## 商品ページやヘルプページは整っているのに、なぜ ledger や history や statement の URL が検索結果に出てしまうのか

これは珍しい話ではありません。

理赔補件撤回のフローをシステムに組み込むと、次のような URL が自動生成されやすいです。

- `/claim/document-withdraw-ledger`
- `/claim/document-withdraw-transactions`
- `/claim/document-withdraw-history`
- `/wallet/claim-withdraw-record`
- `/finance/document-withdraw-statement`
- `/claim/document-withdraw-ledger?case=xxx`

多くのチームはこう考えます。

- 金額、日時、ステータスがあるから薄いページではない
- ユーザーは実際に「補償記録をどこで見るか」を検索する
- ledger ページは result ページより情報量が多く見える
- システムが自動生成したのでそのまま残してしまう

でも時間がたつと、**こうしたページの大半は公開検索向けではなく、特定案件の記録・照合ページでしかない**ことがわかります。

検索ユーザーが本当に知りたいのは、たとえば次のようなことです。

- 補償記録はどこで確認できるか
- なぜ同じ案件で複数の記録が出るのか
- 支払い成功なのに ledger がまだ更新されないのはなぜか
- reversal、frozen、pending は何を意味するのか

そのため SEO 的には、私有の ledger URL を出すより、公開の説明ページを作るほうが適切なことが多いです。

## このページは実際に何を解決しているのか

### 1. 現在のユーザーが金融記録を確認するためのページである

通常は次の情報を確認させる役割があります。

- 補償が計上されたか
- 金額、通貨、日時、ステータス
- ウォレット入金か、元の支払い経路への返金か、内部残高反映か
- どの案件にどの記録が対応するか

つまり Google から初めて来る人向けではなく、すでにフローに入っている人向けです。

### 2. 案件ID、アカウント権限、記録番号に強く依存する

こうしたページには通常、

- claim ID や order ID
- ledger 番号や statement 番号
- 金額、調整額、手数料
- 作成日時、入金日時、更新日時
- 権限チェック
- 財務同期ステータス

などが含まれます。情報が具体的であるほど、公開検索には向きません。

### 3. 業務上の価値は高いが、SEO 向きとは限らない

多くのチームは、

- プロセス内で重要
- 情報量が多い
- エクスポートや次の導線がある

という理由で SEO 価値も高いと思いがちです。

しかし SEO で重要なのは、公開の検索意図に安定して答えられるかどうかです。

## どう扱うべきか。まず 5 つのケースに分ける

### 1. 単なる ledger、transaction、statement detail ページなら、通常は主力 SEO ページにしない

このタイプのページは多くの場合、

- 特定案件にひもづく
- 単一記録を中心に構成される
- アカウント文脈がないと理解しにくい
- 公開解説ではなくプロセス用ページである

要するに、**標準的な流水ページは記録ページであって、公開コンテンツページではありません。**

### 2. 本当の検索意図が「見方」や「金額変動の理由」なら、私有 URL をランクさせようとしない

ユーザーが欲しいのは private URL ではなく、公開の説明です。

### 3. もしそのページ自体が公開ガイドなら、個別に評価する

ledger や history や statement を含むページを全部ブロックすべきとは限りません。ログイン不要で役立つ公開ページなら、評価対象になります。

### 4. ledger、history、statement、detail、export の派生 URL があるなら、まとめて制御する

問題は 1 ページではなく、似た URL のクラスターであることがほとんどです。

### 5. ランキングさせたくないなら、シグナルを全部そろえる

確認すべき項目は次の通りです。

- noindex
- ログイン制御
- sitemap
- canonical
- 内部リンク
- パラメータ URL

## よくある 4 つの SEO ミス

### 1. 項目数が多いほど SEO 価値が高いと思い込む

金額、日時、ステータスがあっても、それだけで公開検索価値になるわけではありません。

### 2. ledger、history、statement、detail、export をまとめてインデックスさせる

重複ページ、弱いページ、混乱したクロールシグナルが増えます。

### 3. 本当に必要な公開説明ページを作らない

「補償記録の見方」や「なぜ複数記録が出るのか」といった公開ページのほうが、私有 URL より機能します。

### 4. ledger SEO は不要と言いながら、sitemap やテンプレートやユーザーセンターが URL を出し続ける

多くの場合、問題はページ自体ではなく、周辺シグナルの矛盾です。

## まとめ

**本当に問うべきなのは、そのページにデータが多いかではなく、私有の案件を処理しているのか、公開検索に答えているのかです。**
