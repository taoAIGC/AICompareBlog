## なぜ多くの越境ECサイトでは、本来検索流入を取るべきヘルプページやルールページではなく、結案ページ・アーカイブページ・closed系URLのほうが表に出やすくなるのか？

かなりよくある話です。

紛失補償、破損補償、アフターサービスの争議、申诉、仲裁などのフローをサイトで回し始めると、システムは次のようなURLを増やしがちです。

- `/claim/closed`
- `/claim/closure`
- `/after-sales/closed`
- `/case/closed`
- `/appeal/closure`
- `/arbitration/closed`
- `/claim/closure?caseId=20260327032`

さらに、結案ページ、アーカイブページ、案件完了ページ、印刷ページ、エクスポートページ、結案確認ページまで増えていきます。

チーム側は最初こう考えがちです。

- 正式なページだから、インデックスされても問題ない
- 「クレームがclosedとは何か」「結案後はどうするか」と検索する人はいる
- 結果、日時、金額、ステータスが出ていて内容もそれなりにある
- システムが自動生成したものだから、整理はあとでやればいい

でも時間がたつと、次の問題が出ます。

- 具体的な注文、案件、ユーザーに強く依存している
- Googleから来た人が最初に見るのが「ログインしてください」「案件はアーカイブ済みです」「アクセス権がありません」になる
- closed / archived / completed / reopen / print / export などの違いだけで近いURLが大量発生する
- 本来検索意図を受けるべき説明ページやFAQやルールページが弱いままになる
- 検索エンジンが公開回答ページではなく、内部フローのページをクロールしてしまう

だから **結案ページをどう扱うか** を考えるときの前提はこれです。

**多くの結案ページは、業務フローを閉じるためのページであって、公開SEOランディングページではありません。運用上は大事でも、そのまま検索流入を取りにいくページだとは限りません。**

## 結案ページは何のためにあるのか？

単なる「この案件は終わりました」と書いてあるページではありません。

### 1. 役割の中心は、すでに案件の中にいるユーザーを案内すること

主に次のためにあります。

- その案件が正式に終了したかを伝える
- 支払い完了で終わったのか、却下で終わったのか、自動クローズなのか、資料不足で閉じたのかを示す
- 再申請できるのか、追加証拠を出せるのか、サポートに進むべきかを示す
- ユーザー、CS、物流会社、プラットフォームが同じ最終状態を見るようにする

### 2. 注文情報や非公開データと強く結びついている

たとえば：

- 注文番号
- 追跡番号
- クレームID
- 結案理由
- 結案日時
- 支払い状況
- 再オープン可否

この時点で、安定した公開検索向けページになりにくいのはほぼ決まっています。

### 3. 業務価値はあるが、SEO価値があるとは限らない

検索ユーザーが本当に知りたいのは、むしろ次のようなことです。

- なぜclosedになったのか
- 結案後も申诉できるのか
- 支払いはいつ入るのか
- 自動クローズとは何か
- closedとarchivedは同じ意味なのか

こうした検索意図を受けるのに向いているのは、たいてい次のような公開ページです。

- 結案の説明ページ
- 結案理由の説明ページ
- 支払いFAQ
- 再オープン条件ページ
- アフターサービスの状態説明ページ

## 結案ページはどう扱うべきか？ まずはこの5つに分けて考えたほうがいい

### 1. 標準的な結案ページ、アーカイブページ、closedページなら、たいていはSEOの主役にしないほうがいい

こうしたページはたいてい：

- ログイン必須
- 注文番号や案件IDが必要
- 1件の案件にだけ結びつく
- ステータス変更で中身も変わる
- 外部ユーザーにとって安定した閲覧価値が低い

つまり、通常はSEOページというよりフローページです。

### 2. 検索価値があるのが「なぜ閉じたのか」「次に何をするのか」「再オープンできるのか」なら、その意図を私有ページに背負わせない

本当に作るべきなのは、たとえば：

- 結案理由ページ
- 結案後の次の手順ページ
- 再オープン条件ページ
- 支払いタイミングFAQ
- closed と archived の違いを説明するページ

です。

### 3. `closed` `closure` `archived` `completed` `print` `export` のような派生URLがあるなら、まとめて管理する

問題は1つのURLではなく、似たページがまとまって増えることです。

### 4. 公開説明ページと私有結案ページが両方あるなら、完全に役割を分ける

「case closed とは何か」を説明する公開ページと、具体的な案件の結案ページは別物です。

**前者は質問に答えるページ、後者は案件を進めるページです。混ぜないほうがいいです。**

### 5. ランクさせるつもりがないなら、noindex、ログイン制御、sitemap、canonical、内部リンクを揃える

よくあるミスは：

- noindex なのに sitemap で送っている
- 本体はログイン必須なのに print や export は公開されている
- canonical が曖昧
- ヘルプセンターや内部検索から継続的に露出している

## 結案ページでよくあるSEOの失敗4つ

### 1. 「最終結果・日時・金額があるから、検索にも強いはず」と思ってしまう

そうとは限りません。

検索ユーザーが欲しいのは、理由、解釈、次の行動です。

### 2. 結案ページ、詳細ページ、アーカイブページ、印刷ページ、エクスポートページ、再オープンページが一緒に出てしまう

これで薄いページ群が一気に増えます。

### 3. 本当に価値のある情報をログイン後ページの中に閉じ込めてしまう

たとえば：

- なぜ自動クローズになったか
- 支払いまでどれくらいかかるか
- 再オープン条件は何か
- まだエスカレーションできるのか

こうした内容は公開ページに切り出したほうが検索に強くなります。

### 4. 「このページでSEOはやらない」と言いながら、技術シグナルがバラバラ

sitemap、テンプレート、内部リンクがそれぞれ別のことを言っていたら、検索エンジン側は混乱します。

## もし今すぐ結案ページを点検するなら、この順番で見るのがおすすめ

### 1段階目：結案関連URLの種類を全部出す

結案、アーカイブ、詳細、print、export、reopen を含めて一覧化します。

### 2段階目：そのページが「具体案件向け」か「公開質問向け」かを判断する

注文や案件やユーザーがないと成立しないなら、ほぼフローページです。

### 3段階目：本当の検索需要を公開ページに分離する

優先したいのは：

- 結案説明ページ
- 自動クローズ理由ページ
- 支払いFAQ
- 再オープン条件ページ
- 結案後の申诉ガイド

### 4段階目：技術シグナルを統一する

noindex、ログイン制御、sitemap、canonical、内部リンクを確認します。

### 5段階目：「結案ページが収録されたか」だけで成否を見ない

見るべきなのは：

- 公開説明ページが流入を取れているか
- ユーザーが正しいページに着地しているか
- CSへの同じ質問が減っているか
- 結案系の派生ページが大量に出なくなったか

## では、結案ページは収録させるべきか？

**私の答えはシンプルです。**

- 私有フローページなら、SEOの主役にしない
- 結案ページ、アーカイブページ、exportページのようなものなら、積極的に収録を狙わないことが多い
- 検索流入が欲しいなら、理由説明、支払い時期、再オープン条件、状態解説を別の公開ページにする
- 例外的に、公開ナレッジベースや公開ヘルプセンターとして作り込んでいる場合だけ、結案関連ページ自体が検索ランディングになることがある

だから **結案ページはどう扱うか** で迷っているなら、私の答えはこうです。

**まずは案件を閉じるためのフローページとして扱う。そのうえで、本当に公開検索意図を受けるべきかを判断する。存在するからという理由だけで、結案ページやアーカイブページや closure パラメータURL を検索結果に流し込まないほうがいいです。**
