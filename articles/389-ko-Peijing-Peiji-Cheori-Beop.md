## 많은 웹사이트에서 '2페이지와 3페이지'를 볼 때 첫 번째 반응은 왜 모두 색인을 생성하지 않는 것일까요?

이것은 매우 흔한 일입니다.

특히 콘텐츠 사이트, 전자상거래 사이트, 도구 사이트에서 작업하는 사람들은 백그라운드에서 팝업되는 여러 가지 항목을 보면 다음과 같습니다.

-`/카테고리/페이지/2`
- `/blog?page=3`
- `/태그/seo/페이지/4`
- `?페이지=5`

첫 번째 반응은 일반적으로 이러한 페이지가 가치 있는지 확인하는 것이 아니라 성급하게 결론을 내리는 것입니다.

- 이 페이지는 내용이 얇습니다.
- 이 페이지는 중점 페이지가 아닙니다.
- 이 페이지가 중복되나요?
- 이 페이지들이 크롤링을 낭비하게 될까요?
- NOINDEX를 통일하면 어떨까?

표면적으로 이것은 매우 "쉬운" 것처럼 보입니다.

하지만 많은 웹사이트의 문제는 여기서부터 시작됩니다.

페이지 매기기 페이지는 본질적으로 품질이 낮은 페이지가 아니기 때문에 본질적으로 목록 구조의 일부입니다. **

차이점을 전혀 모르는 경우:

- 동일한 주제 목록을 계속 이어가는 페이지 매기기 페이지
- 실제로 일반 페이징이 아닌 매개변수 페이지는 무엇입니까?
- 어떤 깊은 페이지 매김이 실제로 가치가 낮습니까?
-프론트 엔드에 "추가 로드"만 되어 있고 검색 엔진에 공식적인 입구가 전혀 없는 페이지는 무엇입니까?

그런 다음 전반적으로 NOINDEX를 적용하면 결국 "정크 페이지 제거"가 아니라 다음과 같이 됩니다.

- 더 깊은 콘텐츠를 위해 크롤링 경로를 차단하세요.
- 롱테일 트래픽을 수신할 수 있는 목록 페이지가 실수로 손상되었습니다.
- 표준, NOINDEX 및 매개변수 규칙이 서로 싸우게 합니다.
- 검색 엔진이 사이트 구조를 이해하기 어렵게 만듭니다.

따라서 최근에 **페이지 매김 처리 방법**을 공부하고 계시다면 매우 직접적인 결론을 내리겠습니다.

**페이지 매김 페이지를 포함할지 최적화할지 여부는 "2페이지"인지 여부에 따라 달라지지 않고 목록 주제, 콘텐츠 구성 및 크롤링 값을 계속 수행할 수 있는지 여부에 따라 결정됩니다. **

## 페이지 매김 페이지는 어떤 문제를 해결하고 있나요? "낮은 품질의 꼬리"가 아니라 목록 구조의 일부입니다.

많은 사람들이 페이지 매김이라고 하면 자동으로 다음을 생각합니다.

- 중복된 페이지
- 얇은 콘텐츠 페이지
- SEO부담

이것은 실제로 절반만 사실입니다.

페이지가 매겨진 페이지는 실제로 중복, 분산 크롤링, 가중치 희석 등의 문제를 일으킬 수 있지만 웹사이트 콘텐츠나 제품 목록을 영원히 한 페이지에 담을 수 없기 때문에 문제가 발생합니다.

최소한 다음 몇 가지 사항을 해결합니다.

### 1. 보다 심층적인 콘텐츠나 제품에 안정적으로 접근할 수 있도록 허용

카테고리에 기사 80개, 상품 120개가 있는데 페이징을 하지 않으면 사용자 경험이 먼저 무너집니다.

검색 엔진에서도 마찬가지입니다.

첫 페이지 너머에 내용이 있다는 것을 알아야 합니다.

### 2. 이 페이지는 관련 없는 페이지가 아니라 연속된 목록의 집합이라는 점을 검색 엔진에 알려주세요.

건전한 카테고리 페이지 구조를 위해서는 일반적으로 홈페이지가 가장 중요할 뿐만 아니라 다음의 페이지네이션도 동일한 주제를 공유합니다.

예를 들면:

- SEO 카테고리 페이지 2, 3페이지
- 특정 제품 카테고리에 대한 후속 제품 목록
- 특정 주제에 관한 역사적 기사의 아카이브 페이지

전혀 의미가 없는 페이지는 아닙니다.

### 3. 사용자에게 안정적인 탐색 경로 제공

특히 콘텐츠 사이트와 전자상거래 사이트에서는 많은 사용자가 계속해서 페이지를 넘기게 됩니다.

SEO 관점에서 이러한 페이지를 "존재하지 않는 것이 더 낫다"고 취급하면서 사용자가 이러한 페이지를 넘기게 할 수는 없습니다.

그래서 **페이지 매김 SEO**의 핵심 이해가 무엇인지 묻는다면 다음과 같이 대답하겠습니다.

**페이징은 순위 경쟁만을 위해 존재하는 것이 아닙니다. 이는 웹 사이트가 콘텐츠를 구성하고 경로를 크롤링하며 탐색 경험을 유지하도록 돕는 것에 관한 것입니다. **

단순히 '모두 수용'이나 '모두 금지'만으로는 문제가 해결될 수 없는 이유다.

## 페이지 매김 페이지를 처리하는 방법은 무엇입니까? 먼저 이 5가지 상황을 별도로 처리하는 것이 좋습니다.

### 1. 카테고리 페이지, 컬럼 페이지 등 명확한 테마로 페이지네이션을 하되, 처음부터 noindex를 사용하지 마세요.

이는 사고로 인한 부상의 가능성이 가장 높은 유형입니다.

예를 들어 귀하의 웹사이트에는 다음이 포함되어 있습니다.

- SEO 기사 목록
- AI 도구 추천 목록
- 제품 카테고리 페이지
- 도움말 문서 콘텐츠 페이지

이러한 페이지의 페이지 매김은 본질적으로 동일한 테마 컬렉션의 연속입니다.

많은 사람들은 첫 페이지가 아닌 다른 내용을 볼 때 조건부 반응을 보입니다.

- 2페이지는 포함하지 않아도 됩니다.
- 3페이지 이후는 더욱 가치가 없습니다.
- 모든 NOINDEX는 사용하기 가장 쉽습니다.

그러나 실제로는 많은 웹사이트의 두 번째와 세 번째 페이지가 생각만큼 "낭비"가 아닙니다.

특히 다음과 같은 상황:

- 목록 자체에는 명확한 주제가 있습니다.
- 각 페이지의 콘텐츠가 관련성이 있음
- 사용자는 실제로 계속 탐색하게 됩니다.
- 다음 콘텐츠에도 롱테일 가치 페이지가 있습니다.
- 사이트 내 링크와 로컬 탐색은 계속해서 후속 페이지 매김을 가리킵니다.

이때, NOINDEX를 전면적으로 적용하게 되면 리스트 링크 전체가 보기 흉한 상태로 잘릴 가능성이 매우 높습니다.

개인적으로 다음 질문을 먼저 물어보는 것이 좋습니다.

- 이 페이지 매김은 여전히 동일한 명시적인 주제를 제공합니까?
- 사용자가 이 주제를 검색할 때 후속 목록 페이지를 방문할 의향이 있습니까?
- 이 페이지에 계속 액세스하여 콘텐츠를 배포할 수 있나요?

대답이 일반적으로 "예"인 경우 noindex로 서두르지 마십시오.

많은 페이지 매기기 페이지에 대한 실제 솔루션은 "사라지게 만드는 것"이 아니라 다음을 수행하는 것입니다.

- 목록이 너무 지저분한가요?
- 제목과 설명이 너무 형식적인가요?
- 페이지의 가독성이 너무 좋지 않나요?
- 내부 링크는 홈페이지에만 집중하고 다음 페이지는 무시하나요?

모든 구조적 문제를 noindex에 맡기지 마십시오.

### 2. "추가 로드" 또는 무한 스크롤을 수행하는 경우 검색 엔진에 실제 크롤링 가능한 페이지 매김 URL을 제공해야 합니다.

이러한 유형의 문제는 최근 몇 년 동안 특히 흔해졌습니다.

경험상 많은 프런트 엔드는 목록을 다음과 같이 만듭니다.

- 아래로 당기면 자동으로 로드됩니다.
- 콘텐츠 로드를 계속하려면 "더 보기"를 클릭하세요.
- 끝까지 무한 스크롤

사용자에게는 편리할 수 있습니다.

그러나 검색 엔진의 경우 중요한 것은 프런트 엔드 애니메이션이 아름다운지 여부가 아니라 다음과 같습니다.

**후속 콘텐츠에 대해 안정적이고 액세스 가능하며 크롤링 가능한 URL이 있습니까? **

그렇지 않으면 다음과 같은 문제가 발생합니다.

- 사용자는 두 번째 화면에서 콘텐츠를 볼 수 있지만 크롤러는 콘텐츠에 접근할 수 없습니다.
- 프런트 엔드는 계속 로드될 수 있으며, 검색 엔진은 첫 페이지만 인식합니다.
- 아래 글과 제품은 이론적으로는 온라인에서 구할 수 있으나 실제로는 구할 수 없습니다.

따라서 귀하의 웹사이트가 현재 무한 스크롤 기능을 갖추고 있다면 특히 다음 사항을 확인해 보시기 바랍니다.

- `/page/2` 또는 `?page=2`와 같은 공식적인 페이징 주소가 있습니까?
- 프런트 엔드 스크립트에 의존하지 않고 이러한 페이징 URL을 정상적으로 열 수 있습니까?
- 이후 페이지 사이에는 정상적인 링크 관계가 있습니까?

많은 사람들이 **목록 페이지 최적화**에 대해 이야기하고 있지만 실제로는 "다음 페이지에 공식 입구가 있는지 여부"조차 알려주지 않았습니다.

이 경우 페이지 매김을 최적화하는 것이 아니라 페이지 매김을 숨기는 것입니다.

그런데 `rel="next"` / `rel="prev"` 태그는 보조 정보로 사용될 수 있지만 이를 구원자로 취급하지 마십시오. 실제 핵심은 URL이 크롤링 가능하고 구조가 명확하며 링크에 액세스할 수 있다는 것입니다.

### 3. 처음부터 독립적으로 존재해서는 안 되는 경우를 제외하고 모든 페이지 매김 페이지를 첫 번째 페이지로 정규화합니다.

이 구덩이는 정말 여러 번 깨졌습니다.

많은 사이트의 처리 방법은 매우 간단하고 조잡합니다.

- 2페이지 표준 1페이지로 이동
- 3페이지 표준 1페이지로 이동
- 모든 페이지 매김은 첫 번째 페이지를 가리킵니다.

그들의 생각은 대개 다음과 같습니다.

"이것은 가장 집중되어 있으며 무게를 분산시키지 않습니다."

합리적으로 들리지만 실제로 검색 엔진에 다음과 같은 내용을 전달하는 경우가 많습니다.

> 이후의 페이지 매김은 중요하지 않으며 첫 번째 페이지만 읽어야 합니다.

문제는 페이지 2와 페이지 3이 원래 서로 다른 콘텐츠 배치를 호스팅하고 이를 모두 첫 번째 페이지로 정규적으로 반환하는 경우 이는 해당 페이지의 의미를 적극적으로 지우는 것과 동일하다는 것입니다.

더욱 문제가 되는 것은 일반적인 결과가 다음과 같은 경우가 많다는 것입니다.

- 뒷 페이지의 내용을 찾기가 더 어렵습니다.
- 페이징 논리와 표준 신호가 서로 싸웁니다.
- 검색 엔진은 목록 구조에 대한 이해가 약합니다.

따라서 표준은 첫 번째 페이지를 다시 가리켜야 하며 기계적으로 처리할 수 없습니다.

내 판단 기준은 다음과 같습니다.

- **일반 페이지 매기기 목록**: 페이지 매김 자체 참조 표준에 우선순위를 부여하여 각 탭이 고유한 형식적 정체성을 유지할 수 있도록 합니다.
- **순수 매개변수 복사 페이지**: 동일한 콘텐츠의 표시 방법만 변경하는 경우 중앙 집중식 신호를 고려합니다.
- **전혀 노출되어서는 안 되는 페이지**: 페이징 로직을 하드 래핑하지 마세요.

한 문장으로:

**"페이지 매김"을 "반복"의 동의어로 생각하지 마십시오. 페이징 페이지와 매개변수 복사 페이지는 완전히 다릅니다. **

### 4. 정렬 페이지, 필터링 페이지, 매개변수 페이지 및 실제 페이징 페이지는 별도로 관리해야 하며, 이를 하나의 포트에 혼합하지 마십시오.

이는 특히 하나의 카테고리처럼 보이기 쉽지만 실제로는 전혀 동일한 카테고리가 아닌 기술적 SEO의 문제이기도 합니다.

예를 들어 다음 URL은 다음과 같습니다.

-`/카테고리/페이지/2`
- `/category?page=2`
- `/category?sort=price_asc`
- `/category?color=black&page=3`
- `/category?brand=xx&size=l&page=4`

많은 팀에서는 이를 "페이지 매김 페이지"라고 통칭하고 균일하게 처리합니다.

그러나 실제로 여기에는 적어도 세 가지 범주가 혼합되어 있습니다.

- 일반 목록 페이징
- 정렬 변경 페이지
- 필터 포트폴리오 페이지

SEO와는 완전히 다른 의미를 갖습니다.

일반 페이징, 일반적으로 명시적인 목록을 계속합니다.
페이지를 정렬하면 표시 순서만 변경되는 경우가 많습니다.
필터 페이지를 선택하면 값이 낮은 매개변수 조합이 갑자기 대량으로 나타날 수 있습니다.

이를 noindex, canonical 및 블록 로봇과 결합하면 결국 일어날 가능성이 가장 높은 일은 다음과 같습니다.

- 실제로 유지되어야 하는 페이지도 실수로 삭제됩니다.
- 대신 실제로 제어해야 할 매개변수 페이지가 누락되었습니다.
- 검색 엔진은 모순된 신호를 포착합니다.

보다 안정적인 접근 방식은 문제를 분리하는 것입니다.

#### 일반 페이징
먼저 명확한 목록을 수락할지 여부와 크롤링 및 색인 생성 값을 유지해야 하는지 여부를 확인하세요.

#### 정렬 페이지
먼저 정렬된 콘텐츠가 동일한 배치인지 확인하세요. 단순히 정렬만 변경하는 경우에는 일반적으로 독립 SEO 포털로 수행할 필요가 없습니다.

#### 필터 페이지
먼저 필터 조합에 실제 검색 수요가 있는지 확인하세요. 그렇지 않다면 무한정 확장하지 마십시오.

SEO는 종종 기술적인 어려움이 아니라 페이지 유형을 먼저 명확하게 구분했는지 여부에 관한 것입니다.

---

### 5. 딥 페이징의 가치가 정말 약할 때는 목록 구조와 크롤링 경로를 먼저 최적화하고, 전체적으로 NOINDEX를 사용하지 마세요.

이것도 중요합니다.

모든 페이지 매김이 포함되어야 한다고 말하는 것은 아닙니다.

일부 웹사이트의 딥 페이징은 실제로 약합니다.

예를 들면:

- 카테고리의 15페이지를 보면 내용이 확실히 오래된 것 같습니다.
- 상품 목록을 맨 뒤로 스크롤하면 사용자가 클릭을 계속하는 경우가 거의 없습니다.
- 페이지당 항목 수가 너무 적어서 페이징 수준이 매우 깊어집니다.
- 사이트 내의 내부 링크는 항상 첫 번째 페이지만 참조하며, 다음 페이지를 방문하는 사람은 거의 없습니다.

이때 딥 페이징의 성능이 좋지 않다고 느낀다면 반드시 틀린 것은 아닙니다.

그러나 문제는 여전히 다음과 같습니다.

**"다음 몇 페이지가 약함"을 "모든 항목에 대한 인덱스 없음"과 직접적으로 동일시하지 마세요. **

대부분의 경우 실제로 먼저 변경해야 할 사항은 다음과 같습니다.

- 페이지당 노출수가 너무 적나요?
- 카테고리가 너무 대략적으로 분류되어 목록이 너무 길어지나요?
- 주요 내용이 사이트에 미리 노출되었는가?
- 목록 정렬 논리가 가장 가치 있는 콘텐츠를 너무 멀리 묻어두나요?
- 토픽 집계, 하위 카테고리, 라벨 구성을 통해 페이징 수준을 더 얕게 만들 수 있나요?

다시 말하면:

딥 페이징 문제가 본질적으로 사이트 구조 문제인 경우 noindex는 증상만 다룰 뿐 근본 원인을 해결하지는 않습니다.

먼저 구조를 바로잡은 다음 어떤 페이지를 보관할 가치가 있는지, 어떤 페이지를 약화시켜야 하는지 결정하여 더 안정적으로 만들 수 있습니다.

---

## 내가 가장 자주 보는 4가지 페이징 SEO 실수는 "무시"하는 것보다 웹사이트를 엉망으로 만들 가능성이 더 높습니다.

### 1. 2페이지 이후의 모든 페이지는 noindexed 되어 있는데, 유일한 이유는 "반복의 두려움"입니다.

이것은 가장 일반적인 프리 사이즈입니다.

문제는 많은 사람들이 이러한 페이지가 정상적인 목록인지 여부조차 확인하지 않는다는 것입니다.

분명 주제가 명확한 카테고리와 페이지 매김이었는데 직접 삭제되기도 했습니다.

결국 반복 횟수가 줄어든 것이 아니라 목록 구조가 깨졌다는 것입니다.

### 2. 모든 페이지 매김은 첫 번째 페이지에 표준적으로 적용됩니다.

이는 가중치를 집중시키는 것처럼 보일 수 있지만 실제로는 후속 페이지 매김의 정체성을 지우는 경우가 많습니다.

특히 다음 페이지에 다른 콘텐츠가 포함되어 있는 경우 이 접근 방식은 크롤링 및 이해를 망칠 가능성이 높습니다.

### 3. 무한 스크롤만 가능, 크롤링 가능한 페이징 URL 없음

사용자는 읽을 수 있지만 검색 엔진은 읽을 수 없습니다.

이와 같은 웹사이트에서는 종종 "왜 다음 콘텐츠는 검색에 나오지 않는가"라고 궁금해하는데, 그 이유는 매우 간단합니다.

크롤러에게 입구를 전혀 제공하지 않았습니다.

### 4. Robots, NOINDEX, Canonical이 겹겹이 쌓여서 결국 주요 신호를 파악하지 못합니다.

예를 들면:

- 로봇에 의해 매개변수 페이지가 차단되었습니다.
- 페이지 매김 페이지에 NOINDEX가 추가되었습니다.
- 표준은 첫 번째 페이지도 가리킵니다.

심각해 보이지만 실제로는 쉽게 지저분해질 수 있습니다.

기술적인 SEO가 가장 두려워하는 것은 하나의 작업을 놓치지 않고 서로 싸우는 신호가 너무 많은 것입니다.

## 지금 웹사이트의 페이지 매기기 설정을 확인하고 싶다면 이 순서대로 진행하는 것이 좋습니다.

### 1단계: "실제 페이징"과 "매개변수 변형" 분리

최소한 먼저 구별하십시오.

- 일반 페이징
- 페이지 정렬
- 필터 페이지
- 검색결과 페이지
- 태그 또는 주제 집계 페이지

나중에 규칙을 어지럽히지 않도록 먼저 개체를 식별하십시오.

### 2단계: 2, 3페이지에 정상적으로 접근 및 크롤링이 가능한지 확인

하이라이트:

- URL이 실제로 존재합니까?
- 페이지가 일반 상태 코드를 반환하는지 여부
- 스크립트에 의존하지 않고 열 수 있나요?
- 페이지에 후속 페이지 매김으로 연결되는 안정적인 링크가 있습니까?

이 단계에서는 이미 많은 페이징 문제를 볼 수 있습니다.

### 3단계: canonical 및 noindex가 실수로 일반 페이징을 손상시켰는지 확인합니다.

다음에 특별한 주의를 기울이십시오:

- 모두 첫 번째 페이지로 돌아갈지 여부
- 템플릿에 NOINDEX가 일괄적으로 포함되어 있는지 여부
- 페이지 매김 페이지와 매개변수 페이지가 지나치게 두꺼운 규칙 세트를 공유하는지 여부

### 4단계: 사이트 구조가 깊은 콘텐츠를 너무 깊게 묻어두는지 확인

예를 들면:

- 페이지당 페이지 수가 너무 적습니다.
- 카테고리가 너무 광범위함
- 내부 링크는 항상 첫 페이지만 관리합니다.
- 중요한 내용은 항상 늦게 묻혀요

이러한 문제를 먼저 해결하지 않으면 나중에 어떻게 페이징 신호를 조정해도 불편할 것입니다.

### 5단계: 온라인 접속 후 크롤링, 색인 생성 및 실제 방문 페이지 성능 관찰

다음 현상에 중점을 둡니다.

- 검색엔진은 첫 페이지만 인식하나요?
- 백레이어 콘텐츠를 발견하기가 점점 더 어려워지고 있나요?
- 매개변수 페이지의 영역이 넓은가요?
- 갑자기 트래픽이나 카테고리 페이지 포함에 이상이 생겼나요?

결과를 보지 않고 규칙만 바꾸지 마세요.

특히 페이징과 같은 문제는 실제 데이터와 연계하여 되돌아볼 필요가 있다.

## 마지막 문장

페이지 매김의 진짜 문제는 "포함되어야 하는가?"라는 단어가 아닙니다.

하지만 먼저 명확하게 생각해야 합니다.

- 가치 있는 목록 테마를 계속 유지하는가?
- 일반 페이징인가요 아니면 매개변수 변형인가요?
- 후속 콘텐츠에 대한 실제 크롤링 입구가 있습니까?
- 캐노니컬, NOINDEX, 로봇은 서로를 훼손하는가?
- 딥 페이징이 약합니다. 페이지 자체의 문제인가요, 아니면 구조적인 문제인가요?

이러한 관계가 정리되면 페이지 매김은 실제로 그다지 무섭지 않습니다.

많은 웹사이트가 실패하는 이유는 페이지네이션이 너무 많아서가 아니라, 페이지네이션을 보고 당황하고, 마지막에 폭력적인 조작을 하여 정상적인 구조를 어지럽히기 때문입니다.

따라서 아직도 **페이지 매김을 처리하는 방법**으로 어려움을 겪고 계시다면 제가 제안하는 한 문장은 다음과 같습니다.

**2페이지를 먼저 숨길 생각을 하지 마시고, 먼저 숨겨야 할지부터 생각해보세요. 유지된 구조의 유지된 구조, 제어 매개변수의 제어 매개변수는 NOINDEX에 전적으로 의존하지 않습니다. **
