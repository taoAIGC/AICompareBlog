## 왜 많은 크로스보더 이커머스 사이트는 공개 도움말 페이지가 아니라 보완서류 재제출 페이지부터 색인되는 걸까?

이 케이스를 꽤 자주 본다.

상품 페이지, 카테고리, 블로그, FAQ, 정책 페이지까지는 잘 정리해 둔 팀도 클레임이나 보완서류 플로우를 붙이는 순간 이런 URL이 늘어나기 시작한다.

- `/claim/document-resubmit`
- `/claim/reupload-document`
- `/case/supplement-resubmit`
- `/after-sales/document-retry`
- `/appeal/material-resubmit`
- `/merchant/claim/retry-upload`

그리고 처음에는 이렇게 생각하기 쉽다.

- 실제로 사용자는 반려 후 재제출 방법을 검색한다
- 버튼과 다음 단계가 있으니 빈 페이지 같지 않다
- 이미 생성된 URL이니 롱테일 유입을 받을 수도 있다
- 실패 페이지나 종료 페이지보다 훨씬 유용해 보인다

겉으로 보면 그럴싸하다.

하지만 시간이 지나면 문제가 분명해진다. **대부분의 재제출 페이지는 결국 특정 케이스에 묶인 워크플로우 페이지다. 이 페이지의 역할은 현재 사용자에게 ‘수정해서 다시 제출할 수 있다’고 알려주는 것이지, 공개 검색 수요를 오래 받는 안정적인 SEO 랜딩 페이지가 아니다.**

검색 사용자가 진짜 알고 싶은 것은 보통 이런 것이다.

- 반려 뒤에도 다시 제출할 수 있는지
- 재제출 전에 무엇을 고쳐야 하는지
- 어떤 자료가 두 번째에도 자주 떨어지는지
- 다시 심사받는 데 얼마나 걸리는지
- 기한이 지나면 구제 방법이 있는지

즉, 실제 SEO 가치는 사적인 재제출 URL이 아니라 공개 가이드, 체크리스트, FAQ, 규정 설명 페이지에 있는 경우가 많다.

## 재제출 페이지는 실제로 어떤 문제를 해결하는가?

### 1. 핵심 역할은 현재 케이스를 계속 진행시키는 것이다

이 페이지는 보통 다음을 위해 존재한다.

- 다시 제출할 수 있는지 알려준다
- 어떤 파일, 스크린샷, 설명을 수정해야 하는지 보여준다
- 재업로드나 재확인의 진입점을 제공한다
- 사용자, 심사자, 고객센터가 같은 상태를 보게 한다

즉 Google에서 처음 들어온 사람보다 이미 클레임 플로우 안에 있는 사람을 위한 페이지다.

### 2. 케이스 문맥 의존도가 매우 높다

자주 포함되는 정보는 다음과 같다.

- 주문번호 또는 클레임 번호
- 직전 반려 사유
- 재제출 마감일
- 남은 제출 가능 횟수
- 반드시 교체해야 하는 자료
- 고객센터 또는 메시지 센터 링크

이처럼 구체적일수록 공개 검색 랜딩 페이지로는 맞지 않는다.

### 3. 운영 가치가 높다고 SEO 가치가 높은 것은 아니다

많은 팀이 여기서 헷갈린다.

- 현재 사용자에게 중요하니 검색에도 중요할 것이다
- 버튼과 동작이 있으니 색인할 가치가 있다
- 플로우 안에서 필요하니 검색엔진에도 보여야 한다

하지만 SEO에서 중요한 질문은 다르다. **이 페이지가 많은 사람에게 반복적으로 도움이 되는 공개적이고 안정적인 문제를 해결하느냐**이다.

대부분의 재제출 페이지는 그렇지 않다.

## 그럼 어떻게 처리해야 할까? 먼저 5가지 경우로 나누는 편이 좋다

### 1. 일반적인 재제출 / 재업로드 페이지라면 보통 핵심 SEO 페이지로 보지 않는 것이 맞다

이런 페이지는 대체로 다음 특징을 가진다.

- 특정 케이스에 묶여 있다
- 수정한 자료를 다시 제출하도록 만든다
- 계정이나 주문 문맥이 없으면 이해하기 어렵다
- 공개 설명 페이지가 아니라 플로우 단계다
- 장기 검색 가치는 약하다

한마디로 **표준 재제출 페이지는 공개 콘텐츠 페이지가 아니라 워크플로우 페이지다.**

### 2. 실제 검색 수요가 ‘반려 후 어떻게 다시 제출하나’라면 그 의도는 비공개 페이지가 아니라 공개 페이지로 받아야 한다

사용자는 `/claim/document-resubmit?id=xxx` 같은 URL을 원하지 않는다.

대신 보통 이런 것을 알고 싶어 한다.

- 먼저 무엇을 확인해야 하는지
- 어떤 자료가 두 번째에도 자주 떨어지는지
- 새 사진, 새 설명, 새 증빙이 필요한지
- 다음 심사까지 얼마나 걸리는지
- 언제 고객센터에 먼저 문의해야 하는지

이런 검색 의도는 공개 도움말 페이지, FAQ, 정책 페이지, 체크리스트가 맡아야 한다.

### 3. 페이지 자체가 공개적이고 안정적인 재제출 가이드라면 별도로 색인 여부를 평가할 수 있다

`resubmit` 가 들어간 URL이라고 모두 막아야 하는 것은 아니다.

예를 들면 이런 페이지다.

- 반려 후 재제출 방법을 설명하는 공개 가이드
- 2차 제출용 공식 체크리스트
- 재제출 시 자주 생기는 실수를 정리한 도움말 페이지
- 재업로드 전 확인사항을 설명하는 공개 페이지

그 페이지가

- 특정 계정이나 케이스 없이도 이해 가능하고
- 목적과 범위가 명확하며
- URL 이 안정적이고
- 실제 검색 의도와 맞고
- 처음 방문한 사용자에게도 독립적인 가치가 있다면

별도로 평가할 수 있다.

### 4. 시스템이 rejected, resubmit, upload, receipt, review URL을 함께 만들면 묶어서 관리해야 한다

문제는 대개 한 URL만의 문제가 아니다.

보통 같이 생기는 URL은 이런 식이다.

- `/claim/document-rejected`
- `/claim/document-resubmit`
- `/claim/document-upload`
- `/claim/document-receipt`
- `/claim/document-under-review`
- `/claim/document-resubmit?case=xxx`

그러면 이런 문제가 생긴다.

- 본문은 거의 비슷하고 상태만 다르다
- rejected, resubmit, upload 가 함께 색인된다
- 메일, 메시지 센터, 내부 링크가 계속 이 URL들을 밀어준다
- 검색엔진이 어떤 URL을 남겨야 할지 헷갈린다

그래서 **재제출 페이지만 보지 말고 관련 클러스터 전체를 봐야 한다.**

### 5. 이런 페이지를 랭킹시키고 싶지 않다면 noindex, 로그인 보호, sitemap, canonical, 파라미터, 캐시를 함께 맞춰야 한다

실제 현장에서는 신호가 서로 충돌하는 경우가 많다.

- `noindex` 가 있는데도 sitemap 에 계속 제출된다
- canonical 이 자기 자신을 가리키며 upload 나 rejected 와 경쟁한다
- 로그인 필요 페이지인데 익명 요청에도 전체 HTML 이 보인다
- 파라미터 변형 URL 이 별도 페이지처럼 크롤링된다
- 캐시나 prerender 때문에 비공개 페이지가 노출된다

랭킹시키지 않기로 했다면 한 군데만 고쳐서는 안 된다.

## 내가 가장 많이 보는 SEO 실수 4가지

### 1. 폼과 버튼이 있으면 검색 가치도 있다고 생각하는 것

### 2. 재제출, 업로드, 반려, 도움말 페이지를 섞어서 거의 비슷한 URL을 대량으로 만드는 것

### 3. 파라미터가 많은 resubmit URL 을 sitemap 이나 내부 링크 구조에 넣는 것

### 4. 공개 정보성 검색을 비공개 워크플로우 페이지로 받으려는 것

### 3단계: 공개 설명 페이지와 비공개 워크플로우 페이지를 완전히 분리한다

### 4단계: 기술 신호를 한꺼번에 점검한다

### 5단계: 재제출 페이지가 색인에서 빠졌는지만 보지 않는다

다음도 함께 봐야 한다.

- 공개 가이드가 실제로 순위를 잡기 시작했는지

- 크롤 예산이 상품, 블로그, 도움말 페이지로 돌아오는지
- 사용자가 실제 답을 더 빨리 찾는지

## 마지막으로

**중요한 질문은 이 페이지가 다시 제출을 가능하게 하느냐가 아니다. 중요한 것은 이 URL 이 특정 개인 케이스를 위한 것인지, 아니면 공개적이고 안정적이며 재사용 가능한 문제를 해결하는 페이지인지다.**

특정 케이스를 플로우 안에서 계속 진행시키는 페이지라면 보통 워크플로우로 관리해야지 SEO 랜딩으로 밀어서는 안 된다. 반대로 “반려 후 어떻게 다시 제출하나”, “재제출 전에 무엇을 확인하나” 같은 검색을 잡고 싶다면 공개 가이드, FAQ, 체크리스트를 따로 만들어야 한다.
