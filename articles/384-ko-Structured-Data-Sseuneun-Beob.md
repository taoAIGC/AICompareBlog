## schema를 넣었는데도 검색 결과가 별로 달라지지 않는 이유는 뭘까?

이런 경우 정말 많습니다.

구조화 데이터가 중요하다고 들으면 보통 이렇게 진행합니다.

- 코드 조각을 하나 가져온다
- 사이트에 붙여 넣는다
- Product, FAQ, Review, Article을 한꺼번에 넣는다
- 리치 결과를 기다린다

그런데 막상 시간이 지나도 눈에 띄는 변화가 없습니다.

문제는 대개 “schema를 넣었느냐”가 아니라 **페이지 유형과 맞지 않는 마크업을 너무 급하게, 너무 넓게 넣었다는 점**에 있습니다.

흔한 실수는 이렇습니다.

- 일반 블로그 글에 Product를 붙인다
- FAQ가 코드에는 있지만 화면에는 없다
- 한 페이지에 메인 타입을 너무 많이 섞는다
- 리뷰가 없는 페이지에도 Review를 넣는다
- 배포 후 검증을 안 한다

그래서 구조화 데이터를 할 때 핵심은 이겁니다.

**넣을 수 있는 걸 다 넣지 말고, 페이지 내용과 실제로 맞는 schema부터 시작하는 것.**

---

## 구조화 데이터가 실제로 도와주는 것

구조화 데이터는 넣는다고 바로 순위가 오르는 마법 버튼이 아닙니다.

저는 오히려 검색엔진에게 주는 “페이지 설명서”에 가깝다고 봅니다.

주요 역할은 세 가지입니다.

### 1. 이 페이지가 무엇인지 더 분명하게 알려준다
상품 페이지인지, 글인지, FAQ인지, 회사 소개인지 검색엔진이 이해하기 쉬워집니다.

### 2. 검색 결과에서 더 풍부한 표시 기회를 만든다
FAQ, 브레드크럼, 기사 정보, 상품 정보 같은 것들은 올바른 마크업이 있어야 후보가 되기 쉽습니다.

### 3. 사이트의 기술 신호를 더 정돈해 준다
규모가 있는 사이트일수록 작성자, 브랜드, 페이지 유형, 구조를 맞춰 두는 이점이 큽니다.

---

## 이제 시작한다면 먼저 이 5가지 페이지를 보자

### 1. 상품 페이지
쇼핑몰, SaaS, 요금제, 구독 상품 페이지라면 Product schema부터 보는 게 좋습니다.

마크업은 실제로 화면에 보이는 정보만 넣는 게 원칙입니다.

- 상품명
- 브랜드
- 가격
- 재고 상태
- 평점
- 리뷰 수

보이지 않는 정보를 코드에만 넣는 건 피하는 편이 안전합니다.

---

### 2. 기사 / 블로그 페이지
블로그 글, 비교 글, 가이드, 튜토리얼에는 Article schema가 꽤 유용합니다.

정리하기 좋은 항목은 다음과 같습니다.

- 제목
- 작성자
- 발행일
- 수정일
- 대표 이미지
- 사이트명 / 발행자

콘텐츠가 많다면 이미 트래픽이 나오는 템플릿부터 시작하는 게 현실적입니다.

---

### 3. FAQ 페이지
FAQPage는 유용하지만 남용되기 쉬운 schema이기도 합니다.

저라면 이렇게 확인합니다.

- 질문이 화면에 분명히 보이는가
- 답변도 사용자가 볼 수 있는가
- 페이지의 주 구조가 실제로 FAQ인가

단지 검색 결과에서 더 눈에 띄고 싶어서 억지로 FAQ 형태로 바꾸는 건 오히려 어색합니다.

---

### 4. 브레드크럼 경로가 안정적인 페이지
사이트 구조가 안정적이라면 BreadcrumbList는 비용 대비 효과가 좋은 편입니다.

특히 이런 페이지에 잘 맞습니다.

- 기사 페이지
- 카테고리 페이지
- 문서 페이지
- 깊은 구조의 콘텐츠 페이지

다만 경로 자체가 안정적이어야 합니다. 소속 카테고리가 자주 바뀌면 혼란을 schema로 공식화하는 셈이 됩니다.

---

### 5. 브랜드 / 회사 페이지
공식 사이트, 회사 사이트, 브랜드 페이지라면 Organization schema도 우선순위가 높습니다.

명확히 정리할 수 있는 정보는 이런 것들입니다.

- 브랜드명
- 공식 사이트
- 로고
- 연락처
- 소셜 프로필
- 조직 정보

바로 트래픽이 늘지 않을 수는 있지만, 사이트 신뢰 신호를 정리하는 데는 분명 도움이 됩니다.

---

## 내가 피하고 싶은 schema 실수 4가지

### 1. 한 페이지에 메인 타입을 너무 많이 넣는 것
여러 엔티티가 있을 수는 있어도 주인공은 분명해야 합니다.

### 2. 사용자에게 보이지 않는 내용을 마크업하는 것
코드에만 있고 화면에 없으면 문제가 됩니다.

### 3. 템플릿을 복붙하고 필드를 확인하지 않는 것
오래된 URL, 잘못된 로고, 다른 브랜드명이 그대로 남는 경우가 의외로 많습니다.

### 4. 배포만 하고 테스트하지 않는 것
Rich Results Test나 Schema Markup Validator로 먼저 검증한 뒤 넓혀야 합니다.

---

## 실제 적용 순서는 이렇게 가는 편이 낫다

1. 먼저 사이트의 페이지 템플릿을 정리한다
2. 가장 안정적인 템플릿부터 시작한다
3. 한 번에 하나의 schema 타입만 넣는다
4. 소수 페이지에서 검증한 뒤 전체로 확장한다
5. 템플릿을 바꾸면 다시 점검한다

## 마무리

구조화 데이터는 처음부터 큰 프로젝트로 만들 필요가 없습니다.

눈에 바로 띄지는 않아도 사이트 바닥을 정리해 주는 기술 SEO의 기본 작업에 가깝습니다.

그러니 오늘 밤 사이트 전체에 schema를 넣으려고 하지 말고, 먼저 이 5가지 페이지부터 시작해서 보이는 정보, 페이지의 주 유형, 마크업이 서로 잘 맞는지부터 맞춰 두는 게 좋습니다.
