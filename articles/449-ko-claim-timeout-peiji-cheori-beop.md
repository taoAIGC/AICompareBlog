## 많은 해외 ​​전자상거래, 독립 웹사이트, DTC 브랜드 웹사이트에서 청구 해결, 부품 교체, 이의 제기 및 수동 검토 과정을 거치면 결국 도움말 페이지가 아닌 일련의 시간 초과 페이지, 시간 초과 페이지 및 세션 만료 URL이 나올 가능성이 가장 높은 이유는 무엇입니까?

저는 지난 2년 동안 이와 같은 문제를 점점 더 많이 보았습니다.

많은 팀이 제품 페이지, 카테고리 페이지, 블로그 페이지, FAQ 페이지를 매우 신중하게 관리해 왔습니다. 결과적으로 청구 제출 시간 초과, 교체 부품 시간 초과, 지불 시간 초과, 인터페이스 시간 초과 및 세션 만료가 발생하면 시스템은 다음과 같은 "시간 초과 관련 URL" 배치를 증가시킵니다.

- `/claim/timeout`
- `/claim/session-expired`
- `/claim/request-timeout`
- `/appeal/submit-timeout`
- `/after-sales/claim-expired`
- `/merchant/claim/timeout-record`

많은 팀이 처음으로 이러한 유형의 페이지를 볼 때 일반적으로 마음 속에 몇 가지 자연스러운 판단이 떠오릅니다.

- 사용자는 원래 "청구 합의가 만료되면 어떻게 해야 합니까?"를 검색합니다.
- 페이지에는 실패 이유, 버튼, 카운트다운이 포함되어 있어 빈 페이지처럼 보이지 않습니다.
- 시스템이 생성되었습니다. 지금은 그대로 두십시오. 어쩌면 우리는 약간의 트래픽을 받을 수도 있습니다.
- 시간 초과 페이지는 실패 페이지보다 더 "구체적"으로 들리며 포함하기에 더 적합해 보입니다.

표면적으로 이러한 아이디어는 그리 터무니없어 보이지 않습니다.

그러나 웹 사이트가 오랫동안 실행되면 실제로 SEO를 엉망으로 만드는 것은 신중하게 작성한 콘텐츠 페이지가 아니라 아무도 혼자서 정리하지 않은 프로세스 페이지라는 것을 알게 될 것입니다. 이러한 페이지는 특정 사례, 특정 계정, 특정 제출 노드 및 특정 시간 초과 이유에 크게 의존합니다. 검색 사용자가 클릭하면 먼저 "계속하려면 다시 로그인하십시오", "현재 소유권 주장 시간이 초과되었습니다. 다시 시작하십시오", "업로드된 자료가 만료되었습니다" 또는 "이 세션이 만료되었습니다"라는 메시지가 표시되는 경우가 많습니다.

결국 검색 엔진이 포착한 것은 비공개 프로세스 페이지였습니다. 정말 "청구권 합의가 만료되면 어떻게 해야 하나요?"라는 설명 페이지가 포함되어야 했습니다. "시간 초과 후에도 정보를 다시 제출할 수 있나요?"

따라서 최근에 **시간 초과 페이지를 처리하는 방법**을 읽고 계시다면 먼저 결론을 내리겠습니다.

**대부분의 시간 초과 페이지는 본질적으로 공개 검색 랜딩 페이지가 아닌 프로세스 예외 페이지, 상태 프롬프트 페이지 및 재작업 입구입니다. 사용자가 자료 제출, 고객 서비스 전환, 운영 문제 해결 등을 수행하는 것은 확실히 가치가 있지만, 대부분의 경우 핵심 SEO 페이지로 홍보하기에는 적합하지 않습니다. 중요한 것은 페이지에 카운트다운과 버튼이 있는지 여부가 아니라 먼저 이 URL이 특정 사례를 제공하는지, 아니면 공개적이고 안정적이며 재사용 가능한 검색 질문에 응답하는지 여부를 구별하는 것입니다. **

## 타임아웃 페이지는 어떤 문제를 해결하나요? 처음 검색하는 사람이 아닌, 그 과정에 막혀 있는 사람에게 서비스를 제공하는 것이 우선입니다.

### 1. 이미 진행 중인 사람들에게 우선적으로 봉사합니다.

시간 초과 페이지의 가장 일반적인 기능은 일반적으로 다음과 같습니다.

- 현재 단계가 시간 초과된 이유를 사용자에게 알려줍니다.
- 로그인이 만료되었는지, 제출 시간이 초과되었는지, 인터페이스 시간이 초과되었는지, 패치 시간이 초과되었는지 사용자에게 알립니다.
- 다음에 다시 제출할지, 다시 로그인할지, 고객 서비스에 문의할지 사용자에게 알려줍니다.
- 타임아웃이 발생한 노드와 시스템 문제 여부를 고객 서비스 및 운영팀에 알려주세요.

즉, 구글에서 클릭하는 낯선 검색 사용자보다는 이미 컨텍스트를 알고 있는 사람들에게 주로 서비스를 제공합니다.

### 2. 일반적으로 사건번호, 계좌 신원, 공소시효 규정과 깊이 연관되어 있습니다.

시간 초과 페이지에는 다음 정보가 포함되는 경우가 많습니다.

- 주문번호
- 청구번호
- 현재 시간 초과 이유
- 타임아웃이 발생한 시간
- 보충서류 마감
- 현재 노드 상태
- 재제출 허용 여부
- 현재 계정 로그인 상태

이는 시간 초과 페이지가 일반적으로 안정적이지 않고 표준화되지 않았으며 장기 검색 트래픽에 적합하지 않다는 것을 결정합니다.

### 3. 비즈니스 가치는 있지만 SEO 가치가 있다는 의미는 아닙니다.

많은 팀이 다음 사항을 혼동할 가능성이 높습니다.

- 페이지는 비즈니스 프로세스에 중요하므로 SEO에도 중요해야 합니다.
- 페이지에는 이유와 액션 버튼이 있으므로 포함할 가치가 있습니다.
- 사용자는 흐름 중에 확실히 이를 볼 수 있으므로 검색 엔진도 이를 볼 수 있어야 합니다.

그러나 검색 사용자가 실제로 검색하는 내용은 다음과 같습니다.

- 청구 정산 시간 초과 후 해야 할 일
- 보충서류의 시간이 초과되어 다시 제출될 수 있나요?
- 항상 세션이 만료되었다는 메시지가 표시되는 이유는 무엇입니까?
- 시간 초과, 실패 및 종료의 차이점은 무엇입니까?
- 타임아웃 후 처리를 재개하는 데 얼마나 걸리나요?

이러한 검색 요구 사항을 처리하는 데 더 적합한 것은 일반적으로 비공개 시간 초과 페이지가 아니지만 다음과 같습니다.

- 클레임 시간 초과 처리 설명 페이지
- 교체 부품 노화 규칙에 대한 도움말 페이지
- 세션 만료 일반적인 이유 설명 페이지
- 시간 초과 후 FAQ 페이지를 다시 제출하세요.
- 시간 초과, 종료 및 오류 차이 페이지

## 시간 초과 페이지를 처리하는 방법은 무엇입니까? 나는 먼저 이 5가지 상황을 개별적으로 처리하는 것을 선호합니다.

### 1. 표준 시간 초과 페이지, 세션 만료 페이지 또는 제출 시간 초과 페이지인 경우 일반적으로 이를 핵심 SEO 페이지로 취급할 필요가 없습니다.

가장 일반적인 카테고리입니다.

이러한 페이지에는 대부분 다음과 같은 몇 가지 공통점이 있습니다.

- 로그인을 하셔야 보실 수 있습니다.
- 사건번호 또는 사용자 신원을 묶어야 함
- 페이지의 콘텐츠가 특정 청구 기록에 중점을 두고 있습니다.
- 재로그인, 재제출, 시스템 복귀를 통해 상태는 계속해서 변경됩니다.
- 외부 사용자에 대한 안정적인 판독값은 거의 없음

한마디로: **표준 시간 초과 페이지는 공개 콘텐츠 페이지가 아니라 프로세스 프롬프트 페이지와 비슷합니다. **

### 2. 실제 검색 값이 "타임아웃 후 수행할 작업" 및 "타임아웃이 발생하는 이유"인 경우 비공개 타임아웃 페이지에서 이러한 질문에 대한 트래픽을 처리하지 않도록 하세요.

많은 팀이 여기서 실수를 저지르는 경향이 있습니다.

사용자는 "timeout", "timeout" 및 "sessionexpirated"를 검색하기 때문에 결론을 내리는 것이 당연합니다. 누군가가 검색하고 있으므로 시간 초과 페이지가 포함되어야 한다는 것입니다.

그러나 사용자가 실제로 알고 싶어하는 것은 일반적으로 다음 사항이 아닙니다.

- 특정 사건을 다시 제출할 수 있나요?
- 느린 인터페이스 또는 불완전한 데이터 전송으로 인해 특정 시간 초과가 발생했습니다.
- 특정 계정에 로그인하자마자 세션이 만료되었다는 메시지가 표시되는 이유는 무엇입니까?

그들이 더 알고 싶은 것은 다음과 같습니다.

- 청구 정산 시간 초과의 이유는 무엇입니까?
- 타임아웃 이후에도 해결이 가능한가요?
- 시간이 초과되면 데이터를 다시 업로드해야 합니까?
- 타임아웃, 실패, 종료는 같은 것입니까?
- 어떤 상황에서 수동 고객 서비스가 개입해야 합니까?

이러한 요구 사항은 특정 사례에 대한 시간 초과 페이지보다는 공개 설명 페이지, FAQ 페이지, 규칙 페이지 및 도움말 페이지에 더 적합합니다.

### 3. 시스템이 시간 초과, 요청 시간 초과, 세션 만료, 만료, 나중에 재시도 등의 변형 URL을 동시에 생성하는 경우 이를 함께 제어해야 합니다.

많은 웹사이트의 실제 문제는 단순한 '/claim/timeout'이 아닙니다.

보다 일반적으로 시스템은 다음과 같이 성장합니다.

- `/claim/timeout`
- `/claim/request-timeout?id=xxx`
- `/claim/session-expired`
- `/appeal/submit-timeout`
- `/after-sales/claim-expired`
- `/claim/timeout/retry`
- `/claim/timeout/failed`
- `/merchant/claim/timeout-record`

이러한 URL이 많으면 쉽게 표시됩니다.

- 내용의 본문은 거의 동일합니다.
- 매개변수, 노드, 시간만 다름
- 타이틀 템플릿은 매우 반복적입니다.
- 재시도, 실패, 기록 등의 페이지도 캡처됩니다.
- 사용자 센터와 판매자 백엔드는 여전히 이 페이지에 접속하고 있습니다.

따라서 시간 초과 페이지를 처리할 때 매우 실용적인 조치는 다음과 같습니다.

**메인 타임아웃 페이지만 보지 마시고, 세션 만료 페이지, 재제출 페이지, 실패 페이지, 기록 페이지를 함께 살펴보세요. **

### 4. '공개 설명 페이지'와 '비공개 시간 초과 페이지'가 모두 있는 경우 두 페이지 유형을 완전히 분리하세요.

많은 웹사이트가 정말 지저분합니다. 페이지 수가 너무 적어서가 아니라 페이지의 역할이 명확하게 정의되지 않았기 때문입니다.

한 세트는 다음과 같은 공개 페이지입니다.

- 청구 정산 시간 초과 후 해야 할 일
- 타임아웃 후에도 패치를 계속할 수 있나요?
- 세션이 만료된 이유는 무엇입니까?
- 타임아웃, 실패, 종료의 차이

다른 세트는 다음과 같은 개인 페이지입니다.

- 특정 주문에 대한 시간 초과 프롬프트 페이지
- 특정 가맹점 계정의 타임아웃 기록 페이지
- 특정 항소 제출 시간이 초과된 후 다시 운영되는 페이지

이 두 가지 유형의 페이지가 혼합된 경우 가장 일반적인 결과는 다음과 같습니다.

- 누구도 대중의 요구를 받아들이지 않는다.
- 비공개 페이지가 검색 엔진에 의해 크롤링되었습니다.
- 사용자는 검색으로 들어와 완전한 정보를 이해하지 못하고 볼 수 없습니다.
- 정말 순위에 해당하는 설명 페이지와 FAQ 페이지는 열 수 없습니다.

### 5. 타임아웃 페이지가 순위에 참여하는 것을 허용하지 않으려면 NOINDEX, 로그인 차단, 사이트맵, 표준 및 내부 링크를 함께 통합하세요.

많은 시간 초과 페이지 SEO 문제는 "이 페이지가 있습니다" 때문이 아니라 신호가 모두 싸우기 때문에 발생합니다.

일반적인 상황은 다음과 같습니다.

- 페이지 템플릿에 NOINDEX가 기록되어 있지만 사이트맵에서는 계속해서 시간 초과 URL을 제출합니다.
- 메인 페이지는 로그인이 필요하지만, 재시도, 실패, 기록 등의 페이지도 공개적으로 접근 가능합니다.
- 혼돈에 대한 표준 지점, 시간 초과 페이지, 실패 페이지, 종료 페이지가 서로 얽혀 있습니다.
- 사용자 센터, 사이트 검색, 도움말 센터에서는 아직 크롤링 가능한 페이지로 진입하고 있습니다.

## 내가 지금 보고 있는 가장 일반적인 4가지 시간 초과 페이지 SEO 실수는 사이트를 망치는 경우가 발생하지 않는 경우보다 더 많습니다.

### 1. "타임아웃"은 사용자의 검색 요구를 나타내는 것으로 간주되므로 당연히 타임아웃 페이지를 포함할 가치가 있습니다.

### 2. 시간 초과 페이지, 실패 페이지, 페이지 닫기, 재시도 페이지를 뒤섞으면 결국 비슷한 의미를 지닌 페이지가 사이트에 늘어납니다.

이러한 페이지의 비즈니스 의미는 본질적으로 다릅니다. 제목, 템플릿, 내부 링크 등이 뒤섞여 있으면 결국에는 더 완전하게 다루지 못하고 서로 단어 경쟁을 하게 되는 경우가 많습니다.

### 3. 세션 만료, 나중에 재시도, 기록, 기록 등의 페이지가 시스템에서 쉽게 해제되어 씬 페이지가 차례로 발생합니다.

이러한 페이지는 내부 프로세스에 유용할 수 있지만 공개 검색에는 대부분 독립적인 가치가 없습니다. 일괄 출시되면 크롤링 예산을 분산하고 중복 콘텐츠를 생성합니다.

### 4. "청구 합의 만료 후 해야 할 일"에 대한 공개 콘텐츠를 분명히 만들어야 하지만, 항상 비공개 프로세스 페이지에서 검색 요구를 처리하기를 원합니다.

이것은 실제로 많은 웹사이트의 근본적인 문제입니다. 사용자 문제가 없다는 것이 아니라 이러한 문제가 공개 콘텐츠로 정리된 적이 없다는 것입니다.
