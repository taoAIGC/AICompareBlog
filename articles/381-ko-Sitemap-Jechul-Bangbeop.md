## 왜 많은 사람들은 sitemap에 가능한 모든 URL을 넣으려고 할까?

정말 자주 보는 실수다.

사이트를 만들고 SEO를 본격적으로 시작하면 보통 이렇게 움직인다.

- sitemap 생성
- 있는 URL 전부 넣기
- Search Console에 제출
- 색인과 순위 기다리기

겉으로 보기엔 열심히 하는 것 같지만, 실제로는 다르다.

**좋은 sitemap은 큰 sitemap이 아니라 깔끔한 sitemap이다.**

테스트 페이지, 파라미터 URL, 중복 페이지, 얇은 페이지, noindex 페이지까지 다 넣으면 검색엔진은 “정리된 사이트”가 아니라 “아직 스스로도 정리가 안 된 사이트”로 받아들인다.

핵심은 이것이다.

**더 많은 URL을 제출하는 것이 아니라, 맞는 URL을 제출해야 한다.**

---

## sitemap은 원래 무슨 역할일까? 강제 색인 버튼이 아니다

sitemap은 기본적으로 URL 목록이다.

검색엔진에게 이런 신호를 주는 것이다.

> 이 URL들이 우리 사이트의 공식 페이지이니 우선적으로 봐 달라.

즉, “빨리 확인해 달라”는 뜻이지 “무조건 지금 색인해 달라”는 뜻은 아니다.

sitemap은 보통 이런 데 도움이 된다.

1. 페이지를 더 빨리 발견하게 함
2. 어떤 URL이 공식 버전인지 알려줌
3. 사이트 구조를 스스로 정리하게 만듦

하지만 콘텐츠 품질, 내부 링크, 검색 의도까지 대신 해결해 주지는 못한다.

## sitemap 제출은 어떻게 해야 할까? 나는 이 5가지를 먼저 맞춘다

### 1. 정말 색인시키고 싶은 공식 URL만 넣는다

내 기준은 단순하다.

**이 페이지가 오늘 색인되어도 정말 괜찮은가?**

아니라면 sitemap에 넣지 않는다.

보통 제외하는 것은:

- 추적 파라미터가 붙은 URL
- 사이트 내부 검색 결과 페이지
- 테스트 페이지와 미리보기 페이지
- 명확하게 중복되는 태그/필터 페이지
- canonical이 다른 페이지를 가리키는 URL

### 2. 새 사이트라면 핵심 페이지부터 제출한다

신규 사이트는 sitemap이 부족해서가 아니라 약한 페이지를 너무 빨리 보여줘서 문제인 경우가 많다.

처음에는 다음 정도면 충분하다.

- 홈
- 핵심 카테고리
- 중요한 글 몇 개
- 구조가 완성된 페이지

### 3. lastmod를 억지로 갱신하지 않는다

실제로 바뀐 게 없는데 모든 URL 날짜를 오늘로 맞추는 경우가 있다. 이런 신호는 오래 가면 신뢰를 떨어뜨린다.

더 안전한 방식은:

- 본문이 실제로 바뀌었을 때만 업데이트
- 아주 작은 수정에는 건드리지 않기
- 모든 URL에 같은 시간값 주지 않기

### 4. 페이지가 많아지면 유형별로 분리한다

예를 들면:

- 글
- 상품
- 카테고리
- 언어별 페이지

이렇게 나누면 나중에 문제를 찾기가 훨씬 쉽다.

### 5. 제출 후에는 상태를 반드시 본다

많은 사람이 제출까지만 한다. 하지만 중요한 건 그다음이다.

내가 주로 보는 것은:

- 제출한 URL 수가 실제와 맞는지
- “발견됨 - 현재 색인 안 됨”이 계속 쌓이는지
- 제외되는 URL이 너무 많은지
- canonical, noindex, robots와 충돌하는지

---

## 나는 어떤 페이지를 sitemap에 잘 넣지 않을까?

보통 이런 페이지는 제외한다.

- 404, 410 페이지
- 리디렉션된 URL
- noindex 페이지
- canonical이 다른 곳을 가리키는 URL
- 파라미터 중복 페이지
- 내부 검색, 로그인, 장바구니 같은 기능 페이지
- 아직 미완성인 페이지

짧게 말하면,

**내가 메인으로 보여주고 싶지 않은 페이지를 검색엔진에 정식으로 제출하지 않는 것이다.**

---

## sitemap을 제출했는데도 색인이 느리다면, 먼저 이 4가지를 확인한다

### 1. 그 페이지가 정말 색인될 가치가 있는가

### 2. 그 페이지로 들어오는 내부 경로가 있는가

내부 링크가 없으면 페이지는 쉽게 고립된다.

### 3. 기술 신호가 서로 충돌하지 않는가

예를 들면:

- sitemap은 A를 제출
- canonical은 B를 가리킴
- robots는 일부 경로를 차단
- 페이지에는 noindex까지 있음

### 4. 크롤링이나 서버 상태에 문제가 없는가

사이트가 느리거나 오류가 많다면 sitemap을 다시 제출해도 큰 효과가 없다.

## 마지막으로

sitemap 제출 자체는 어렵지 않다.

가장 흔한 실수는 만드는 법을 모르는 게 아니라, 색인이 느릴 때 누르는 만능 버튼처럼 생각하는 것이다.

내 방식은 단순하다.

- 먼저 URL 정리
- 공식 페이지만 제출
- 상태 확인
- 문제 있으면 페이지 품질과 내부 구조부터 다시 점검

이렇게 해야 sitemap이 실제로 도움이 된다.
