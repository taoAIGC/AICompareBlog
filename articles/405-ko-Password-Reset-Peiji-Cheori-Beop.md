## 많은 웹사이트에서 멤버십 시스템, SaaS 백엔드 또는 강좌 플랫폼을 개발할 때 가장 먼저 나타날 가능성이 가장 높은 것은 콘텐츠 페이지가 아니라 일련의 비밀번호 검색 페이지, 비밀번호 재설정 페이지 및 토큰이 포함된 복구 링크인 이유는 무엇입니까?

이것은 정말 흔한 일입니다.

특히 회원 사이트, 도구 사이트, SaaS, 커뮤니티 또는 전자 상거래 백엔드를 구축할 때 시스템이 온라인 상태가 되면 사이트에 다음 주소를 쉽게 추가할 수 있습니다.

- `/forgot-password`
- `/reset-password`
-`/recover`
- `/password/reset`
- `/reset?token=abc123`
- `/reset-password?email=xxx@example.com`
- `/recover?next=/dashboard`
- "링크 만료 페이지", "재설정 성공 페이지", "계정 잠김 페이지", "신원 확인 페이지"와 같은 상태 페이지도 있습니다.

많은 사람들이 이런 종류의 페이지를 볼 때 일반적으로 첫 번째 반응은 다음과 같습니다.

- 공식 기능 페이지이므로 그대로 두셔도 문제가 없습니다.
- 사용자가 여기까지 검색을 했고, 그 의도가 매우 강할 것입니다.
- 이미 '누구나 비밀번호 재설정'을 검색하고 계시는 분들도 계시기 때문에 포함시켜 놓는 것이 더 편리할 수도 있습니다.
- 시스템은 기본적으로 생성합니다. 지금은 그대로 두고 나중에 정리하세요.

완전히 불합리한 것처럼 들리지는 않습니다.

하지만 실제로 오랫동안 해보면 이런 종류의 페이지가 종종 다음과 같이 나타나는 것을 발견하게 될 것입니다.

- 페이지의 내용이 매우 얇습니다. 핵심은 입력 상자, 버튼 및 몇 가지 프롬프트입니다.
- 여러 토큰, 이메일 및 상태 매개변수를 사용하면 대략적인 URL을 쉽게 표기할 수 있습니다.
- 사용자가 검색 결과에서 클릭합니다. 문제는 아직 해결되지 않았습니다. 먼저 그는 자신의 이메일 주소를 입력하라는 양식을 봅니다.
- 검색 요구 사항을 처리하는 데 실제로 적합한 도움말 페이지, FAQ 및 계정 지원 페이지가 강화되지 않았습니다.
- 여러 프로세스 페이지와 상태 페이지에 의해 천천히 분산되는 리소스와 사이트 신호를 캡처합니다.

그래서 최근에 **비밀번호 복구 페이지 처리 방법**을 공부하고 계시다면 먼저 매우 간단한 판단을 내리겠습니다.

**대부분의 비밀번호 복구 페이지는 본질적으로 "공개 검색 랜딩 페이지"가 ​​아닌 "계정 복구 프로세스 페이지"입니다. 기존 사용자들에게 계정을 복구하는 것은 확실히 중요하지만, 대부분의 경우 핵심 SEO 페이지로 홍보하기에는 적합하지 않습니다. 실제로 해야 할 일은 단순히 이를 방치하거나 전면적으로 차단하는 것이 아니라 먼저 이 URL이 사용자의 신원 복구를 완료하기 위한 것인지, 아니면 공개 검색 질문에 답하기 위한 것인지 구별하는 것입니다. **

## 비밀번호 검색 페이지는 어떤 문제를 해결하나요? 단지 "비밀번호 재설정 버튼을 넣는 것"이 ​​아닙니다.

비밀번호 검색 페이지에 대한 많은 사람들의 이해는 여전히 다음과 같이 제한되어 있습니다.

- 비밀번호를 잊어버린 경우 클릭
- 인증번호를 받으려면 이메일이나 휴대폰 번호를 입력하세요.
- 새로운 비밀번호를 설정하세요
- 아주 평범한 기능 페이지

이러한 이해는 틀린 것은 아니지만 여전히 약간 얕습니다.

실제 비밀번호 복구 페이지는 일반적으로 최소한 세 가지 문제를 해결합니다.

### 1. 핵심 임무는 정보 기반 검색을 수행하는 것이 아니라 계정 액세스를 복원하는 것입니다.

비밀번호 페이지를 검색하는 핵심 작업은 일반적으로 다음과 같습니다.

- 사용자 신원 확인
- 재설정 링크 또는 인증 코드 보내기
- 사용자에게 비밀번호를 재설정하도록 요청
- 사용자를 원래 로그인 프로세스로 복원합니다.

즉, '아직도 상품을 검색하고 비교하고 있는 사람'보다는 '이미 계정이 있지만 가입할 수 없는 사람'을 대상으로 하는 경우가 많습니다.

### 2. 토큰, 인증코드, 점프 매개변수, 유효하지 않은 상태와 연관되는 경우가 많습니다.

예를 들어:

- 이메일의 토큰 재설정
- SMS 인증코드
- `next`, `redirect`, `return_to`와 같은 점프 매개변수
- 링크 만료, 시간 초과, 계정 동결 등의 상태

이는 비밀번호 검색 페이지가 공용 색인 작성에 적합한 안정적인 단일 장기 페이지가 아닌 경우가 많으며 상태에 따라 계속 변경되는 프로세스 페이지 세트임을 판별합니다.

### 3. 지원 가치는 있지만 SEO 구현 가치가 있다는 의미는 아닙니다.

비밀번호 복구 페이지는 물론 중요합니다.

그렇지 않으면 기존 사용자는 전혀 다시 돌아올 수 없으며 고객 서비스에 대한 부담이 커질 수 있습니다.

그러나 중요한 사업이 자연탐색을 수행하기에 적합하다는 것을 의미하지는 않는다는 점에 유의해야 한다. **

사용자가 다음 단어를 검색하는 경우:

- 비밀번호를 잊어버린 경우 어떻게 해야 하나요?
- 재설정 메일을 받지 못한 경우 어떻게 해야 하나요?
- 재설정 링크가 실패하면 어떻게 해야 하나요?
- 계정이 잠긴 경우 어떻게 해야 합니까?
- 특정 플랫폼에 로그인할 수 없으면 어떻게 해야 하나요?

수행하기에 더 적합한 것은 일반적으로 기본 재설정 양식 페이지가 아니지만 다음과 같습니다.

- 도움말 센터 기사
- FAQ 페이지
- 계정 복구 안내 페이지
- 로그인 문제 해결 페이지
- 고객 서비스 지원 포털 페이지

비밀번호 검색 양식을 통해 이러한 모든 질문에 답할 수 있다면 입구를 교묘하게 디자인한 것이 아니라 콘텐츠 레이어와 프로세스 레이어가 혼합되어 있을 가능성이 높습니다.

## 비밀번호 검색 페이지를 어떻게 처리하나요? 먼저 이 5가지 상황을 별도로 처리하는 것이 좋습니다.

### 1. 단지 표준적인 "비밀번호 찾기" 입력 페이지라면 일반적으로 핵심 SEO 페이지로 취급할 필요가 없습니다.

가장 일반적인 카테고리입니다.

예를 들어:

- 회원시스템 비밀번호 분실 페이지
- SaaS 콘솔 비밀번호 재설정 포털
- 전자상거래 백그라운드 비밀번호 재설정 페이지
- 강좌 플랫폼 계정 복구 페이지
- 로그인 자격 증명 페이지를 검색하는 커뮤니티 사이트

이러한 유형의 페이지는 기존 사용자에게 실제로 중요하지만 문제도 분명합니다.

**대부분의 공개 검색 요구 사항을 충족하기 위해 존재하지 않습니다. **

사용자가 검색 엔진에서 클릭한 후 일반적인 반응은 다음과 같습니다.

- 아직 문제를 파악하지 못했습니다. 이메일 주소를 직접 입력하려면 어떻게 해야 하나요?
- 이메일을 받을 수 없으면 어떻게 해야 하는지 알고 싶습니다.
- 제가 만난 것은 단순히 비밀번호를 잊어버린 것이 아니라 링크가 만료되었다는 것이었습니다.
- 문제 해결 지침은 없고 양식이 하나만 있는 이유는 무엇입니까?

따라서 표준 비밀번호 복구 페이지는 일반적으로 초점 SEO 페이지보다는 프로세스 페이지로 더 적합합니다.

한마디로: **비밀번호 검색 페이지는 매우 중요할 수 있지만 이 중요성은 자연 검색의 의미가 아닌 계정 복구 프로세스의 의미에서 가장 중요합니다. **

### 2. 실제 검색 값이 "로그인 문제 지침" 및 "계정 복구 도움말"인 경우 재설정 양식이 해당 트래픽을 인계받지 않도록 하세요.

이런 오해도 매우 흔합니다.

일부 팀은 무의식적으로 다음과 같은 느낌을 받을 것입니다.

- 사용자가 '비밀번호 찾기'를 검색하여 재설정 페이지로 바로 이동하는 것이 더 빠르지 않을까요?
- 사용자는 이미 강력한 요구 사항을 갖고 있으므로 양식 페이지에서도 이를 수용할 수 있습니다.
- 어차피 비밀번호는 결국 재설정해야 하므로 먼저 이 페이지로 가셔도 괜찮습니다.

그러나 SEO 관점에서 보면 이는 불안정한 경우가 많습니다.

사용자는 종종 다음 이외의 항목을 검색하기 때문입니다.

- 재설정 버튼은 어디에 있나요?
- 이메일을 입력하는 상자는 어떤 모양인가요?

대신에:

- 왜 재설정 이메일을 받을 수 없나요?
- 링크가 실패하면 어떻게 해야 하나요?
- 계정이 잠긴 경우 복구하는 방법은 무엇입니까?
- 제3자가 로그인한 계정을 검색하는 방법
- 비밀번호나 등록된 이메일 주소를 잊어버린 경우 어떻게 해야 하나요?

이러한 질문은 다음이 더 적절하게 질문합니다.

- 도움말 센터 기사
- 로그인 예외 FAQ
- 계정 복구 안내 페이지
- 고객 서비스 포털에 문의하세요.
- 브랜드 지원 페이지

인수합니다.

**명확하게 설명해야 할 문제를 모두 숨기지 말고 재설정 양식을 검색 결과 앞쪽으로 밀어 넣으세요. **

### 3. 비밀번호 검색 시스템이 토큰, 이메일, 상태, 다음과 같은 매개변수 URL을 생성하는 경우 먼저 "기능 URL"과 "인덱스 URL"을 구별해야 합니다.

많은 웹사이트의 비밀번호 복구 페이지의 실제 문제는 `/forgot-password`가 하나만 있다는 것이 아니라 다양한 변형도 생성된다는 것입니다.

- `/reset?token=abc123`
- `/reset-password?email=user@example.com`
- `/recover?next=/checkout`
- `/password/reset?status=expired`
- `/reset-password?from=app`
- `/forgot-password?redirect=/dashboard`

물론 이 주소는 프로그램 흐름에 유용합니다.

그러나 SEO의 경우 다음을 의미하는 경우가 많습니다.

- 거의 중복된 URL이 다수 있음
- 매개변수는 페이지 신호를 분해합니다.
- 가져온 리소스는 상태 페이지에서 소모됩니다.
- 외부 링크가 실수로 잘못된 버전을 가리킬 수 있습니다.
- 민감한 상태의 일부 주소도 공개적으로 노출됩니다.

따라서 비밀번호 검색 페이지를 처리할 때 특히 실용적인 조치는 다음과 같습니다.

- 먼저 계정 복구 과정에 필요한 URL이 무엇인지 구별하세요.
- 검색엔진이 실제로 봐야할 표준입구가 무엇인지 구별
-토큰 유형, 상태 유형, 점프 유형 매개변수 페이지는 기본적으로 모든 위치에 출력됩니다.
- 템플릿, 이메일, 버튼, 도움말 문서의 링크를 통합하는 것이 가장 좋습니다.

**비밀번호 검색 페이지에서 제어할 수 없을 가능성이 가장 높은 것은 텍스트가 아니라 매개변수 및 상태인 경우가 많습니다. **

### 4. 비밀번호 검색 프로세스가 본인 확인 페이지, 새 비밀번호 설정 페이지, 링크 실패 페이지, 성공 페이지에도 연결되어 있는 경우 복구 링크 전체가 공개 씬 페이지가 되지 않도록 하세요.

많은 팀은 `/forgot-password` 항목에만 집중합니다.

그러나 실제로 나오는 것은 종종 다음과 같은 전체 문자열입니다.

- 비밀번호 입력 페이지를 잊어버렸습니다.
- 이메일 또는 휴대폰 번호 확인 페이지
- 인증코드 페이지 입력
- 새 비밀번호 설정 페이지
- 만료된 페이지 링크
- 비밀번호 변경 성공 페이지
- 계정 이상 또는 페이지 잠김
- 로그인 후 해당 페이지로 이동

이러한 페이지가 존재할 수 없는 것은 아니지만, 기본적으로 모두 검색에 포함되어야 한다는 것입니다.

특히 다음과 같은 상황에서는 롤오버가 쉽습니다.

- 고객센터에 계정 복구 안내가 없습니다.
- FAQ에서는 일반적인 오류 시나리오를 다루지 않습니다.
- 고객센터 입구가 명확하지 않습니다.
- 결과적으로 검색 엔진에는 처음에는 여러 상태 페이지와 양식 페이지가 포함되었습니다.

이는 사이트 무결성이 아니라 구조적 정렬 오류입니다.

### 5. 비밀번호 복구 페이지가 순위에 참여하는 것을 허용하지 않으려면 NOINDEX, 사이트맵, 내부 링크, 표준 및 매개변수 규칙을 함께 통합하세요.

많은 비밀번호 복구 페이지 SEO 문제는 "이런 페이지가 있는지"가 아니라 신호가 서로 싸우기 때문에 발생합니다.

일반적인 상황은 다음과 같습니다.

- 팀에서는 비밀번호 복구 페이지는 중요하지 않다고 합니다.
- 사이트맵 재설정 관련 URL도 제출했습니다.
- 바닥글, 팝업, 템플릿, 도움말 문서를 일반 페이지처럼 링크할 수 있습니다.
- 이메일 내 파라미터 주소는 2회 전파도 가능합니다.
- 정식 규칙이 통일되지 않았습니다.
- 일부 성공 페이지와 실패 페이지도 공용 출입구로 반복적으로 공유됩니다.

결국에는 다음과 같이 됩니다.

- 검색 트래픽을 수신하도록 허용할 의도가 없습니다.
- 검색 엔진에는 "이 페이지가 중요할 수 있습니다"라는 힌트가 계속 표시됩니다.
- 실제로 추천해야 할 도움말 페이지, 지원 페이지, 제품 페이지가 충분히 집중되어 있지 않습니다.

비밀번호 복구 페이지의 SEO 항목에 집중하지 않기로 결정한 경우 일반적으로 보다 안정적인 아이디어는 다음과 같습니다.

- 명확하고 표준적인 계정 복구 입구만 유지하세요.
- 사이트맵은 실제로 순위에 참여하고 싶은 도움말 페이지 및 콘텐츠 페이지 제출에 중점을 둡니다.
- 파라메트릭 재설정 URL은 최대한 적게 노출되어야 합니다.
- 내부 링크를 위해 상태 페이지를 콘텐츠 페이지로 포장하지 마세요.
- NOINDEX, Canonical, Jump 및 템플릿 출력이 일관됩니다.

기술적인 신호로 싸우지 마세요. 이는 "비밀번호 복구 페이지 포함 여부" 자체보다 더 중요합니다.

## 비밀번호 페이지를 검색할 때 가장 흔하게 발생하는 4가지 SEO 실수는 이를 수행하지 않는 것보다 웹사이트를 엉망으로 만들 가능성이 더 높습니다.

### 1. 사용자가 "비밀번호 재설정"을 검색할 것으로 예상되므로 양식 페이지는 자연스럽게 순위에 적합합니다.

사용자는 검색할 수 있지만 이는 양식 페이지가 최선의 답변이라는 의미는 아닙니다.

많은 경우 사용자가 실제로 해결하고 싶은 것은 입력 상자를 즉시 ​​보는 것보다 "왜 실패했는지"와 "다음에 무엇을 해야 하는지"입니다.

### 2. 문제 해결 지침, 예외 처리 및 계정 복구 도움말을 프로세스에 포함하지만 공개 도움말 페이지는 없습니다.

이는 지원 콘텐츠와 프로세스 항목을 혼합하는 일반적인 현상입니다.

프로세스 페이지는 사용자가 조치를 취할 수 있도록 하는 역할을 담당합니다.
나는 당신을 위해 모든 문제를 완전히 설명할 책임이 없습니다.

### 3. 토큰 페이지, 유효하지 않은 페이지, 성공 페이지, 잠긴 페이지가 모두 크롤링 가능한 URL로 노출됩니다.

시스템이 정상적으로 실행되는 것 같지만 실제로 사이트에는 몇 가지가 더 있습니다.

- 상태 얇은 페이지
- 매개변수 변형 페이지
- 민감한 프로세스 페이지
- 독립적인 검색값이 없는 복구 페이지

이러한 페이지가 너무 많으면 사이트 내의 신호가 쉽게 희미해집니다.

### 4. 비밀번호 검색 페이지 SEO를 하고 싶지 않다고 했으나 결국 사이트맵, 템플릿, 내부 링크, 캐노니컬이 모두 제 역할을 했습니다.

당신이 가장 두려워하는 것은 잘못된 방법을 선택한 것이 아니라, 그것이 중요하지 않다고 말하면서도 기술적으로는 "중요하다"고 계속 암시하는 것입니다.

이러한 종류의 신호 싸움은 단순히 비밀번호 복구 페이지를 유지하는 것보다 더 해로운 경우가 많습니다.

## 비밀번호 검색 페이지를 포함해야 하나요? '통합 답변'을 먼저 묻지 말고 공개 검색 요청을 수락하는지 먼저 물어보세요.

많은 사람들이 결국 같은 질문으로 돌아올 것입니다.

**비밀번호 복구 페이지가 포함되어야 합니까? **

나의 판단 방법은 항상 매우 간단했습니다. 남들한테 어떻게 하는지 물어보기보다는 먼저 이 페이지는 뭐하는 페이지인지 물어봅니다.

주로 다음 작업을 수행하는 경우:

- 이메일 또는 휴대폰 번호 수신
- 인증 코드 보내기 또는 링크 재설정
- 토큰 또는 신원 상태 확인
- 사용자가 새 비밀번호를 설정하도록 허용
- 사용자를 로그인 프로세스로 다시 보냅니다.

그러면 여전히 기능 페이지나 프로세스 페이지일 가능성이 높습니다.

이러한 유형의 페이지는 비즈니스에 매우 중요할 수 있지만 대부분의 경우 주요 자연 검색 항목이 될 필요는 없습니다.

공개 계정 복구 헬프 센터나 브랜드 지원 포털을 구축 중이고 페이지 자체가 수많은 사용자 질문에 독립적으로 답변할 수 있는 경우와 같이 드문 경우에만 기존 비밀번호 복구 페이지 이상의 기능을 수행할 수 있습니다.

하지만 표준 복구 프로세스 페이지일 뿐이라면 SEO KPI를 외우도록 강요하지 마세요.
