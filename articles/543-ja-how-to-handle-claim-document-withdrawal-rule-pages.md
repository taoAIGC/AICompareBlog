## 設定ページ、オプションページ、プリファレンスページまで整理したのに、なぜ rule 系のURLだけ残ってしまうのか？

理由は単純で、rule という言葉が誤解を生みやすいからです。

バックエンド設定ページほど内部向けには見えず、結果ページほど使い捨てにも見えません。だから多くのチームが、もしかすると検索流入を取れるのでは、と迷います。

よくあるURLはこんな形です。

- /claim/document-withdraw-rule
- /claim/document-withdraw-rules
- /claim/document-withdraw/match-rule
- /claim/document-withdraw/channel-rule
- /claim/document-withdraw/approval-rule
- /claim/document-withdraw/rule-detail

でも実際には、こうしたページの大半は公開検索ニーズに答えるものではありません。内部ロジック、つまりどの条件で撤回できるか、どのロールが優先されるか、どのチャネルがどれを上書きするかを管理するページです。

検索ユーザーが本当に知りたいのは、むしろ次のようなことです。

- ルールがなぜ反映されないのか
- 優先順位はどう決まるのか
- なぜアカウントごとに別のルールが当たるのか
- ルールを変えたのにフロント側が変わらないのはなぜか
- rule、setting、preference の違いは何か

## ルールページは実際に何をしているのか

### 1. ワークフローの動きを定義する

多くのルールページは、次のようなことを扱います。

- 撤回を許可する条件、止める条件
- ロール、チャネル、サイトごとの判定
- 優先順位と上書きロジック
- 通知、承認、例外時の処理

つまり、まず内部運用ページであって、公開SEOページではありません。

### 2. 文脈依存がかなり強い

典型的には次の情報が入っています。

- tenant、site、channel、role
- マッチ条件、優先順位
- ステータス、バージョン、適用範囲
- 承認フロー、例外処理

具体的であればあるほど、公開検索流入には向きません。

### 3. 重要な業務ページでも、インデックス向きとは限らない

業務上重要ということと、SEOで残す価値があるということは別です。検索エンジンが見るのは、そのURLが公開的で再利用可能な疑問に答えているかどうかです。

## どう扱うべきか。私はこの5つに分けて考えるのをすすめます

### 1. 標準的な rule page、match rule page、channel rule page なら、基本的にSEOの主力ページではない

これは一番よくあるケースです。

### 2. 実際の検索需要が「なぜルールが効かないのか」なら、私有のルールページを無理に順位付けさせない

この場合に必要なのは、公開ヘルプページ、FAQ、切り分けガイドです。

### 3. 公開向けルール説明ドキュメントなら、別で評価する

次の条件を満たすなら、インデックスを検討できます。

- ログインなしで理解できる
- 私有設定ではなく公開ルールを説明している
- URL が安定している
- 例や確認手順がある
- 実際の検索ニーズに対応している

### 4. rule、rules、match-rule、channel-rule、rule-detail、rule-log のような派生URLがあるなら、まとめて管理する

メインURLだけ塞いでも、派生URLが開いたままだと問題は残ります。

### 5. そもそも検索流入を取らせないなら、noindex、ログイン制御、sitemap、canonical、パラメータ制御をまとめて揃える

よくあるミスは次の通りです。

- noindex なのに sitemap に残っている
- メインページは要ログインなのに派生ページは見えてしまう
- canonical がバラバラ
- パラメータURLが複製を量産する

## よくあるSEOの失敗

### 1. ルールが多いほどSEO価値が高いと思うこと

実際には内部の複雑さが高いだけです。

### 2. メインの rule ページだけを処理すること

detail や log が残っていれば、問題は解決していません。

### 3. 公開検索ニーズに対して私有ページを当てようとすること

多くの場合、必要なのは公開説明ページです。

### 4. ルールを変えたのに、インデックス方針を変えないこと

ロジック、権限、レンダリングを変えたなら、sitemap や canonical も見直すべきです。

## 監査するときの順番

### ステップ1: ルール関連URLを全部洗い出す

### ステップ2: 本当の検索意図を確認する

### ステップ3: 公開ドキュメントと内部ページを分ける

### ステップ4: クロールとインデックスのシグナルをまとめて確認する

### ステップ5: 正しい結果を見る

インデックスから消えたかだけでなく、低価値URLが減ったか、本当に見せたい公開ページが強くなったかを見るべきです。

---

## まとめ

大事なのは条件や優先順位の数ではありません。そのページが内部フローのためのものか、公開検索ニーズへの答えなのかです。

内部ロジック、ロール差分、チャネル判定、承認フローのためのページなら、基本的には内部ページとして扱うべきです。ルール不具合や優先順位で流入を取りたいなら、そのための公開ドキュメントを別に作るほうが健全です。
