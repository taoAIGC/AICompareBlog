## 왜 많은 팀이 화이트스크린, 멈춤, 폴링, 재연결 URL은 정리하면서도 retry-offline 페이지는 놓칠까?

오프라인 페이지는 겉보기에 덜 위험해 보이기 때문입니다.

완전히 망가진 것처럼 보이지 않고, 오히려 다음과 같은 안내 문구가 잘 붙어 있는 경우가 많습니다.

- 네트워크를 사용할 수 없습니다
- 시스템이 오프라인 모드로 전환되었습니다
- 로컬 캐시 데이터를 표시하고 있습니다
- 네트워크가 복구되면 자동으로 이어집니다
- 중복 클릭하지 마세요. 시스템이 자동 재시도합니다

그래서 팀 입장에서는 **사용자 보호용 임시 페이지일 뿐인데, 검색엔진이 봐도 큰 문제는 아니겠지**라고 생각하기 쉽습니다.

하지만 실제로 이런 페이지의 대부분은 공개 콘텐츠가 아닙니다. 네트워크 단절, 앱 백그라운드 복귀, webview bridge 오류, 로컬 캐시 fallback 같은 상황에서 **지금 진행 중인 한 건의 작업이 완전히 끊어지지 않게 잡아두는 임시 URL**에 가깝습니다. 예를 들면 다음과 같습니다.

- `/claim/document-withdraw/retry-offline`
- `/claim/document-withdraw/retry-offline-mode`
- `/claim/document-withdraw/retry-disconnected-state`
- `/merchant/claim/document-withdraw/retry-offline?case=xxx`
- `/claim/document-withdraw/retry-network-unavailable`

이 페이지가 해결하는 문제는 공개 검색 질문이 아니라,

- 왜 갑자기 오프라인 모드로 들어갔는지
- 지금 보이는 정보가 로컬 캐시인지 실제 서버 상태인지
- 재시도가 자동으로 이어지는지, 사용자가 다시 시작해야 하는지
- 기다려야 하는지, 새로고침해야 하는지, 다시 열어야 하는지

같은 **진행 중인 플로우 문제**입니다.

---

## 오프라인 페이지는 무엇을 위해 존재할까?

### 1. 핵심 역할은 연결이 끊겨도 플로우가 완전히 무너지지 않게 붙잡아 두는 것

일반적인 retry offline 페이지는 다음과 같은 일을 합니다.

- 요청이 정상적으로 서버에 도달하지 못했음을 알린다
- 중복 제출을 막기 위해 핵심 버튼을 잠시 잠근다
- 로컬 초안이나 캐시가 남아 있는지 보여 준다
- 네트워크 복구 후 재시도나 이전 단계 복귀를 이어 준다
- 지원팀과 개발팀이 공통으로 볼 수 있는 장애 접점을 만든다

즉 본질적으로는 **워크플로 보호 페이지**이지, 콘텐츠 페이지가 아닙니다.

### 2. 케이스 정보, 권한, 기기, 네트워크 상태에 강하게 의존한다

이런 URL은 원래 문맥을 벗어나면 의미가 급격히 약해집니다. caseId, 권한, 유입 경로, 네트워크 상태, 로컬 캐시, service worker, webview 상태 등에 의존하기 때문입니다.

문맥 의존성이 강할수록 공개 SEO 페이지로는 맞지 않습니다.

### 3. 업무상 유용하다고 해서 SEO 가치까지 있는 것은 아니다

많은 팀이 여기서 헷갈립니다.

- **업무 가치**: 사용자를 플로우 안에 붙잡아 두고 실수를 줄인다
- **SEO 가치**: 공개 검색 질문에 안정적으로 답해야 한다

대부분의 오프라인 페이지가 실제로 답하는 것은 결국

**“현재 이 작업은 오프라인 상태이며, 네트워크 복구를 기다리는 중입니다.”**

이 정도입니다. 장기적인 검색 유입을 받을 공개 답변으로는 약합니다.

## 어떻게 처리해야 할까? 먼저 이 5가지 상황으로 나눠서 보자

### 1. 표준적인 retry-offline / offline-mode / disconnected-state 페이지라면, 보통 핵심 SEO 페이지로 취급하지 않는 편이 낫다

보통 이런 특징이 있습니다.

- 현재 케이스를 벗어나면 단독으로 이해하기 어렵다
- 네트워크, 캐시, 세션 상태에 따라 내용이 바뀐다
- 검색 유입보다 현재 사용자 구조에 가치가 있다

즉 **공개 SEO 랜딩 페이지보다는 내부 워크플로 URL로 관리하는 편이 자연스럽다**는 뜻입니다.

### 2. 실제 검색 수요가 “왜 오프라인 모드가 되는가”, “오프라인 이후 어떻게 해야 하는가”라면, 설명은 공개 도움말 페이지가 맡아야 한다

사용자가 궁금한 것은 내부 URL 자체가 아니라,

- 왜 갑자기 오프라인 모드가 켜졌는지
- 로컬 네트워크 문제인지, 세션 만료인지, 백엔드 장애인지
- 화면 정보가 오래된 캐시인지
- 네트워크 변경, 재로그인, 새로고침, 문의 중 무엇을 먼저 해야 하는지

같은 일반화된 답변입니다. 이런 것은 FAQ, 도움말, 트러블슈팅 문서가 더 잘 맞습니다.

### 3. 같은 URL이라도 기기·네트워크·시간에 따라 결과가 바뀐다면, 색인은 더 신중해야 한다

오프라인 페이지는 본질적으로 불안정합니다.

같은 URL이 회사 Wi‑Fi에서는 offline, 모바일 핫스팟에서는 복구, 앱에서는 캐시 화면, 브라우저에서는 정상 동작처럼 보일 수 있습니다.

페이지 자체가 안정적인 답이 아니라면, 검색 결과에 안정적인 답변으로 남길 이유도 약합니다.

### 4. offline / offline-mode / disconnected-state / network-unavailable 같은 변형이 여러 개 있다면, 묶어서 감사해야 한다

이 문제는 보통 한 페이지로 끝나지 않습니다. 비슷한 URL이 여러 개 생기면,

- 구조가 거의 중복되고
- 크롤 예산을 잡아먹고
- canonical 신호가 흔들리고
- 앱, 이메일, 지원 동선에서 계속 노출됩니다

그래서 한 페이지씩 보지 말고 묶어서 봐야 합니다.

### 5. 애초에 순위 대상이 아니라면, noindex·로그인 제어·canonical·sitemap·캐시·노출 경로를 함께 정리해야 한다

자주 보는 실수는 다음과 같습니다.

- noindex로 두고 싶지만 sitemap에는 계속 실린다
- 로그인 필수여야 하지만 일부 링크는 익명 접근이 된다
- offline, status, result 페이지끼리 canonical이 충돌한다
- service worker와 로컬 캐시 때문에 크롤러와 사용자가 보는 상태가 다르다
- 앱, 이메일, 내부 도구가 URL을 계속 노출한다

이 경우 한 군데만 고쳐서는 해결되지 않습니다.

## 자주 보는 SEO 실수 4가지

### 1. “그냥 네트워크 안내 페이지니까 색인돼도 괜찮다”라고 생각한다

플로우에 도움이 된다고 해서 검색에도 도움이 되는 것은 아닙니다.

### 2. 화이트스크린이나 명확한 오류만 정리하고 오프라인 페이지는 남겨 둔다

그 결과 가치 낮은 URL이 계속 살아남습니다.

### 3. 공개 도움말을 만들지 않고 private offline URL에 검색 유입을 맡기려 한다

검색에서 성과가 나는 것은 보통 FAQ와 가이드이지, 내부 대기 페이지가 아닙니다.

### 4. 색인 여부만 보고, URL이 어디서 계속 노출되는지는 보지 않는다

앱, 이메일, 지원 동선이 그 URL을 계속 뿌리고 있다면 문제는 끝나지 않습니다.

## 지금 바로 audit하려면 이 순서가 좋다

### Step 1: offline 계열 URL을 전부 모은다

offline 본문, disconnected state 변형, 파라미터 URL, 앱·이메일·지원·로그 유입 경로를 모두 확인합니다.

### Step 2: 공개 검색 의도와 내부 플로우 요구를 분리한다

사용자가 진짜 알고 싶은 것은 왜 offline mode가 켜졌는지, 왜 복구가 안 되는지, 왜 앱과 웹이 다른지입니다.

### Step 3: 공개 설명 페이지와 private workflow 페이지를 분리한다

검색 수요는 FAQ, 도움말, 트러블슈팅으로 받고, 내부용 URL은 워크플로 내부에 남깁니다.

### Step 4: 기술 신호와 offline recovery 체인을 함께 점검한다

noindex, canonical, 로그인 제어, 파라미터, sitemap, 캐시, service worker, deep link, 이메일 이동을 함께 봅니다.

### Step 5: “색인에서 빠졌는가”만으로 성공을 판단하지 않는다

저가치 URL이 줄었는지, 크롤 예산이 중요한 페이지로 돌아왔는지, 공개 문서가 노출을 받기 시작했는지도 함께 봐야 합니다.
