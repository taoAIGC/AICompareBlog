## 검토 중인 페이지는 어떤 문제가 해결되나요? 처음 검색하는 사람이 아닌, 그 과정에 있는 사람에게 서비스를 제공하는 것이 우선입니다.

### 1. 이미 청구 절차에 참여하고 있는 사람들에게 서비스를 우선적으로 제공합니다.

리뷰 페이지의 가장 일반적인 기능은 일반적으로 다음과 같습니다.

- 사용자에게 사례가 성공적으로 제출되었으며 검토를 기다리고 있음을 알립니다.
- 현재 카드가 사전심사인지, 심사인지, 직접검사인지, 통신사 확인인지 사용자에게 알려줍니다.
- 지금 계속 기다려야 하는지, 재료를 보충해야 하는지, 고객 서비스에 문의해야 하는지 사용자에게 알려줍니다.
- 현재 케이스가 어느 노드에 있는지, 시간이 초과되었는지, 수동 개입이 필요한지 여부를 고객 서비스 및 운영팀에 알립니다.

즉, Google에서 처음으로 클릭하는 익숙하지 않은 사용자가 아닌 이미 컨텍스트를 알고 있는 사용자에게 서비스를 제공하는 것입니다.

### 2. 일반적으로 사건번호, 계좌 신원, 검토 노드, 처리 시간 제한과 깊이 연관되어 있습니다.

- 재제출 허용 여부

이는 검토 중인 페이지가 일반적으로 안정적이지 않고 표준화되지 않았으며 장기적인 검색 트래픽에 적합하지 않다는 것을 결정합니다.

### 3. 비즈니스 가치는 있지만 SEO 가치가 있다는 의미는 아닙니다.

많은 팀이 혼동하는 가장 일반적인 사항은 다음과 같습니다.

- 페이지는 프로세스에서 중요하므로 SEO에서도 중요해야 합니다.

## 검토 중인 페이지를 어떻게 처리하나요? 나는 먼저 이 5가지 상황을 개별적으로 처리하는 것을 선호합니다.

### 1. 표준 리뷰 페이지, 처리 중인 페이지, 리뷰할 페이지인 경우에는 일반적으로 핵심 SEO 페이지로 취급할 필요가 없습니다.

가장 일반적인 카테고리입니다.

이러한 페이지에는 대부분 다음과 같은 몇 가지 공통점이 있습니다.

- 로그인을 하셔야 보실 수 있습니다.
- 사건번호 또는 사용자 신원을 묶어야 함
- 페이지의 콘텐츠가 특정 청구 기록에 중점을 두고 있습니다.
- 검토가 진행되고 보완이 이루어지며 수작업이 진행되면서 상태는 계속해서 변경됩니다.
- 외부 사용자에 대한 안정적인 판독값은 거의 없음

한마디로: **표준 리뷰 페이지는 공개 콘텐츠 페이지가 아닌 프로세스 페이지에 가깝습니다. **

### 2. 실제 검색 값이 "검토 중에 결과가 공개되는 데 시간이 얼마나 걸립니까?", "왜 항상 검토 중입니까?"라면 비공개 검토 페이지가 이러한 질문에 대한 트래픽을 차지하지 않도록 하세요.

많은 팀이 여기서 실수를 저지르는 경향이 있습니다.

사용자들은 "검토중", "검토중", "검토중"을 검색하기 때문에, 검색하는 사람들이 있으므로 검토중 페이지를 포함해야 한다는 결론을 내리는 것이 당연합니다.

그러나 사용자가 실제로 알고 싶어하는 것은 일반적으로 다음 사항이 아닙니다.

- 특정 사건에는 어떤 감사관이 배정되나요?
- 이번에는 계정 상태의 일부만 표시되는 이유는 무엇입니까?
- 특정 기록이 어제 사전 검토에서 검토로 옮겨졌나요?
- 이제 특정 가맹점이 백엔드에서 보충 버튼을 클릭하는 것이 가능합니까?

그들이 더 알고 싶은 것은 다음과 같습니다.

- 일반적인 검토시간은 얼마나 되나요?
- 검토가 너무 오래 걸렸는데도 결과가 나오지 않는 이유는 무엇입니까?
- 검토 중에도 정보를 추가할 수 있나요?
- 어떤 상황에서 고객 서비스에 적극적으로 연락해야 합니까?
- 검토중, 대기중, 검토의 차이점은 무엇입니까?

이러한 유형의 요청은 특정 사례의 검토 페이지보다는 공개 설명 페이지, 도움말 페이지, FAQ 페이지 및 규칙 페이지에 더 적합합니다.

### 3. 시스템에서 under-review, in-review, 보류 중인-review, review-status, review-record 등의 변형 URL을 동시에 생성하는 경우 반드시 함께 제어해야 합니다.

많은 웹사이트의 실제 문제는 단순한 '/claim/under-review'가 아닙니다.

보다 일반적으로 시스템은 다음과 같이 성장합니다.

- `/청구/검토중`
- `/claim/in-review?id=xxx`
- `/appeal/pending-review`
- `/case/대기-검토`
- `/after-sales/review-status`
- `/판매자/클레임/검토-기록`
- `/청구/검토 내역`
- `/support/case/in-review-detail`

이러한 URL이 많으면 쉽게 표시됩니다.

- 내용의 본문은 거의 동일합니다.
- 매개변수, 노드, 시간만 다름
- 타이틀 템플릿은 매우 반복적입니다.
- 기록, 이력, 상세정보 등의 페이지도 캡처됩니다.
- 사용자 센터, 판매자 백엔드 및 사이트 메시지는 계속해서 이러한 페이지에 대한 입구를 제공합니다.

따라서 리뷰 페이지를 다룰 때 매우 실용적인 조치는 다음과 같습니다.

**메인리뷰페이지만 보지 마시고, 현황기록페이지, 내역페이지, 상세페이지, 처리페이지를 함께 살펴보세요. **

### 4. '공개 설명 페이지'와 '비공개 리뷰 페이지'가 모두 있는 경우 두 페이지를 완전히 분리하세요.

많은 웹사이트가 정말 지저분합니다. 페이지 수가 너무 적어서가 아니라 페이지의 역할이 명확하게 정의되지 않았기 때문입니다.

한 세트는 다음과 같은 공개 페이지입니다.

- 청구 검토 결과를 받는 데 보통 얼마나 걸리나요?
- 심사 중에도 자료를 추가할 수 있나요?
- 검토중, 검토중, 대기중의 차이점은 무엇입니까?
- 검토 시간이 초과되면 어떻게 해야 합니까?
- 상태가 업데이트되지 않은 이유는 무엇입니까?

다른 세트는 다음과 같은 개인 페이지입니다.

- 주문의 검토 상태 페이지
- 특정 가맹점 백엔드의 감사기록 페이지
- 특정 보완 후 검토할 세부정보 페이지
- 계정 전용 수동 검토 진행 페이지

이 두 가지 유형의 페이지가 혼합된 경우 가장 일반적인 결과는 다음과 같습니다.

- 누구도 대중의 요구를 받아들이지 않는다.
- 비공개 페이지가 검색 엔진에 의해 크롤링되었습니다.
- 사용자는 검색으로 들어와 완전한 정보를 이해하지 못하고 볼 수 없습니다.
- 정말 순위에 해당하는 설명 페이지와 FAQ 페이지는 열 수 없습니다.

### 5. 리뷰에 포함된 페이지가 순위에 참여하는 것을 허용하지 않을 경우 NOINDEX, 로그인 차단, 사이트맵, 표준 및 내부 링크를 함께 통합합니다.

감사에서 페이지 중간 SEO 문제가 많이 발생하는 이유는 '이 페이지가 있기 때문'이 아니라 신호가 모두 싸우기 때문입니다.

일반적인 상황은 다음과 같습니다.

- 페이지 템플릿에는 NOINDEX가 기록되어 있지만 사이트맵에서는 계속해서 검토중 URL을 제출하고 있습니다.
- 메인페이지는 로그인이 필요하며, 내역, 상세정보, 기록 등의 페이지도 공개적으로 접근이 가능합니다.
- Canonical은 혼란을 의미하며 검토 중인 페이지, 대기 중인 페이지, 검토 페이지가 서로 얽혀 있습니다.
- 사용자 센터, 도움말 센터, 사이트 검색을 통해 이 페이지에 대한 크롤링 가능한 입구가 아직 만들어지고 있습니다.

리뷰 페이지가 주요 SEO 항목이 되어서는 안 된다고 판단했다면, 중간 단계로 진행하지 마세요.

보다 안정적인 접근 방식은 일반적으로 다음과 같습니다.

- 로그인 완료 후 페이지 수준을 NOINDEX 또는 접근권한으로 통일
- 사이트맵은 그러한 프로세스 URL을 제출하지 않습니다.
- 표준 중간 상태 페이지를 무작위로 가리키지 마세요.
- 리뷰 페이지를 내부 링크용 콘텐츠 페이지로 패키징하지 마세요.
- 내역, 세부정보, 기록 및 매개변수 페이지가 자동으로 크롤링 가능한 항목이 됩니다.

## 중간 페이지 감사에서 확인되는 가장 일반적인 4가지 SEO 실수는 이를 수행하지 않는 것보다 사이트를 엉망으로 만들 가능성이 더 높습니다.

### 1. '검토 중'은 사용자가 검색할 수 있다는 의미이므로 검토 중인 페이지는 당연히 포함될 가치가 있다고 생각됩니다.

사용자는 "항상 검토 중인 이유"를 검색합니다. 이는 특정 사례의 검토 중인 URL이 순위를 매길 가치가 있다는 의미는 아닙니다. 전자는 공개 문제이고 후자는 일반적으로 비공개 프로세스 상태입니다.

### 2. 리뷰 페이지, 리뷰 페이지, 보류 페이지, 결과 페이지를 혼합합니다. 결국 사이트에는 유사한 의미를 지닌 여러 페이지가 있게 됩니다.

이러한 페이지의 비즈니스 의미는 본질적으로 다릅니다. 제목, 템플릿, 내부 링크 등이 뒤섞여 있으면 결국에는 더 완전하게 다루지 못하고 서로 단어 경쟁을 하게 되는 경우가 많습니다.

### 3. 내역, 기록, 세부사항, 진행상황 등의 페이지가 시스템에서 쉽게 공개됩니다. 결과적으로 얇은 페이지는 한 레이어 안에 다른 레이어 안에 배치됩니다.

이러한 페이지는 내부 프로세스에 유용할 수 있지만 공개 검색에는 대부분 독립적인 가치가 없습니다. 일괄 출시되면 크롤링 예산을 분산하고 중복 콘텐츠를 생성합니다.

### 4. "클레임 검토 중 결과를 얻는 데 시간이 얼마나 걸리는지"에 대한 공개 내용을 명확하게 만들어야 하지만, 검색 요구는 항상 비공개 프로세스 페이지에서 처리하도록 만들고 싶습니다.

이것은 실제로 많은 웹사이트의 근본적인 문제입니다. 사용자 문제가 없다는 것이 아니라 이러한 문제가 공개 콘텐츠로 정리된 적이 없다는 것입니다.

## 지금 홈페이지의 중간리뷰 페이지 설정을 확인하고 싶다면 이 순서대로 진행하시는 것을 추천드립니다.

### 1단계: 먼저 검토 중인 관련 URL을 모두 찾습니다.

최소한 다음 유형을 꺼내서 살펴보십시오.

- 검토 중인 페이지
- 보류 중인 검토 페이지
- 리뷰 상태 페이지
- 내역/기록/상세페이지
- 매개변수가 포함된 검토 중인 URL
- 판매자 백엔드 및 사용자 센터의 검토 프로세스 페이지

### 2단계: 공개 콘텐츠 페이지에서 처리해야 하는 요구 사항 결정

사용자가 실제로 무엇을 검색하고 있는지에 집중하세요.

-클레임 검토에 보통 얼마나 걸리나요?
- 왜 항상 검토중이라고 표시되나요?
- 심사 중에도 자료를 추가할 수 있나요?
- 검토중, 대기중, 검토의 차이점은 무엇입니까?
- 검토 시간이 초과된 후 해야 할 일

### 3단계: 공개 설명 페이지와 비공개 프로세스 페이지를 완전히 분리합니다.

검색을 처리할 수 있는 페이지는 설명 페이지, FAQ 페이지, 도움말 페이지로 구성됩니다. 특정 사례에만 서비스를 제공할 수 있는 항목은 프로세스 페이지로 관리됩니다.

### 4단계: 기술적 신호를 통합하고 색인이 생성되지 않은 동안 페이지가 계속해서 크롤링되도록 푸시되지 않도록 합니다.

NOINDEX, Canonical, 사이트맵, 로그인 차단, 내부링크 진입을 함께 살펴보세요. 한 점만 바꾸고 끝났다고 생각하지 마세요.

### 5단계: 결과를 볼 때 '검토 중인 페이지가 포함되었는지'만 보지 마세요.

더 살펴봐야 할 것은 다음과 같습니다.

- 공개 설명 페이지에서 "검토 중 결과가 발표되기까지 시간이 얼마나 걸리나요?"와 같은 검색어를 허용하기 시작했나요?
- 과소검토, 기록, 이력 등 프로세스 페이지가 점차 지수에서 빠져나가고 있는가?
- 크롤링 예산이 제품 페이지, 카테고리 페이지, 블로그 페이지 및 도움말 페이지로 돌아가나요?
- 사용자를 검색한 후 정말 유용한 답변을 더 빨리 찾을 수 있나요?
