## なぜ白画面やフリーズ、ポーリング、再接続URLは整理したのに、retry-offline 系のページだけは見落とされやすいのか？

理由は単純で、**壊れて見えないから**です。

オフラインページはクラッシュ画面のように派手ではありません。むしろ普通に開けて、「ネットワークが利用できません」「オフラインモードに切り替わりました」といった案内も出るので、チーム側が「これは一時的な待避ページだし、検索エンジンに見つかっても大きな問題ではない」と判断しがちです。

でも実際には、こうしたページの多くは公開コンテンツではありません。ネットワーク断、アプリのバックグラウンド復帰、webview の橋渡し失敗、ローカルキャッシュへのフォールバックといった場面で、**いま進んでいる1件の処理を落とさないための一時URL**です。たとえば次のようなURLです。

- `/claim/document-withdraw/retry-offline`
- `/claim/document-withdraw/retry-offline-mode`
- `/claim/document-withdraw/retry-disconnected-state`
- `/merchant/claim/document-withdraw/retry-offline?case=xxx`
- `/claim/document-withdraw/retry-network-unavailable`

このページが答えているのは、公開検索の悩みというより、

- なぜ急にオフラインモードに入ったのか
- 画面に出ているのはローカルキャッシュか、本当のサーバー状態か
- リトライは自動継続されるのか、自分でやり直す必要があるのか
- 待つべきか、更新すべきか、開き直すべきか

といった**進行中フローの問題**です。

---

## オフラインページは何のために存在しているのか

### 1. 役割の中心は、接続が切れてもフローを完全に落とさないこと

典型的な retry offline ページは、次のようなことを行います。

- リクエストが正常にサーバーへ届かなかったことを伝える
- 二重送信を防ぐために重要ボタンを一時ロックする
- 下書きやローカルキャッシュが残っているかを示す
- ネットワーク復旧後に再試行や前ステップ復帰を行う
- サポートや開発に共通の障害受け皿を作る

つまり本質的には**ワークフローの保護ページ**であって、コンテンツページではありません。

### 2. ケース情報、権限、端末、ネットワーク状態に強く依存する

この種のページは、元の文脈を外れると意味がかなり薄くなります。caseId、権限、流入元、回線状態、ローカルキャッシュ、service worker、webview 状態などに依存するからです。

文脈依存が強いほど、公開SEOページには向きません。

### 3. 業務上便利でも、SEO上価値があるとは限らない

ここを混同するチームは多いです。

- **業務上の価値**: ユーザーをフロー内にとどめ、誤操作を減らす
- **SEO上の価値**: 公開検索の安定した疑問に答える

多くのオフラインページが実際に答えているのは、

**「いまこのタスクはオフライン状態で、ネットワーク復帰待ちです」**

という一点だけです。これでは長期的な検索流入先としては弱いです。

## どう処理するべきか？ まずはこの5パターンに分けて考える

### 1. 標準的な retry-offline / offline-mode / disconnected-state でしかないなら、基本的には主力SEOページにしない

よくある特徴は次の通りです。

- 現在のケースを離れると単独では理解しづらい
- ネットワーク、キャッシュ、セッション状態で内容が変わる
- 価値の中心は検索流入ではなく、進行中ユーザーの救済にある

要するに、**公開SEO入口より、私有フローURLとして扱うほうが自然**です。

### 2. 本当に検索されているのが「なぜオフラインモードになるのか」「オフライン後はどうするのか」なら、その説明は公開ヘルプページで受けるべき

ユーザーが知りたいのは内部URLそのものではなく、

- なぜ急にオフラインモードになったのか
- ローカル回線なのか、セッション切れなのか、バックエンド障害なのか
- 画面上の情報は古いのか
- まず回線変更、再ログイン、更新、問い合わせのどれをすべきか

といった一般化された答えです。これは FAQ やヘルプ、障害切り分け記事に向いています。

### 3. 同じURLでも端末・回線・時間で表示が変わるなら、インデックスはさらに慎重に

オフラインページは不安定です。Wi‑Fi では offline、4G では復帰、アプリではキャッシュ表示、ブラウザでは正常、ということが普通に起きます。

ページ自体が安定した答えでないなら、検索結果で安定した答えとして残す理由は薄いです。

### 4. offline / offline-mode / disconnected-state / network-unavailable のような変種が複数あるなら、まとめて監査する

こうした問題は1ページだけで終わらないことが多いです。複数の似たURLが並ぶと、

- 構造がほぼ重複する
- クロールバジェットを消費する
- canonical がぶれる
- アプリ、メール、サポート導線から露出し続ける

といった問題が起きます。

### 5. そもそも順位対象にしないなら、noindex・ログイン制御・canonical・sitemap・キャッシュ制御・露出元をまとめて整える

よくある失敗は、

- noindex にしたいのに sitemap では送っている
- ログイン必須のはずなのに一部リンクは匿名で開ける
- offline と status と result の canonical が競合する
- service worker やローカルキャッシュの都合でクローラーとユーザーの見える状態が違う
- アプリやメールや社内ツールがURLを出し続ける

といったものです。1か所だけ直しても収まりません。

---

## よく見るSEOの失敗パターン4つ

### 1. 「ただのオフライン通知だから、インデックスされても大丈夫」と考える

フローに役立つことと、検索に役立つことは別です。

### 2. 白画面や明確なエラーだけ整理して、オフラインページを残す

その結果、価値の低いURLが残り続けます。

### 3. 公開ヘルプを作らず、私有の offline URL に検索流入を背負わせようとする

検索で勝ちやすいのは FAQ やガイドであって、内部の待避ページではありません。

### 4. インデックス有無だけ見て、URL がどこから露出しているかを見ない

アプリ、メール、サポート導線がそのURLを出し続けている限り、問題は終わりません。

## 今すぐ audit するなら、この順番がおすすめ

### Step 1: offline 系URLを全部集める

offline ページ本体、disconnected state の変種、パラメータ付きURL、アプリ・メール・サポート・ログ経由の入口を洗い出します。

### Step 2: 公開検索意図と内部フロー要件を分ける

ユーザーが本当に知りたいのは、なぜ offline mode になるのか、なぜ復帰しないのか、なぜアプリとWebで挙動が違うのか、です。

### Step 3: 公開説明ページと私有フローページを分離する

公開検索の受け皿は FAQ、ヘルプ、トラブルシューティングに。内部用は workflow URL のまま管理します。

### Step 4: 技術信号と offline recovery チェーンを一緒に確認する

noindex、canonical、ログイン制御、パラメータ、sitemap、キャッシュ、service worker、deep link、メール遷移をまとめて見ます。

### Step 5: 「インデックスから落ちたか」だけで成功判断しない

低価値URLが減ったか、クロールバジェットが重要ページへ戻ったか、公開ドキュメントが表示を取り始めたかも見てください。

---

## 最後に

retry offline ページは、ほとんどの場合「コンテンツページ」ではありません。接続が切れたときにタスクが崩れないようにするための、一時的な緩衝レイヤーです。

プロダクト体験には重要でも、SEO価値が高いとは限りません。まず workflow の都合と index の都合を切り分けることが大事です。
