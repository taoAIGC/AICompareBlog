## 번역 페이지를 다 만들어 놨는데도 왜 구글은 자꾸 엉뚱한 언어 페이지를 보여줄까?

이건 정말 흔합니다.

영문, 일본어, 독일어, 프랑스어 페이지까지 다 열어놨는데 검색 결과에서는:

- 일본 사용자가 영어 페이지로 들어가고
- 영국 사용자가 미국 버전으로 가고
- 브라질 사용자가 일반 포르투갈어 페이지로 가고
- 로컬 페이지 대신 글로벌 메인 페이지가 노출됩니다

그러면 팀에서는 이런 생각을 하죠.

- 번역 품질이 별로인가?
- 로컬 페이지 권한이 약한가?
- 콘텐츠를 더 늘려야 하나?
- IP 기준 자동 리다이렉트를 더 세게 걸어야 하나?

물론 이런 요소가 영향을 줄 수는 있습니다. 하지만 실제로는 더 기본적인 문제인 경우가 많습니다.

**hreflang 매핑이 제대로 안 되어 있는 것**입니다.

검색엔진은 폴더만 나눠 놓았다고 해서, 이 페이지들이 같은 내용의 다른 언어판인지 다른 국가판인지 완벽하게 자동 이해하지 않습니다.

명확하게 설명하지 않으면 결국 추측하게 됩니다.

SEO에서 검색엔진에게 추측을 맡기는 건 대체로 좋은 선택이 아닙니다.

## hreflang은 실제로 무슨 역할을 할까?

hreflang은 순위를 단번에 올려 주는 마법 버튼이 아닙니다. 오히려 검색엔진에게 주는 “언어/국가 설명서”에 가깝습니다.

즉 이런 뜻입니다.

> 이 페이지들은 서로 대응되는 버전이지만, 대상 언어와 대상 지역이 다르다.

hreflang이 도움 되는 핵심은 세 가지입니다.

1. 페이지 버전 관계를 분명히 해 준다
2. 사용자에게 더 맞는 언어/국가 페이지를 보여줄 가능성을 높인다
3. 다국어 사이트에서 흔한 “잘못된 나라에 잘못된 페이지가 뜨는 문제”를 줄인다

**hreflang 설정 방법**을 찾고 있다면 핵심은 이겁니다.

**태그를 붙이는 것보다 더 중요한 건 어떤 페이지끼리 연결할지, 어떤 코드로 쓸지, 다른 SEO 신호와 충돌하지 않게 할지입니다.**

---

## hreflang 설정은 이 5가지 상황을 나눠서 봐야 한다

### 1. 같은 내용의 다국어 버전이 있다면 페이지 단위로 1:1 대응시켜야 한다

홈페이지만 설정해 놓고 끝내면 부족합니다.

같은 상품 페이지가 한국어, 영어, 일본어, 독일어로 있다면 각 페이지가 서로를 alternate로 가리켜야 합니다.

자주 보이는 실수는:

- 홈만 설정함
- 한쪽만 가리키고 반대쪽은 안 가리킴
- “여러 언어 지원” 자체로 충분하다고 생각함

그렇게는 부족합니다.

### 2. 언어는 같아도 국가가 다르면 지역 코드를 분리해야 한다

여기서 많이 흐려집니다.

예를 들면:

- `en-us`
- `en-gb`
- `en-au`
- `pt-br`
- `pt-pt`

언어는 비슷하지만 가격, 통화, 철자, 배송, 프로모션이 달라질 수 있습니다.

그런데 전부 `en` 또는 `pt`로만 묶어 버리면 신호가 너무 거칠어집니다.

사용자는 “맞는 언어”만 필요한 게 아니라, “맞는 시장 버전”도 필요한 경우가 많습니다.

### 3. 국가 선택 페이지나 글로벌 진입 페이지가 있으면 `x-default`를 써야 한다

국제 사이트에서는 흔히 있습니다.

- 언어 선택 페이지
- 국가 선택 페이지
- 글로벌 메인 페이지
- 특정 지역이 없는 공용 랜딩 페이지

이럴 때 `x-default`가 적합합니다.

이건 검색엔진에게 “더 적합한 언어/국가 버전을 찾지 못하면 이 페이지를 기본으로 사용하라”고 알려 주는 역할입니다.

### 4. 실제로 대응되는 페이지끼리만 연결해야 한다

hreflang은 많이 연결한다고 좋은 게 아닙니다.

흔한 문제는:

- 모든 영어 페이지가 모든 일본어 페이지를 가리킴
- 카테고리 페이지가 다른 국가의 상품 페이지를 가리킴
- 삭제된 페이지가 아직 세트에 남아 있음
- 도움말 글이 홈으로 연결됨

이건 명확함이 아니라 잡음입니다.

**주제, 역할, 페이지 단계가 같은 것끼리만 alternate로 묶는 것**이 기본입니다.

### 5. canonical, noindex, 리다이렉트도 함께 봐야 한다

여기서 조용히 망가지는 경우가 많습니다.

- hreflang은 있는데 canonical이 글로벌 페이지로 돌아감
- hreflang 세트에 있지만 실제 접속하면 다른 곳으로 이동함
- 지역 페이지가 noindex인데도 alternate에 남아 있음
- 자동 리다이렉트가 너무 강해서 크롤러가 안정적으로 못 봄

독립된 언어판/국가판으로 유지할 페이지라면 접근 가능하고, 색인 가능하고, 안정적으로 열려야 합니다.

## 자주 보이는 hreflang 실수 4가지

### 1. 한 방향만 설정
A는 B를 가리키는데 B는 A를 안 가리킴.

### 2. 코드 오류
언어 코드와 국가 코드를 섞거나 형식을 잘못 씀.

### 3. 과한 IP 리다이렉트
사용자도 크롤러도 원본 페이지를 보기 어려워짐.

### 4. 깨끗하지 않은 대상 URL
404, 리다이렉트 URL, 테스트 페이지, 파라미터 URL을 넣어 버림.

## hreflang 점검 체크리스트

### 1단계: 실제 대응 관계가 있는 페이지부터 정리한다
홈, 핵심 상품 페이지, 주요 카테고리, 트래픽이 큰 콘텐츠부터 시작하면 됩니다.

### 2단계: 상호 참조인지 확인한다
A가 B를 가리키면 B도 A를 가리켜야 합니다.

### 3단계: 언어 코드, 국가 코드, 최종 URL을 확인한다
중간에 한 번 더 이동하는 URL이 아니라 최종 도착 URL이어야 합니다.

### 4단계: canonical, noindex, 리다이렉트도 같이 본다
hreflang만 따로 보면 놓치는 게 많습니다.

### 5단계: 배포 후 검색 결과를 관찰한다
계속 다른 국가 버전이 노출된다면 매핑이 아직 정리되지 않은 겁니다.

## 마지막 한마디

hreflang이 어려운 이유는 태그 문법 때문이 아닙니다.

진짜 어려운 건,

- 어떤 페이지가 진짜 대응 버전인지
- 비슷해 보여도 묶으면 안 되는 건 무엇인지
- 국가별 분리가 필요한지
- x-default를 어디에 둘지
- 다른 SEO 신호와 충돌하고 있지 않은지

를 먼저 정리해야 하기 때문입니다.

이 관계가 정리되면 hreflang은 꽤 강력합니다.
반대로 관계가 흐린데 태그만 늘리면 검색엔진을 더 헷갈리게 만들 뿐입니다.

**구글이 언어 버전을 추측하게 두지 마세요. 명확하게 표시할 수 있으면, 명확하게 표시하는 게 가장 낫습니다.**
